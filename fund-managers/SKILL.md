# 基金经理方法论矩阵 Skill

6 位基金经理 = 6 个**透镜**。同一标的，每人透过自己的透镜独立觀察，并排呈现结果，不做综合。

## 编排流程

```
用户输入
   │
   ▼
┌─────────────────────────────────────────────────┐
│ ① 意图判断                                       │
│    明确 → 继续；不明确 → 追问（不问废话）            │
│    排除闲聊/非投资话题                             │
└──────────────────────┬──────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────┐
│ ② 取基础信息（宽，一次拉够多数场景需要）            │
│    引擎调用加 --output-dir 写盘                   │
│    写 fund-managers-output/<datetime>/data/      │
│    用时预期 ≤ 5 分钟                              │
│    回退机制兜底，仍然失败则中止分析                 │
└──────────────────────┬──────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────┐
│ ③ 选透镜                                         │
│    用户已指定 → 照办                               │

└──────────────────────┬──────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────┐
│ ④ 透过透镜（串行 Agent，隔离执行）                 │ ───→  子代理①
│                                                    │        │
│    父进程为每位经理启动一个独立子代理                │        读 method.md

│     · 标的代码/名称                                 │        按需补数据

└─────────────────────────────────────────────────┘
```

## 引用规则（全局约束）

所有输出（任何分支）遵守：

- 分清三种话：**原话** / **推演** / **需核实**
- 引用原话时自然标注来源（时间+出处），不说文件路径
- 标注方式因叙事风格不同而异（见各分支具体要求），但三种话的边界始终保持

---

## ① 意图判断

收到用户输入后，先判断意图：

| 用户说什么 | 意图 | 后续 |
|-----------|------|------|
| 提到经理姓名 + 没提标的 | **讲解经理方法**（见 §讲解） | 读该经理 method.md → 按模块输出 |
| 提到基金代码或名称（含"基金""混合""ETF"等） | **分析基金** | 有代码直接走 ②；仅有名称→websearch 查出代码→向用户确认→走 ② |
| 提到个股代码或名称（沪/深/北交所） | **分析个股** | 有代码直接走 ②；仅有名称→websearch 查出代码→向用户确认→走 ② |
| 提到经理姓名 + 行业/主题 | **推演**（见 §推演） | 读该经理 method.md → 标注推演 → 输出 |
| 其他 | 不清楚 | 追问（不要无意义提问） |

**原则：** 清楚就走，不清楚就问。不问废话。

**完成标志：** 用户意图可归入表内某一类 → 明确，走对应后续。否则 → 追问。
追问一次后仍无法归入表内 → 给出可选项列表让用户点选，不反复追问。

**跨分支标注：** 注意表内的"见 §讲解"和"见 §推演"行——走的是独立分支，不经过②-⑤流程。

## ② 取基础信息

### 标的类型与引擎映射

引擎清单、调用参数、输出文件名见 `scripts/ENGINE.md`。
标的类型决定调哪个引擎组——基金走 `fund/` 下引擎，个股走 `stock/` 下引擎。
代码未知时先在 ① 阶段通过 websearch 查代码、向用户确认后继续，不启动本地搜索。

### --output-dir 行为

传 `--output-dir` 时：引擎写 JSON 到目录下的文件，脚本只打印文件路径到 stdout
不传时：引擎打印 JSON 到 stdout（旧行为不变）

### 回退与中止

每个引擎调用内部有重试逻辑（3 次）。引擎仍失败时：
- 如实报告："XX 数据获取失败"
- AI 自行判断剩余数据是否足以支撑分析
- 不足以支撑 → 中止，告知用户缺了什么、为什么没法继续
- 足以支撑 → 继续，注明"XX 数据缺失"

等一个引擎不影响后续（如经理信息缺失但持仓拿到了）→ 不中止，继续。

### 输出目录

所有文件写入 `fund-managers-output/<datetime>/data/`，其中 `<datetime>` 格式为 `YYYY-MM-DD_HHMM`。

每次执行建立新 `data/` 目录，不缓存、不复用。

### 数据获取协议

无论父进程还是子代理，获取任何数据都遵守同一协议：

```
① 根据 ENGINE.md 确定引擎名和预期输出文件名 {code}_{engine}.json
② 读 data/manifest.md，检查该文件名是否存在？
    是 → 文件已在池中，直接读，跳过引擎调用
    否 → 调引擎脚本 --output-dir data/
         文件落盘后，追加一笔到 data/manifest.md
③ 读数据文件
```

### manifest.md

`data/manifest.md` 是数据目录的目录表，由获取数据的进程维护。格式：

```markdown
# Data Manifest

## 600519_price.json
- 描述：日线行情（前复权，2000 年起）
- 引擎：stock/price

## 600519_valuation.json
- 描述：PE/PB/PS 历史序列+百分位
- 引擎：stock/valuation

## 001513_holdings.json
- 描述：前十大持仓（全部披露季度）
- 引擎：fund/holdings
```

追加操作：在文件末尾加一个 `## {filename}\n- 描述：{label}\n- 引擎：{engine}\n` 段。
不需要加锁——子代理串行执行，无并发问题。

### 输出目录示例

```
fund-managers-output/
└── 2026-06-26_1430/
    ├── data/
    │   ├── manifest.md
    │   ├── 001513_holdings.json
    │   ├── 001513_nav.json
    │   ├── 001513_scale.json
    │   └── 001513_managers.json
    ├── 郑希/
    │   └── analysis.md
    ├── 姜诚/
    │   └── analysis.md
    └── summary.md
```

## ③ 选透镜

**用户已指定 → 照办。**

**用户未指定：** 从 `data/` 判断标的风格偏向。行业集中度 > 60%（单一赛道）→ 选 2-3 个最契合的经理，不追问。行业分布均衡（均方差 < 15%）→ 问用户选哪几个。

**完成标志：** 经理名单确定（每人一份 method.md 路径）。进入下一步。

**例（需追问时）：**
> "这只基金持仓分散、风格均衡。你希望我用哪几位经理来分析？郑希（景气度）、姜诚（深度价值）、张坤（GARP）、刘格菘（行业集中）、周蔚文（均衡）、葛兰（医药）、刘元海（科技趋势）、莫海波（逆向轮动）、蓝小康（价值红利），还是全部来看？"

## ④ 透过透镜

经理名单确定后，父进程为每位经理启动一个**独立的子代理**，串行执行（一个跑完再跑下一个）。子代理之间完全隔离：互不知晓存在、不读彼此的文件、不通信。

### 父进程传给子代理的信息

| 字段 | 内容 |
|------|------|
| 角色身份 | "你是郑希的分析执行器，按郑希的投资方法分析标的" |
| 标的 | 代码 + 名称 |
| 用户意图 | 结构化转述（如"用户想看这只基金在成长风格上是否契合，以及深度价值框架下是否值得买"） |
| data 目录 | `fund-managers-output/<datetime>/data/` |
| 输出目录 | `fund-managers-output/<datetime>/<经理>/` |
| 方法文件 | `references/<经理>/method.md` |

### 子代理执行步骤

```
子代理启动
   │
   ▼
读 method.md（理解框架）
   │
   ▼
读 data/manifest.md，按需读 data/ 下的 JSON 文件（了解标的）
   │
   ▼
检查数据是否满足框架需求，不满足时：
   │ ① 检查 manifest，数据已有 → 直接读
   │ ② 数据不在 manifest → 调引擎 --output-dir data/，追加 manifest
   │    （遵循 §② 数据获取协议）
   ▼
写 analysis.md
   │
   ▼
正常返回（无论数据是否完整）
```

### 约束

- 子代理**不知道**其他经理的存在 → 不读他人 method.md，不读他人已产出的 analysis.md
- 子代理**不知道**自己被点名还是被补充进来的
- 补数据目标目录始终是全局 `data/`，不是自己的输出目录
- 子代理之间不检查彼此目录——通过 `manifest.md` 避免冗余调用
- 数据不足以支撑分析 → 能写多少写多少，如实标注缺失部分，正常返回
- 引擎调用失败 → 标记"XX 数据缺失"，继续写分析，不抛给父进程
- 最终产出：`<输出目录>/analysis.md`

### analysis.md 输出要求

- **第一人称叙事** — 以经理的口吻写（例："我看持仓……"），每个文件独立成篇
- **不区分 原话/推演** — 框架观点自然融入论述，不标注来源（时间、媒体均不出现）。所有关于经理观点、方法、判断的陈述，都必须在 `method.md` 中有对应依据；不得编造经理未持有过的观点。
- **需核实** — 数据类存疑仍标注，如 `（需核实）`
- **自由风格** — 不固定模板，不要求多人结构一致
- **不做评分** — 不给分/不给排名/不给维度加权总分
- **不做维度对齐** — 每个经理用自己的框架表述，不强求统一维度列表
- **证据链可见** — 看法背后有数据支撑，引用 `data/` 中的数据

## ⑤ 摆出透镜

所有子代理完成后，父进程**读各 analysis.md**，为每位经理提炼一段话判断，写到 `summary.md`：

```
fund-managers-output/
└── 2026-06-26_1430/
    ├── data/
    │   ├── manifest.md
    │   ├── 001513_holdings.json
    │   ├── 001513_nav.json
    │   ├── 001513_scale.json
    │   └── 001513_managers.json
    ├── 郑希/
    │   └── analysis.md
    ├── 姜诚/
    │   └── analysis.md
    ├── summary.md         ← 父进程汇总
```

summary.md 不做综合判断，只做每人一段话的提炼（1-3 句，不超过一个自然段）：

```
## summary

分析标的：001513 易方达信息产业混合
分析经理：郑希 · 姜诚 · 张坤 · 葛兰

郑希：持仓方向符合框架，AI 算力链占比高，景气方向明确。但流动性条件偏紧，需要持续跟踪。
姜诚：前十大集中度偏高但个股质地扎实，安全边际足够。行业分散保护了好的一面，整体风险可控。
张坤：持仓集中度偏高，且个股 ROIC 韧性存疑，安全边际不够。需要更谨慎地评估回撤保护。
葛兰：医药占比高但标的多为创新药龙头，政策风险可控。估值处于历史低位区间，有一定安全边际。
```

父进程不对 summary 做增删改写，只从各 analysis.md 提炼。

## 讲解

用户只想了解某位经理的投资方法（不涉及特定标的）。

1. 读对应经理的 `references/<经理>/method.md`
2. 每条都有原话佐证，引用时自然标注来源（时间+出处）
3. **不要把文件路径、章节号写入回答**

---

## 推演

用某经理的框架分析他本人没有公开评论过的行业/主题/个股。

1. 读该经理的 `method.md` 掌握框架
2. 开头写一句声明：**"以下内容按 XX 的方法推演，不是他的原话"**，后续不再每句重复
3. 原话仍有来源时正常标注
4. 需核实的数字标 **"需核实"**
