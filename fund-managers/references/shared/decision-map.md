## #7: 通用方法蒸馏模板

Blocked by: #3, #4, #5, #6

### Question

郑希的 `method.md` 是手工蒸馏的。5-6 位经理，每人一份，需要一套半自动化的蒸馏模板和流程来保证一致性。

关键设计问题：
- 模板结构：是否统一为"一句话框架 → 选股维度 → 风险观 → 行为特征"？
- 原话佐证的粒度：每条都要配原话？还是允许有些维度是归纳而非引用？
- 一致性校验：如何保证 A 经理和 B 经理的方法描述颗粒度相当？

产出：一份 `method_template.md` + 填充说明。

### Answer

模板已创建：`references/method_template.md`

结构设计（7 个必填模块 + 1 个可选 + 1 个附录）：

| 模块 | 必填 | 说明 |
|------|------|------|
| 一句话框架 | ✔ | ≤60 字，一眼能区分该经理 |
| 哲学基础 | ✔ | 他相信什么、不相信什么 |
| 选股维度 | ✔ | 3-5 个，配原话+数字阈值 |
| 特有框架/视角 | ✔ | 最独特的工具（如周蔚文"点线面体"） |
| 买点与卖点 | ✔ | 入场时机、卖出条件、持有期 |
| 组合管理 | ✔ | 集中度/仓位/行业数/换手率，用表格 |
| 风险观 | ✔ | 怎么理解风险、怎么控回撤 |
| 额外维度 | ✘ | 经理特有维度 |
| 可操作流程 | ✔ | 从研究到决策的步骤串 |
| 语料来源附录 | ✔ | 核心语料交叉引用 |

一致性规则：
- **每条判断必须配原话引用**，不允许无证据的归纳
- 原话后用正常文字做要点提炼（来源标注到具体采访/报告）
- "选股维度"统一控制在 3-5 条（太多碎片化，太少不够用）
- 组合管理统一用表格输出，确保各项指标可以直接横向对比
- 可操作流程统一用编号列表，把抽象框架落地为步骤

## #8: 标的分析引擎设计

Blocked by: #7
Type: Prototype

### Question

输入一只基金或个股 → 输出 N 个经理视角的独立分析。核心设计：

- 数据准备层：基金走已有 `fetch_any_fund` 逻辑，个股需要新数据源（财务数据、估值指标、行业分类）
- 分析层：每位经理的维度框架对应一组分析逻辑（有的是数值计算，有的是文本判断）
- 输出层：并排对比矩阵

需要 prototype 验证：
- 个股数据源选哪个（东方财富 API？Tushare？AKShare？）？
- 维度分析的自动化程度有多高（哪些可算、哪些得人工判断）？

### Answer

设计文档已创建：`references/engine_design.md`

核心设计决策：

**数据层：**
- 基金：复用 `fetch_any_fund.py`（已有，直接可算前十大集中度、换手率代理、行业分布）
- 个股：推荐 AKShare（免费、无需 token、覆盖率好），需要 ROE/ROIC/FCF/负债率/PB/PE 历史分位
- 统一数据接口向下传递

**分析层（5 独立模块，维度不统一）：**
- 郑希：沿用现有 6 维度 100 分制（已有 `score_fund.py` 和 `scorecard.md`）
- 丘栋荣：3 条件校验（低风险+低估值+高弹性）→ 三条件达标率
- 张坤：ROIC 初筛通过率 + 安全边际 + 基础回报率（股息+回购）+ 期权价值
- 刘格菘：供需格局 + DCF 四阶段定位 + 全球比较优势 + 错误定价机会 → 5 分制契合度

| 用途 | 接口 | 状态 |
|------|------|------|
| 财务摘要（ROE/净利润/营收） | `ak.stock_financial_abstract_ths(symbol, indicator)` | ✅ 可用（同花顺后端，过 proxy） |

问题是：要不要给"综合结论"？还是各经理独立判断，不做加权平均？

### Answer

**不做加权总分。** 理由：
- 各经理维度不同（郑希的25分≠丘栋荣的10分），强行加总无意义
- 使用者的需求是"X 经理会怎么看待这个标的"，不是"谁的评分高"
- 统一分数反而会让用户忽略维度差异

输出原则：
1. **各经理独立**：用自己的维度体系评分（不统一分数制）
2. **不排名**：我们不是评级机构，是"X 经理会怎么想"的模拟
3. **判语必须定性**：每格配"一句话判语"（文字定性>数字定量）
4. **证据链可追溯**：每个分数/判断都附数据引用
