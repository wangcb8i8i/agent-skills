# Glossary — SQL Decompile

本技能核心术语的定义。**加粗术语** 在本文件中另有定义。

## Decompile

本技能的 **leading word**。在软件工程中，反编译（decompile）指的是将机器码/字节码恢复为可读的源代码。在这里的映射：**SQL 是编译后的业务逻辑**——业务需求经过数轮开发、建模、优化后凝固为 SQL 文本；decompile 逆向这个过程，从 SQL 结构中恢复原始的业务意图。

Decompile 不为 SQL 语法加注释——那是每个开发者都能做的——它重建的是"为什么这个查询长这样"的业务上下文。一个 decompile 合格的输出，是读者能合上 SQL 用自己的话复述查询的业务目的。

_避免_：解释、翻译、分析、加注释

## Grain（业务粒度）

结果集中每一行所代表的业务单位。由 GROUP BY 列定义。一个 grain 描述是回答"每一行是什么"的唯一答案。

> GROUP BY user_id, status → grain = "一个用户在一种状态下的汇总"

不正确的 grain 理解是 SQL 误解最常见的来源。

_避免_：粒度、聚合级别

## Business Entity（业务实体）

数据库中一个表所代表的真实世界概念——"订单"、"用户"、"商品"——而非表名本身。同一个表在不同上下文中可能映射到不同的实体（`order` 在进销存中 = 采购订单，在电商中 = 销售订单）。

实体映射是 decompile 的第一个输出产物，也是后续所有步骤的基础。

_避免_：表、对象、模型

## Business Relationship（业务关系）

两个实体之间的关联在 SQL 中的编码形式——由 JOIN 的类型和条件定义。关系映射回答：为什么这两张表连在一起、业务上这意味着什么。

_避免_：关联、连接

## Business Measure（业务度量）

聚合函数计算出的业务 KPI——COUNT 是"条数"，SUM 是"总和"，AVG 是"均值"。每种度量都有业务单位。

_避免_：指标、聚合值

## Business Attribute（业务属性）

表中代表实体某一侧面数据的列——"名称"、"状态"、"金额"、"时间"等。属性映射回答：这个列存的是什么业务信息。

_避免_：字段、列

## Business Filter / Rule（业务过滤规则 / 业务规则）

WHERE 子句中的每个谓词所代表的业务选择逻辑。目标不是复述条件，而是回答"业务上为什么只保留这些"。

_避免_：条件、过滤

## Intermediate Concept（中间业务概念）

CTE 或子查询包装的临时结果集，代表了业务逻辑分解中的一个自然步骤。识别出它——而不是复述 SQL——才是 decompile 的增值点。

_避免_：子查询、中间表、临时结果

## Schema Layer（数仓分层）

常见的数仓分层命名约定及其业务含义：

| 层 | 含义 | 特征 |
|---|---|---|
| ODS | 原始数据层，贴源入仓 | 数据结构与源系统一致，无业务逻辑变更 |
| DWD | 明细层，清洗后的事实数据 | 去重、去噪、标准化后的业务明细 |
| DWS | 汇总层，按维度聚合 | 来自 DWD 或 DWS 的上卷聚合 |
| ADS | 应用层，按业务需求定制 | 直接对接报表和应用的最终数据 |

_避免_：层、层级

## Confidence（置信度）

每一行实体映射的可靠程度。使用三档标注：

| 档位 | 含义 | 判断条件 |
|---|---|---|
| 高 | 有直接证据 | DDL 注释、列名明确语义、会话上下文已确认 |
| 中 | 有间接证据 | 别名惯例、命名模式、多列交叉印证 |
| 推测 | 无直接依据 | 纯从表名推导 |

标注推测不代表质量低——明确标注不确定性本身就是质量。

_避免_：可能性、猜测
