# 可视化模式 —— SQL Decompile 输出图集

本文件定义 SQL decompile 输出中使用的 ASCII 可视化模式。每个模式解决一类数据流的可视化需求。

## CTE Chain（CTE 链）

展示数据在命名 CTE 之间的流动。适合以 `WITH ... SELECT` 结构为主的查询。

```
                    ┌──────────────────────┐
                    │  ods_order           │  ← 源表 1（实体：订单原始数据）
                    └────────┬─────────────┘
                             │
                    ┌────────▼─────────────┐
                    │  dwd_order_clean     │  ← CTE 1（清洗：去重、格式标准化）
                    └────────┬─────────────┘
                             │ LEFT JOIN
                    ┌────────▼─────────────┐
                    │  dim_user            │  ← 源表 2（实体：用户维度）
                    └────────┬─────────────┘
                             │
                    ┌────────▼─────────────┐
                    │  paid_orders         │  ← CTE 2（过滤：status = 'paid'）
                    └────────┬─────────────┘
                             │
                    ┌────────▼─────────────┐
                    │  final               │  ← 最终 SELECT（按用户聚合订单金额）
                    └──────────────────────┘
```

### 何时用

- 查询包含 ≥2 个 CTE
- 查询有嵌套子查询（可展开为扁平的链状结构）
- 数据流是线性或近乎线性的

### 规则

- 每层至少标注实体名或业务含义
- 在变换节点标注操作类型（清洗 / 过滤 / 聚合 / JOIN）
- 箭头方向 = 数据流动方向 = 执行方向

## Join Tree（JOIN 树）

展示多个表之间的 JOIN 关系。适合"星型"或"雪花型"多表关联。

```
                    ┌─────────────┐
                    │   订单      │  ← 事实表（核心）
                    └──────┬──────┘
                           │
          INNER ┌──────────┼──────────┐ LEFT
                │          │          │
       ┌────────▼───┐ ┌───▼────────┐ ┌▼──────────┐
       │   用户     │ │   商品     │ │   支付    │
       │ (维度表)   │ │ (维度表)   │ │ (事实表)  │
       └────────────┘ └────────────┘ └───────────┘
```

### 何时用

- 查询中心是一张事实表 JOIN 多张维度表
- JOIN 数量 ≥3
- 需要展示 JOIN 类型差异（INNER vs LEFT）

### 规则

- 中心表（通常是行数最多的表）放在顶部或中心
- 每条线标注 JOIN 类型
- 每个节点标注实体名和表类型

## Aggregation Pipeline（聚合管道）

展示从明细到聚合的变换过程。适合含 GROUP BY 的查询。

```
[明细层]
  dwd_order                        ← 每行 = 一笔订单
      │
      │ GROUP BY user_id
      ▼
[聚合层]
  user_order_stats                 ← 每行 = 一个用户
      │
      │ COUNT(*)     → 订单量（笔）
      │ SUM(amount)  → 总金额（元）
      │ AVG(score)   → 均分（分）
      │
      │ HAVING COUNT(*) > 5
      ▼
[输出层]
  final                            ← 只保留订单量 >5 的用户
```

### 何时用

- 查询含 GROUP BY + 多个聚合
- 需要展示 grain 的变化（"每行从 A 变成 B"）
- 含 HAVING 过滤

### 规则

- 用层标题标注 grain（[明细层] / [聚合层] / [输出层]）
- 聚合列标注业务含义 + 单位
- HAVING 标注在过滤发生的位置

## 单表过滤流

适合简单的单表查询。紧凑格式。

```
[dwd_order] ── WHERE status='paid' ──> [输出：已支付订单]
                │
                │ AND created_at >= '2024-01-01'
                │
                └──> 结果：2024 年后的已支付订单
```

### 何时用

- 单表查询
- WHERE 条件 ≤3 个

---

## 子查询展开

适合有 EXISTS / IN 子查询的查询。展示其语义等价形式。

```
SELECT * FROM orders o
WHERE EXISTS (
  SELECT 1 FROM payments p
  WHERE p.order_id = o.id AND p.status = 'success'
)

     ──── 等价于 ────

[orders] ── INNER JOIN ──> [payments (status='success')]
                               │
                               └── 只保留有成功支付记录的订单
```

### 何时用

- 含 EXISTS / NOT EXISTS / IN / NOT IN
- 子查询不复杂（无需独立展开为完整 CTE Chain）

### 规则

- 仅当 EXISTS/IN 子查询逻辑简单时才用此模式
- 复杂的子查询 → 先用 CTE Chain 独立展开，再引用
