## 步骤类型识别

定位结构关键字作为步骤分界，按以下类型标注：

| 类型 | 判定 | 输出特征 |
|---|---|---|
| **数据集** | `SELECT INTO` / `FETCH` / 游标操作 | 附 SQL 摘要 |
| **决策** | `IF` / `ELSIF` / `CASE` | 附分支数 + 各分支业务含义 |
| **循环** | `LOOP` / `FOR` / `WHILE` / 游标 FOR LOOP | 附每次迭代的业务单位 |
| **DML** | `INSERT` / `UPDATE` / `DELETE` / `MERGE` | 附影响的实体和列 |
| **调用** | 调用其他过程/函数 | 标注被调方名称 |
| **事务** | `COMMIT` / `ROLLBACK` / `SAVEPOINT` | 附事务边界含义 |

## 依赖分类与展开

对外部引用做三档分类：

| 依赖类型 | 判定标准 | 处理方式 |
|---|---|---|
| **工具依赖** | 不含业务逻辑——日志、格式化、简单 getter | 标注名称。`标记：[工具] log_utils.log_info` |
| **业务依赖** | 含影响假设的规则——计算、校验、决策、状态变更 | 展开调用方合约（见下） |
| **副作用依赖** | 表上触发器、物化视图刷新、异步队列 | 标注存在。`标记：[副作用] refunds 表含 INSERT 触发器` |

### 业务依赖展开规则

- **范围**：只展开一级（直接调用），不递归。依赖内部的依赖不追
- **深度**：只提取调用方合约——业务规则（一句话）、输出含义、可见副作用
- **停止条件**：框架不再依赖黑盒。如果替换为「不知道内部做什么」会让框架站不住脚，就继续提取

提取内容：

| 提取项 | 粒度 | 例 |
|---|---|---|
| 业务规则 | 一句话 | "余额 = 账户余额 − 冻结金额 − 在途资金" |
| 输出含义 | 返回值/OUT 参数的业务语义 | "返回 ≥0 表示可退款金额，<0 表示透支" |
| 可见副作用 | 调用方需感知的状态变化 | "会写 audit_log，不影响主流程" |
