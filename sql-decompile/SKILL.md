# SQL / PL/SQL Decompile

将复杂 SQL 或 Oracle PL/SQL **decompile** 为业务含义，让不懂业务的技术人在脑中运行这个心智模型。

**加粗术语** 在 [`GLOSSARY.md`](GLOSSARY.md) 中定义。可视化模式在 [`references/patterns.md`](references/patterns.md) 中定义。

> **上下文提示**：若项目根目录存在 `.recon/RECON.md`，先读取以快速建立项目背景和业务域语境。

## 输入检测

自动检测输入类型并路由到对应分支：

| 输入特征 | 路由 |
|---|---|
| `SELECT` / `INSERT` / `UPDATE` / `DELETE` / `MERGE` / `WITH` 开头（独立语句） | **分支 A：SQL** |
| `CREATE [OR REPLACE] PROCEDURE\|FUNCTION\|PACKAGE\|TRIGGER` | **分支 B：PL/SQL** |
| `DECLARE` / `DECLARE ... BEGIN ... END` / 裸 `BEGIN ... END`（匿名块） | **分支 B：PL/SQL** |
| `BEGIN` 内含 `SELECT ... INTO` / `EXECUTE IMMEDIATE` | **分支 B：PL/SQL** |
| 混合：CREATE TABLE + INSERT 批量脚本 | 按**主要意图**判断 |
| 不确定（片段不完整） | 提问用户 |

检测输出：`输入识别为 {SQL | PL/SQL}，进入 {分支 A | 分支 B} 模式。`

## 阶段一 · 分类
#### ① 确定模式

| 用户说 | 模式 |
|---|---|
| "在做什么"、"解释这段代码"、裸代码文本 | **Decompile** — 完整业务语义反编译 |

#### ② 范围确认

| 特征 | 策略 |
|---|---|
| `< 30` 行 | 全量 **decompile**，一次性输出 |
| `≥ 30` 行 | 先输出骨架图，标注"详情逐层展开"；之后每层单独输出 |
| 含外部调用（UDF / 其他过程调用） | 标记 `[外部依赖：{名称}]`，不展开被调用方内部逻辑 |
| 无 DDL 可用且命名无业务线索 | 标注 `推测语义`，低置信度输出 |

**完成标准**：已输出「输入识别为 {SQL | PL/SQL}，进入 {分支 A | 分支 B} 模式。」确认语句。

## 阶段二 · 业务重建

核心原则：**不逐行读代码。先建全景，再用细节验证。** 不要从 FROM 开始——那是最差的起点。从最能揭示业务意图的线索开始扫描，形成假设，再用代码细节验证。

### 分支 A：SQL 业务重建

#### ① 业务扫描 → 场景假设

按信号密度从高到低扫描表层线索：

| 扫描序 | 线索 | 要回答的问题 |
|---|---|---|
| 1 | SELECT 输出列 | 这段代码在产出什么？报表？指标清单？一行一个业务对象？ |
| 2 | GROUP BY | 每行代表什么业务单位？ |
| 3 | WHERE 关键过滤 | 排除了什么业务状态？什么时间范围？ |
| 4 | 表名（FROM） | 属于哪个业务域？ |
| 5 | 聚合函数 | 在算什么度量？ |

产出：**一句业务假设**（≤50 字）+ 置信度标注。这是阶段三「一句话摘要」的来源。

> 例："这段代码在计算每个用户在过去一年的已支付订单总金额和订单数"（置信度：高——SELECT 列、聚合、WHERE 时间范围互相印证）

**完成标准**：能用一句话说出业务问题，且与 ≥2 个表层线索自洽（如 SELECT 列 + WHERE 范围互相印证）。

#### ② 验证假设 → 逐层对照

用代码每一层检验假设。每层问：**这段逻辑支撑假设，还是和假设矛盾？** 矛盾优先——发现矛盾时修正假设，重新对照。

| 层 | 验证问题 | 支撑例 | 矛盾例 |
|---|---|---|---|
| FROM | 数据来源是否属于假设的业务域？ | 假设"订单汇总"，FROM `orders` ✓ | FROM `inventory`——这是库存域，假设可能错了 |
| JOIN | 关联方式和假设一致吗？ | LEFT JOIN users → 用户可能不存在，合理 | CROSS JOIN → 笛卡尔积，假设可能漏了维度 |
| WHERE | 过滤范围和假设一致吗？ | `status='paid'` → 确实只管已支付 ✓ | `type='refund'` → 居然还含退款，假设需修正 |
| GROUP BY | 粒度匹配吗？ | `GROUP BY user_id` → 每行一个用户 ✓ | `GROUP BY user_id, month` → 比假设多一个维度 |
| 聚合 | 衡量指标和假设一致吗？ | `SUM(amount)` → 金额 ✓ | `AVG(rating)` → 还有评分？假设不完整 |
| HAVING | 什么阈值？为什么？ | `HAVING COUNT(*) > 5` → 过滤低频用户 | 无 HAVING 跳过 |
| SELECT | 输出列匹配假设用途吗？ | 列和假设一致 ✓ | 多了 `refund_count` → 假设需要扩展 |
| ORDER BY | 为什么这样排？ | `ORDER BY total_amount DESC` → 关注高消费用户 | 无 ORDER BY 跳过 |
| CTE/子查询 | 为什么需要这个中间步骤？ | CTE 清洗数据 → 中间加工，合理 | CTE 引入新数据源 → 假设范围可能不够 |

**完成标准**：假设与代码细节之间无矛盾。如有矛盾，已修正假设。

#### ③ 数据流 → 可视化

根据查询结构选择模式（详见 [`references/patterns.md`](references/patterns.md)）：

| 特征 | 模式 |
|---|---|
| ≥2 个 CTE 或嵌套子查询 | CTE Chain |
| 一张事实表 JOIN ≥3 张维度表 | Join Tree |
| 含 GROUP BY + 多个聚合 | Aggregation Pipeline |
| 单表 + ≤3 个 WHERE 条件 | 单表过滤流 |

**完成标准**：数据流图模式已选定并绘制。门槛见阶段三·最小视觉门槛。

### 分支 B：PL/SQL 业务重建

PL/SQL 是命令式的——业务意图分散在多个维度上。假设形状不是一个句子，而是一个**四槽框架**：触发场景、操作、输入/输出、范围边界。

#### ① 业务扫描 → 填充框架

按信号密度扫描：

| 扫描序 | 线索 | 填充哪个槽 |
|---|---|---|
| 1 | 过程/函数名 | **操作**：什么业务动作？ |
| 2 | 参数名 + 方向（IN/OUT） | **输入/输出**：吃什么？吐什么？ |
| 3 | 关键 DML（INSERT/UPDATE/DELETE 的目标表） | **操作**（细化）、**范围**：改了哪些实体？ |
| 4 | 异常名 + RAISE 语句 | **边界**：什么情况视为失败？ |
| 5 | COMMIT/ROLLBACK | **范围**：事务原子边界在哪？ |

产出：四槽业务框架 + 置信度标注。

> 例：`process_refund(p_order_id, p_amount OUT)`
> - 触发：被外部调用（非定时） / 操作：处理退款——校验余额，扣款，记录 / 输入：订单ID、退款金额 → 输出：处理状态码 / 范围：涉及 orders、refunds；不处理部分退款（无 partial 标记）

**完成标准**：能说清触发场景、操作、输入/输出、范围。

#### ② 验证框架 → 对照代码

分三个维度验证。

**主流程验证**：按「一切顺利」路径走一遍，跳过异常分支。定位结构关键字作为步骤分界，逐段对照框架：

| 对照项 | 验证问题 |
|---|---|
| 游标/数据集 | 操作的数据窗口和框架一致吗？ |
| 循环体 | 每轮迭代的业务单位和框架一致吗？ |
| IF/CASE | 分支条件在框架范围内吗？ |
| DML | 改的表在框架实体列表内吗？ |

步骤类型识别详见 [`references/plsql-verification.md`](references/plsql-verification.md)。

**分支/异常验证**：

| 对照项 | 验证问题 |
|---|---|
| IF/CASE 各分支 | 每个分支的业务含义和框架一致吗？有没有框架没覆盖的分支？ |
| EXCEPTION handler | 失败模式和框架边界一致吗？有没有悄悄吞掉的异常？ |

**依赖验证**：对外部引用做三档分类，详见 [`references/plsql-verification.md`](references/plsql-verification.md)。

**完成标准**：框架与代码无矛盾；业务依赖调用方合约已提取。

#### ③ 决策与异常 → 可视化

从 IF/CASE/ELSIF 提取决策树。展示**业务条件 → 业务走向**，而非复现代码。深度 ≤5 级。

从 EXCEPTION 提取异常恢复表（格式见 [`references/plsql-verification.md`](references/plsql-verification.md)）：

**完成标准**：能说清「什么条件走哪条路」；EXCEPTION handler 全部标注触发条件和恢复策略。

#### ④ 副作用全景

| 类别 | 内容 |
|---|---|
| DML 影响 | 改了哪些表、什么业务影响 |
| 事务边界 | COMMIT/ROLLBACK/SAVEPOINT 标记的原子操作范围 |
| 依赖清单 | 所有被引用的表/视图/函数/过程 + 分类标注 |

**完成标准**：实体和事务边界无遗漏。

## 阶段三 · 输出

以验证后的假设为骨架，以代码证据为血肉。**叙事优先，不是标签列表。**

### 叙事顺序

SQL：**业务全景**（阶段二①产出）→ **关键业务规则**（JOIN/WHERE/HAVING 的业务原因）→ **数据流图**（阶段二③）→ **粒度与度量**（每行是什么、在算什么）

PL/SQL：**业务全景**（阶段二①产出）→ **主流程叙事**（阶段二②正常路径步骤）→ **决策与异常**（阶段二③）→ **副作用全景**（阶段二④）

### 共享约束

- 对话输出 ASCII 图，禁 mermaid。写文件时才用 mermaid
- SQL 关键字保留英文，业务映射用中文
- 类比必须附带失效声明：`这个类比在 {映射点} 上成立，但在 {失效处} 不成立`

### 最小视觉门槛

- SQL 分支：≥1 张数据流图；例外：≤15 行且无子查询
- PL/SQL 分支：≥1 张业务流程图；例外：≤20 行

## 阶段四 · 追问循环

| 用户说 | 处理 |
|---|---|
| "X 部分再细点" | 同概念深度输出更细粒度 |
| "那 Y 呢" | 范围迁移，重新分类 |
| "继续" | 输出下一个未展示的块 |

**规则**：
- 追问无深度上限。第 3 次递归起，末尾附方向提示：`继续往下是 {边界条件/极端情况/错误路径}，要继续吗？`
- 用户说"继续" → 解除提示
- 用户沉默 → 结束。不主动问"还有问题吗"

---
## 硬边界

不得：
- 输出修改建议、优化建议、批评或审查
- 在未读取代码文本的情况下作答
- 输出填充语、自检过程、内部流程标签
- 将推断呈现为已确认的事实——必须标注置信度
- 解释超出用户要求范围的内容
- 回答后主动评价（"理解了吗"）
- 按行复述代码——必须按业务逻辑块组织
