# Recon Map 产出规范

> 被 [project-recon](SKILL.md) 引用的格式参考。Recon Map 核心是写给项目新读者看的——让一个人能在最短时间内抓住关键信息。只在你需要写文件或检查格式时查阅本节。

## 文件结构

目标项目根目录下 **`.recon/`** 子目录，集中存放所有产出物：

```
{目标项目}/
└── .recon/
    ├── RECON.md                 ← Overview + Coverage 表（唯一索引）
    ├── glossary.md              ← 项目专用术语
    ├── architecture.md          ← 粗架构图 + 工程模块清单 + 核心抽象 + 数据模型
    ├── user-journeys.md         ← 用户旅程（每步标注负责实体）
    ├── entry-points.md          ← 运行时入口清单 + 配置
    ├── dev-loop.md              ← 构建/测试/lint/CI 命令与环境搭建
    └── modules/                  ← 第二轮按需产出
        ├── {区域}.md            ← 第一块 + 流转 + 洞察
        ├── {区域}-深入.md       ← 代码层深入
        └── {子系统}/            ← 大型项目：按子系统分组
            ├── {区域}.md
            └── {区域}-深入.md
```

不写 `.gitignore`——Recon Map 默认进入版本控制，可被团队共享和审查。如果决定后续排除，一行 `.recon/` 到 `.gitignore` 即可。

## user-journeys.md 格式

一个 user-journeys.md 通常包含：
- **用户画像**：谁在用、角色、目标
- **旅程**：从触发到完成的端到端路径。每步是用户推进价值的进展节点，标注负责实体——用名称（→ auth），不写类型后缀或文件路径

（首轮侦察时一次性产出。）

## modules/{区域}.md 格式

每个区域一个 `.recon/modules/{区域}.md`，在用户首次指向该区域时一次性产出三块。深入不在本文件中——参见下方 `{区域}-深入.md`。

三块按理解递进关系依次产出：

**第一块：背景（按需）+ 职责 + 生态位**
回答"这个区域是干什么的"。读者能据此说清该区域在系统中的角色和边界。
- **证据标记**（必写）：从 architecture.md 架构图对应节点复制证据来源（` ✓` 多文件交叉验证、`？` 从代码推断、` ○` 外部依赖未检查），格式为 `证据：✓`
- **背景**（按需）：理解本区域职责所需、但区域自身代码不包含的前置知识。落笔标准——读者阅读职责时被项目内部的抽象概念或领域术语拦住——即为背景必要
- **职责 + 生态位**：用 2-5 句描述该区域负责什么、与哪些其他模块交互。上下游用模块名（→ orders），不写文件路径

**第二块：流转**
回答"这个区域怎么干的（不看代码也能看懂）"。聚焦于外部行为模式，不触及实现细节。用领域语言编写，不用实现语言（函数名、方法签名、参数名、调用路径）。写完后删除所有实现语言词汇；如果剩余文本不成一个有意义的行为描述，说明抽象层不够。

**第三块：洞察（0-2条）**
回答"剥离了该项目选择后，这个区域在做的本质是什么"。换一种实现方式，这个"本质"仍然成立。没有则跳过，不凑数。

## modules/{区域}-深入.md 格式

用户确认深入时产出。每个深入区域一个独立文件，内容回答"内部怎么做"。

- **内部结构**：组件、层、抽象边界、连接关系
- **代码路径**：核心路径追踪，精确到 `文件名::函数名`
- **错误处理**：不只 happy path，包含错误处理路径
- **设计模式**：观察到的模式 + 效果。有文档/注释支撑的标注来源；纯从代码推断的不编造动机


### 区域拆分

当第一块、流转或深入描述的是「区域内的某个子实体」而非区域本身时，迁到子文件 `modules/{区域}/{子实体}.md`。不靠行数或实体数量，靠内容与标题抽象层是否对齐。

开始写一个区域前，若出现以下信号，应先创建子文件，而不是在父文件中用大段内容覆盖：

- 单一区域内包含多个可独立理解的职责或执行路径；
- 流转需要按不同来源、不同阶段分别描述；
- 区域内存在明显的分层（如接入层、调度层、执行层）。

拆分决策应在创建 `modules/{区域}.md` 之前做出，避免写完后才迁移。子文件路径统一为 `modules/{区域}/{子实体}.md`；不得通过父文件内的大量子标题来替代独立子文件。

## RECON.md 格式

### Coverage 表

| 区域 | 状态 | 最后访问 |
|------|------|---------|
| 认证 | basic | 2026-07-17 |
| 消息推送 | deep | 2026-07-16 |

### 字段规则

**区域**：用户指定的侦查区域——可以是工程模块（如"认证"），也可以是横切概念（如"日志系统"）。按用户的理解单元来，不是代码目录。

**状态**：
- `basic`：`modules/{区域}.md` 已产出——概念层理解已完成。该区域的背景/职责/生态位、流转、洞察三块已写。读者能在不读代码的情况下理解该区域"干什么"和"怎么干的"。
- `deep`：`modules/{区域}-深入.md` 已产出——代码层理解已完成。在 basic 基础上深入到了代码实现——内部组件、代码路径、错误处理、设计模式已测绘。

**最后访问**：最后一次操作日期。仅作为时间戳记录。

**子表**：大区域可展开为子表——主表该行标记 `→`，子表在同一文件下面另起一节，子区域行缩进。

```
| 消息推送 | basic | 2026-07-17 |
|   → 消息组装 | basic | 2026-07-17 |
|   → 渠道适配 | deep | 2026-07-17 |
```
