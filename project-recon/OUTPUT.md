# Recon Map 产出规范

> 被 [project-recon](SKILL.md) 引用的披露参考。Recon Map 核心是写给项目新读者看的——让一个人能在最短时间内抓住关键信息。只在你需要写文件或检查格式时查阅本节。

## 文件结构

目标项目根目录下 **`.recon/`** 子目录，集中存放所有产出物：

```
{目标项目}/
└── .recon/
    ├── RECON.md                 ← Overview + 覆盖状态表（唯一索引）
    ├── glossary.md              ← 项目专用术语
    ├── architecture.md          ← 模块/组件间依赖关系（含外部系统）+ 核心抽象 + 数据模型 + 设计决策
    ├── user-journeys.md         ← 用户旅程（每步标注负责实体）
    ├── entry-points.md          ← 运行时入口清单（main、CLI、路由注册、导出API）+ 运行时配置
    ├── dev-loop.md              ← 构建/测试/lint/CI 命令与环境搭建
    ├── design-decisions.md      ← 设计决策（跨模块统一归档，深入了解模块时产出）
    └── modules/                  ← 模块专属文件（Step 3b 创建骨架，Step 4 增补流转/洞察，Step 6 产出深入）
        ├── auth.md              ← 骨架 + 流转 + 洞察
        ├── auth-深入.md          ← 深入（Step 6 产出）
        ├── orm.md
        ├── orm-深入.md
        ├── logging.md
        └── {subsystem}/         ← 大型项目：按子系统分组
            ├── {模块}.md
            └── {模块}-深入.md
```

不写 `.gitignore`——Recon Map 默认进入版本控制，可被团队共享和审查。如果决定后续排除，一行 `.recon/` 到 `.gitignore` 即可。

## user-journeys.md 格式

一个 user-journeys.md 通常包含：
- **用户画像**：谁在用、角色、目标
- **旅程**：从触发到完成的端到端路径。每步是用户推进价值的进展节点，标注负责实体——用名称（→ auth），不写类型后缀或文件路径

（首次侦察时产出。后续用户指向具体模块时按需补充。）

## modules/{模块}.md 格式

每个模块一个 `.recon/modules/{模块}.md`，内容按侦察深度逐步积累。深入不是本文件的一部分——参见下方 `{模块}-深入.md` 格式。

- **骨架**：首要目标是**导航定位**——读者能在这张认知地图上找到"那个东西在哪、关联了什么"。随后记录职责 + 上下游。上下游用模块名（→ orders），不写文件路径。深度与复杂度匹配——薄模块 2-3 句即可。不逐一罗列文件
  - **证据标记**（必写，骨架首行）：从架构图对应节点复制证据来源（` ✓` 多文件交叉验证、`？` 从代码推断、` ○` 外部依赖未检查），格式为 `证据：✓`。读者一眼知道该模块描述的可信度
  - **背景**（按需，骨架子项）：理解本模块职责所需、但模块自身代码不包含的前置知识。信息来源包括模型对第三方系统/协议/生态的知识，以及全项目扫描的跨文件发现。落笔标准——读者阅读本模块的职责或流转时，被项目内部定义的核心抽象或非通用的领域专用术语拦住——即为背景必要，不可跳过。
- **流转**：模块在系统中的角色描述——对外提供什么能力、接收什么输入、产出什么结果。聚焦于外部行为模式，不追溯内部步骤。使用领域语言编写，不用实现语言（函数名、方法签名、参数名、精确调用路径）。写完后删除所有实现语言词汇，如果剩余文本不成一个有意义的行为描述，说明抽象层不够，需要提升。

  **概念——流转用领域语言描述模块，不涉及实现语言。** 领域语言是模块在业务中扮演的角色（做什么、产出什么、在什么时机被调用），不是代码的内部路径（管线顺序、函数名、分支选项）。它的读者站在模块门外，阅读流转不应该要求他先打开内部代码。
- **洞察**（按需）：对模块的本质理解或思路的蒸馏——「说白了在干什么或设计思路是什么」。不超过两条。每条应非显而易见——盖住模块名后仍描述了某个具体特性，读者单从模块名无法预测。没有这样的条目则不写，不凑数。

## modules/{模块}-深入.md 格式

Step 6 产出。每个深入模块一个独立的 `.recon/modules/{模块}-深入.md`，内容回答"内部怎么做"。

- **内部结构**：组件、层、抽象边界、连接关系
- **代码路径**：核心路径追踪，精确到 `文件名::函数名`
- **设计模式**：观察到的模式 + 效果。有文档/注释支撑的标注来源；纯从代码推断的不编造动机


### 模块拆分

当流转或深入描述的是「模块内的某个东西」而非模块本身，迁到子文件 `modules/{模块}/{子实体}.md`。不靠行数或实体数量，靠内容与标题抽象层是否对齐。

开始写一个模块前，若出现以下信号，应先创建子模块，而不是在父模块中用大段内容覆盖：

- 单一模块内包含多个可独立理解的职责或执行路径；
- 流转需要按不同来源、不同媒体/数据类型、不同阶段分别描述；
- 模块内存在明显的分层（如接入层、调度层、执行层、结果解析层）。

拆分决策应在创建 `modules/{模块}.md` 之前做出，避免写完后才迁移。子模块文件路径统一为 `modules/{模块}/{子实体}.md`；不得通过父模块文件内的大量子标题来替代独立子文件。

## RECON.md 格式

### Module Coverage

| Module | 粒度 | 骨架 | 流转 | 洞察 | 深入 | Last Visit |
|--------|------|------|------|------|------|------------|
| auth/  | ◉    | ✓    | ✓    | ✓    | ✓    | 2026-06-27 |
| orm/   | ◉    | ✓    | ✓    | ○    | -    | 2026-06-27 |
| logging/ | ·   | ✓    | ○    | -    | -    | 2026-06-26 |

### 字段规则

**Module**：任何有独立职责和边界的实体——业务模块、横切关注点、非功能层、配置体系都在内。

**粒度**：`◎` = 粗，一条完整业务线/域，需要骨架+流转+洞察；`◉` = 中，有独立逻辑的子系统，需要骨架+流转；`·` = 细，原子关注点，骨架即完成。

**骨架**：`✓` = 背景(如有)+职责+上下游已写；`-` = 未写。

**流转**：`✓` = 已写；`○` = 评估后无需写；`-` = 未评估。

**洞察**：`✓` = 已写；`○` = 评估后无需写；`-` = 未评估。

**深入**：`✓` = 已写；`-` = 未写。

**Last Visit**：最后一次操作日期。超过 30 天且有最新 commit 变化则建议 revisit。

**子表**：大模块可展开为子表——主表该行标记 `→`，子表在同一文件下面另起一节，子模块行缩进。

```
| core/ | ✓ | ✓ | ✓ | ✓ |
|   skeleton | ✓ | ✓ | ✓ | ✓ |
|   adaptation | ✓ | ✓ | ✓ | ✓ |
```
