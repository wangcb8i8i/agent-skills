# Recon Map 产出规范

> 被 [project-recon](SKILL.md) 引用的披露参考。Recon Map 核心是写给项目新读者看的——让一个人能在最短时间内抓住关键信息。只在你需要写文件或检查格式时查阅本节。

## 文件结构

目标项目根目录下 **`.recon/`** 子目录，集中存放所有产出物：

```
{目标项目}/
└── .recon/
    ├── RECON.md                 ← Overview + 覆盖状态表（唯一索引）
    ├── glossary.md              ← 项目专用术语
    ├── architecture.md          ← 模块/组件间依赖关系（含外部系统）+ 核心抽象 + 数据模型 + 设计决策
    ├── user-journeys.md         ← 用户旅程（每步标注负责实体）
    ├── entry-points.md          ← 运行时入口清单（main、CLI、路由注册、导出API）+ 运行时配置
    ├── dev-loop.md              ← 构建/测试/lint/CI 命令与环境搭建
    ├── design-decisions.md      ← 设计决策（跨模块统一归档，深入了解模块时产出）
    └── modules/                  ← 模块专属文件（Step 3 创建骨架，Step 4/6 增补）
        ├── auth.md              ← 职责/上下游 + 流转 + 深入
        ├── orm.md
        ├── logging.md
        └── {subsystem}/         ← 大型项目：按子系统分组
            └── {模块}.md
```

不写 `.gitignore`——Recon Map 默认进入版本控制，可被团队共享和审查。如果决定后续排除，一行 `.recon/` 到 `.gitignore` 即可。

## user-journeys.md 格式

一个 user-journeys.md 通常包含：
- **用户画像**：谁在用、角色、目标
- **旅程**：从触发到完成的端到端路径。每步是用户推进价值的进展节点，标注负责实体——用名称（→ auth），不写类型后缀或文件路径

（首次侦察时产出。后续用户指向具体模块时按需补充。）

## modules/{模块}.md 格式

每个模块一个 `.recon/modules/{模块}.md`，内容按侦察深度逐步积累：

- **骨架**：职责 + 上下游。上下游用模块名（→ orders），不写文件路径。深度与复杂度匹配——薄模块 2-3 句即可。不逐一罗列文件
- **流转**：模块完成职责的概念性流程——输入什么、经过哪些处理阶段或数据变形、产生什么效果、交给谁。视角按模块特性定
- **洞察**（按需）：对模块的本质理解或思路的蒸馏——「说白了在干什么或设计思路是什么」。不超过两条
- **深入**：内部结构、组件、观察到的设计模式（模式 + 效果）。承接代码路径追踪，精确到 `文件名::函数名`。有文档支撑的设计理由标注来源；无从代码推断的不编造动机。


### 模块拆分

当流转或深入描述的是「模块内的某个东西」而非模块本身，迁到子文件 `modules/{模块}/{子实体}.md`。不靠行数或实体数量，靠内容与标题抽象层是否对齐。

## RECON.md 格式

### Module Coverage

| Module | 骨架 | 流转 | 洞察 | 深入 | Last Visit |
|--------|------|------|------|------|------------|
| auth/  | ✓    | ✓    | ✓    | ✓    | 2026-06-27 |
| orm/   | ✓    | ✓    | ○    | -    | 2026-06-27 |
| logging/ | ✓  | ○    | -    | -    | 2026-06-26 |

### 字段规则

**Module**：任何有独立职责和边界的实体——业务模块、横切关注点、非功能层、配置体系都在内。

**骨架**：`✓` = 职责+上下游已写；`-` = 未写。

**流转**：`✓` = 已写；`○` = 评估后无需写；`-` = 未评估。

**洞察**：`✓` = 已写；`○` = 评估后无需写；`-` = 未评估。

**深入**：`✓` = 已写；`-` = 未写。

**Last Visit**：最后一次操作日期。超过 30 天且有最新 commit 变化则建议 revisit。

**子表**：大模块可展开为子表——主表该行标记 `→`，子表在同一文件下面另起一节，子模块行缩进。

```
| core/ | ✓ | ✓ | ✓ | ✓ |
|   skeleton | ✓ | ✓ | ✓ | ✓ |
|   adaptation | ✓ | ✓ | ✓ | ✓ |
```
