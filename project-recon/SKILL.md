# project-recon

**语言：所有 Recon Map 产出物使用中文。** 代码标识符保留原文。

对陌生项目做系统化**侦察**——目标不是"读代码"，而是持续产出一份可分享的 **Recon Map**。

核心假设：你不知道那里有什么。每次侦察都从外围到核心逐层推进，每层有明确完成标准阻止过早宣称"懂了"。

> Bold terms → 见 [GLOSSARY.md](GLOSSARY.md)

**module 在本技能中不限于代码目录。** 它指任何有独立职责和边界的实体——目录包（auth/）、横切关注点（logging）、非功能层（orm 封装）、配置体系、部署脚本。判断标准：这个实体用户是否可能想单独理解它。

## 工作方式

Agent 每次调用先读取 `.recon/RECON.md`（如存在）确定已完成的工作，再根据用户意图确定本次范围。

| 状态 | 用户意图 | Agent 做什么 |
|------|---------|-------------|
| 无 RECON.md | — | 首次侦察：Step 1→2→3→4(仅用户旅程)→5，产出 RECON.md |
| 有 RECON.md | 含"重新看""重审""变更"等词（"重新看看 X 模块""项目变了，重审""重新看看这个项目"） | **校准**：按用户指定粒度先读实际代码 → 对比 RECON.md → 更新过时内容或补采 |
| 有 RECON.md | 指向具体模块（"看看X""X怎么设计的"） | 读该模块覆盖状态：流转未写 → Step 4；流转已写、深入未写 → Step 6；全已写 → 告知已有深度，问是否重新审视 |
| 有 RECON.md | 指向具体模块，但模块不在 Coverage 表中 | 确认 X 是真实代码实体 → 补 Step 3（为 X 创建 `modules/{X}.md` 骨架、更新 `architecture.md`、RECON.md 增行）→ 按流转未写处理，执行 Step 4 |
| 有 RECON.md | 无具体目标（"还有什么可看"） | 汇报覆盖状态，指出缺口，建议"想看哪个模块？" |
| 有 RECON.md | "怎么跑""怎么开发""搭建环境" | Step 5 |

Agent 通过 `.recon/RECON.md` 的 Module Coverage 表管理状态。**汇报时永远不向用户暴露分支名称、Coverage 数值或步骤编号**——用户只需要知道项目本身。

> **校准**：与首次侦察（Step 1→2→3→4→5）不同，已有 RECON.md 基底。不从头走全流程，而是按用户指定粒度先读实际代码 → 再对比 RECON.md 差异 → 更新过时内容。三个粒度：函数/文件级读单个实现、模块级读核心文件后补流转或深入、项目级快速回访关键流程。

## 汇报规则

发现式汇报——汇报项目发现，不汇报侦察过程。

**中途定位**：首次侦察时 Step 1-2 完成后先给一句话定位（项目是什么 + 技术栈），不等全部跑完。

| 时机 | 汇报什么 | 长度 |
|------|---------|------|
| 首次侦察完成时 | ①定位 ②问题域 ③各部分职责 ④关键架构决策 ⑤快速开始 ⑥推荐入口。①②③⑤⑥落盘到 RECON.md Overview；④落盘到 architecture.md，汇报时提及但不在 RECON.md 展开 | 一段话，点序明确 |
| 深入模块完成时 | 该模块设计思路 + 关键 tradeoff + 意外发现 | 3-5 句 |
| 补漏完成时 | 补了哪些部分，各一句核心职责 | 每项 1-2 句 |
| 发现关键信息时 | 实际发现（如"README 与实际代码不符"） | 2-3 句 |

所有汇报只谈项目本身。不 dump 文件内容、不说"已完成""执行完毕"、不使用内部术语。

**目的：** 让对话输出成为 **"我知道了你该知道什么"** 的指引，而不是"文件已生成"的通知。用户读完汇报后，对这个项目已经有方向感；细节去 `.recon/` 里找。

## 六大步

**产出原则：每份产出文件的抽象层次对齐文件标题。** `modules/X.md` 的所有内容（流转、深入）回答的都是"X 这个模块"的问题。发现自己在描述 X 内部某个更低层的实体——那段内容属于对应粒度的子文件。

### 1. Perimeter — 外围侦察

| 动作 | 具体内容 |
|------|---------|
| 读 | README, CONTRIBUTING, CHANGELOG, LICENSE |
| 读 | 项目元数据 (package.json, Cargo.toml, pyproject.toml, go.mod 等) |
| 扫 | 目录树 2 层深（`tree -L 2` 或等价操作） |
| 录 | 项目专用术语（领域概念，如 MTOP、CookieCloud），按主题分组写入 `.recon/glossary.md`——每组独立二级标题，同类概念放一起。术语不含模块名称——模块名称属于 RECON.md 模块地图。其它文件出现术语时用 `→ glossary.md#{主题}` 回链 |

**完成标准**：读者看完产出后能用一句话说清"这个项目做什么、用什么技术栈、顶层目录各负责什么"。关键术语已在 glossary.md 中解释，可随时查阅。

**陷阱**：README 可能过时。交叉验证 README 说的 vs 代码实际暴露的。

### 2. Entry — 入口侦察

| 动作 | 具体内容 |
|------|---------|
| 找 | main 函数、CLI 入口、HTTP 路由注册、导出 API |
| 找 | 运行时配置——环境变量（.env*）、配置文件（config.*, .yaml, .toml）、CLI flags |
| 列 | 所有入口点的清单，写入 `.recon/entry-points.md` |

**完成标准**：读者读完 entry-points.md 后能看到完整的入口清单，清楚启动流程从哪开始、哪些入口是生产关键路径、启动前需要配置哪些环境变量和文件。

### 3. Architecture — 架构侦察

| 动作 | 具体内容 |
|------|---------|
| 建 | 为每个模块创建 `.recon/modules/{module}.md` 骨架，写**职责**+**上下游**（格式见 OUTPUT.md）。骨架不含文件罗列——文件层面信息属于深入 |
| 画 | 内部依赖方向 + 外部系统（数据库、第三方 API、消息队列等），写入 `.recon/architecture.md`（Mermaid 或 ASCII） |
| 找 | 核心抽象——traits、interfaces、abstract base classes、核心数据结构。描述概念本身（如"YAML 配置系统，三层合并优先级"），不标文件路径 |
| 看 | 数据模型——ORM schema、migration 文件、DDL。列出核心实体和关系，写入 `architecture.md` |
| 量 | 文件数、模块规模分布 |
| 记 | 关键架构决策及理由，写入 `architecture.md`「设计决策」节 |

**完成标准**：读者看完 architecture.md 后能看到 内部依赖关系图（含外部系统）、核心抽象清单和数据模型概览——知道改一个模块会波及哪些地方。每个模块的职责和上下游在 `modules/{模块}.md` 中（Step 3 创建）。

**校准时**：必须先按粒度读实际代码，再读 RECON.md 比对差异，不得先读 RECON.md 再"回想"代码。

### 4. Journey — 旅程侦察

**最重要的步骤。** 分两层——全局旅程（`user-journeys.md`）和流转（`modules/{m}.md`）。首次侦察时产出全局旅程；流转在用户指向具体模块时按需产出。

#### 用户旅程（User Journey）

产出全局 `user-journeys.md`，从用户视角追踪价值的交付顺序，每步标注负责实体。

| 动作 | 具体内容 |
|------|---------|
| 找 | 用户画像——谁在用这个系统？有几类用户？ |
| 找 | 证据来源：README 的使用示例/tutorial、CLI `--help` 输出、UI 截图/录制、集成测试文件（e2e test 是最可靠的用户旅程编码）、API 文档的 Endpoint 调用顺序 |
| 描 | 从首次接触到用出价值的完整路径——"导入资产 → 编辑内容 → 发布上线"。**每个步骤是用户推进价值的一个进展节点** |
| 标 | 每步标注**负责实体**——只用名称（如 → messages，→ 闲管家），不加类型标签、不写文件路径 |

**完成标准**：读者看完后理解用户的典型使用路径——每步做什么、由哪些负责实体承载。知道用户最关心的价值交付顺序，能从用户视角对系统功能和负责实体有个基本了解。**用户旅程中不应出现系统自动执行的内部步骤；每个旅程应覆盖一个连贯的端到端路径，而非多个独立入口的清单。**

**陷阱：**

| 问题 | 说明 |
|------|------|
| 内部步骤混入 | 把系统自动执行的管线步骤写成用户旅程步骤。|
| 多入口合一 | 把多个独立入口的零散操作塞进一个旅程。|
| 表面推断 | 不要从 UI 界面文字推断用户实际上在做什么。|
| 开发者操作混入 | 把 git clone、npm install、配置 .env 等开发搭建步骤写成用户旅程。安装部署属于 dev-loop，不是用户旅程。|

**跳过规则**：单次调用的工具库可跳过——按需机制自然覆盖。不要为没有用户旅程的项目编造一个。

#### 流转

对**关键模块**（从用户旅程涉及模块中选取）写入 `.recon/modules/{模块}.md` 的流转节。首次侦察只做全局旅程不产出流转；后续用户指向具体模块时按需补充。

流转描述模块完成职责的概念性流程——输入什么、经过哪些处理阶段或数据变形、产生什么效果、交给谁。视角按模块特性定：数据流动、处理阶段、或两者结合，不预设子节。流转层级的描述不需要知道底层代码实现就能读懂。

整个流转不是必须存在的——评估后无需则标记 `○`。

对已产出流转的模块，从流转中提炼本质理解（1-2 条），写入该模块文件的「洞察」节——说白了这模块在干什么。没有值得提炼的则跳过。

**完成标准**：
- 首次侦察后：`user-journeys.md` 已产出（含负责实体列）
- 深入模块后：目标模块的流转 + 洞察已产出；无需流转的模块已标记 `○`
- 校准后：按粒度先读代码 → 验证现 RECON.md 是否准确 → 更新流转 + 洞察 + 深入

**如果写出来只有"用户发请求，然后数据被处理，然后保存到数据库"——不合格。

流转超过 100 行 → 拆成子模块，写入 `.recon/modules/{子系统}/{子模块}.md`。

### 5. Dev Loop — 开发回路侦察

| 动作 | 具体内容 |
|------|---------|
| 找 | 构建设置、测试框架、lint 配置 |
| 找 | CI 配置（.github/workflows, .gitlab-ci.yml）——它告诉你哪些入口是"生产重要"的、构建流水线如何运转 |
| 录 | 写入 `.recon/dev-loop.md` |
| 验 | 确认已知构建+测试+lint 命令（不一定要跑，除非用户要求） |

**完成标准**：读者读完 dev-loop.md 后知道从拉代码到跑通测试的完整命令序列，可以据此在本机搭建开发环境。构建/测试/lint 各至少列一条可执行命令，且命令前置依赖（如依赖安装步骤）已说明。

### 6. 深入 — 纵深侦察

在用户要求深入了解指定模块时执行。深入与流转的抽象层次不同：流转描述模块的概念性流程，聚焦处理逻辑和数据变形，不触及实现细节；深入则下沉到代码层面，读核心文件，测绘内部结构，追踪具体代码路径，识别设计模式。

**做什么**：

- 读该模块的所有核心文件（不是全部，是核心）
- 拆内部组件——子模块、层、抽象边界、连接关系
- 追踪代码路径，精确到 `文件名::函数名`
- 看错误处理路径（不只 happy path）
- 识别设计模式：描述观察到的结构、约定和抽象边界。有文档/注释支撑的 → 写设计理由并标注来源；纯从代码推断的 → 只描述「模式是什么 + 效果是什么」，不编造动机
- 校验「洞察」节的已有结论——对照代码验证是否准确，如有偏差则更新

**完成标准**：读者读完深入节能看到该模块的内部结构图（有什么组件、如何连接）、关键代码路径和观察到的设计模式（模式是什么 + 效果是什么）。有文档支撑的设计理由应标注来源；无从代码推断的模式不编造动机。该模块中出现的项目术语已同步更新到 glossary.md。

**校准时**：必须先重新读核心代码文件，再将深入节与代码比照更新，不得直接编辑深入节而不验证代码。

## Recon Map

核心产出。术语见 [GLOSSARY.md](GLOSSARY.md)，格式见 [OUTPUT.md](OUTPUT.md)（技能文件，不写入项目 `.recon/`）

## 失败模式

| 问题 | 症状 |
|------|------|
| 前置完成 | 读了 README 就说"懂了" |
| 强行解释 | 把惯性、复制粘贴、语言惯例归因为有意的设计决策。没有文档佐证的前提下编造"为什么这样设计"的理由，产出虚假 tradeoff |
| 幸存者偏差 | 只读了主流程，没看错误处理 |
| 地图≠疆域 | 代码结构和运行时架构不一致 |
| 分析瘫痪 | 在某个工具函数里深挖出不来 |
| 虚假熟悉 | 能复述 RECON.md 但不能从代码重新走 |
| 复盘变复述 | 重新审视模块时直接读 RECON.md 了事，没回到代码 |
