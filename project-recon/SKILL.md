# project-recon

**语言：所有 Recon Map 产出物使用中文。** 代码标识符保留原文。

对陌生项目做系统化**侦察**——目标不是"读代码"，而是持续产出一份可分享的 **Recon Map**。

核心假设：你不知道那里有什么。每次侦察都从外围到核心逐层推进，每层有明确完成标准阻止过早宣称"懂了"。


**module 是用户的一个理解单元，不是代码的一个目录。** 它对应读者会自然地说"我想单独了解 X"的那个 X——可以是一条完整业务线（下单→支付→履约），也可以是一个横切点（日志系统），甚至是一个关键数据结构（交易快照）。划分原则：用户如果会将多个代码实体放在一起理解 → 一个 module；用户如果会分开理解 → 多个 module。代码树中的同名目录是常见起点但不是约束。文件位于不同目录但属于同一理解单元 → 合并；一个目录包含多个理解单元 → 拆分。判断标准始终是读者视角，不是文件系统。

## 工作方式

Agent 每次调用先读取 `.recon/RECON.md`（如存在）确定已完成的工作。**产出边界由区域覆盖状态决定，不由用户措辞猜测深浅。**

### 第一层：按输入类型分叉

| 输入类型 | 用户说法 | Agent 行为 |
|---------|---------|-----------|
| 了解性 | "看X""讲X""X是做什么的""X怎么工作"等 | 读 X 覆盖状态 → 按**状态路由**执行 |
| 审视性 | "重新看X""X好像变了""重审"等 | 当作无产出，重置重做 |
| 默认（无 RECON.md） | — | 首轮：外围+入口 → 架构侦察 → 用户旅程 + 开发回路 |

### 第二层：状态路由

RECON.md 中区域覆盖状态使用两个标签：**`basic`**（`modules/{区域}.md` 已产出——概念层理解已完成：该区域的背景/职责/生态位、流转、洞察三块）、**`deep`**（`modules/{区域}-深入.md` 已产出——代码层理解已完成：内部组件、代码路径、错误处理、设计模式已测绘）。

| 覆盖状态 | Agent 做什么 | 停点 |
|---------|-------------|------|
| 不在表中 | 创建 `modules/{区域}.md`（第一块 + 流转 + 洞察） | **停**，汇报摘要 + 问"是否深入看代码实现？" |
| `basic` | 汇报该区域在 RECON.md 中的状态和产出摘要 | **停**，问"是否深入看代码实现？" |
| `deep` | 汇报该区域在 RECON.md 中的状态和产出摘要 | 问"是否有新关注点需要重新审视？" |

**每条路径都有显式停点。Agent 不得越过停点继续执行。**

### 状态同步规则

每次操作完成后同步更新 Coverage 表：

| 操作 | Coverage 表更新 |
|------|---------------|
| 创建 `modules/{区域}.md`（basic） | 增行：区域=X，状态=basic，日期=今天 |
| 完成代码深入（deep） | 改行：状态=deep，日期=今天 |
| 汇报摘要（basic / deep） | 改行：更新"最后访问"到今天 |
| 重新看 X | 删行：移除 X 对应行 |
| 重新看项目 | 清空：保留表头，删除所有行 |

Agent 通过 `.recon/RECON.md` 的 Coverage 表管理状态。**汇报时永远不向用户暴露分支名称或状态标签**——用户只需要知道项目本身。

### 失败模式

| 问题 | 症状 |
|------|------|
| 前置完成 | 同一轮次内跳跃——骨架未扎实就直接写旅程 |
| 强行解释 | 把惯性、复制粘贴、语言惯例归因为设计决策，编造虚假 tradeoff |
| 幸存者偏差 | 深入时只读主流程，漏错误处理路径 |
| 地图≠疆域 | 代码结构与运行时架构不一致 |
| 首轮深挖 | 在第一轮扎进代码细节，跳出架构层 |
| 虚假熟悉 | 能背出产出物但不能从代码复现 |
| 子模块后置 | 写完发现模块过大，再临时拆分 |
| 越停 | 未在停点询问就直接进入下一步 |

## 汇报规则

发现式汇报——汇报项目发现，不汇报侦察过程。

**中途定位**：首轮侦察时外围+入口完成后先给一句话定位（项目是什么 + 技术栈），不等全部跑完。

| 时机 | 汇报什么 | 长度 |
|------|---------|------|
| 首轮侦察完成时 | ①定位 ②问题域 ③工程模块职责 ④快速开始 ⑤推荐入口。①②④⑤落盘到 RECON.md Overview；③落盘到 architecture.md | 一段话，点序明确 |
| 概念理解完成时（basic） | 该区域职责 + 流转 + 洞察各一句 | 3-5 句 |
| 代码深入完成时（deep） | 内部结构 + 关键代码路径 + 设计模式 | 3-5 句 |
| 重新审视完成时 | 哪些内容已重写、核心变化 | 2-3 句 |
| 发现关键信息时 | 实际发现（如"README 与实际代码不符"） | 2-3 句 |

所有汇报只谈项目本身。不 dump 文件内容、不说"已完成""执行完毕"、不使用内部术语。

**目的：** 让对话输出成为 **"我知道了你该知道什么"** 的指引，而不是"文件已生成"的通知。用户读完汇报后，对这个项目已经有方向感；细节去 `.recon/` 里找。

## 侦察流程

### 产出原则

每份产出文件的抽象层次对齐文件标题。第一轮产出物（architecture.md、user-journeys.md 等）回答项目级问题（这个项目是什么、怎么跑）；`modules/{区域}.md` 回答"这个区域是什么"（第一块 + 流转 + 洞察）；`modules/{区域}-深入.md` 回答"内部怎么做"（代码路径、结构、模式）。发现自己在描述某个更低层的实体——那段内容属于对应粒度的子文件。

### 完成标准原则

完成标准衡量的是读者阅读产出后的理解效果，而非产出文件的内容存在性。
错误模式："读者能看到依赖关系图"——存在性检验，检查"文件里有没有这张图"。
正确模式："读者能根据该图说清模块间依赖关系并判断修改波及范围"——有效性检验，检查"读者能否从图中提取信息并形成判断"。
诊断方法：如果一条完成标准可以仅凭确认"文件里有某段内容"就通过，它就是存在性标准而非读者有效性标准。

### 第一轮：全局侦察

外围+入口 → 架构侦察 → 用户旅程 + 开发回路（可并行）

#### 1. 外围侦察

| 动作 | 具体内容 |
|------|---------|
| 读 | README, CONTRIBUTING, CHANGELOG, LICENSE |
| 读 | 项目元数据 (package.json, Cargo.toml, pyproject.toml, go.mod 等) |
| 扫 | 目录树 2 层深（`tree -L 2` 或等价操作） |
| 找 | main 函数、CLI 入口、HTTP 路由注册、导出 API |
| 找 | 运行时配置——环境变量（.env*）、配置文件（config.*, .yaml, .toml）、CLI flags |
| 列 | 所有入口点清单，写入 `.recon/entry-points.md` |
| 录 | 项目专用术语（领域概念），按主题分组写入 `.recon/glossary.md`——每组独立二级标题，同类概念放一起。术语不含模块名称。其它文件出现术语时用 `→ glossary.md#{主题}` 回链 |

**完成标准**：读者能用一句话说清楚"这个项目做什么、用什么技术栈、顶层目录各负责什么、入口在哪、怎么启动"。关键术语已在 glossary.md 中解释。

**陷阱**：README 可能过时。交叉验证 README 说的 vs 代码实际暴露的。

#### 2. 架构侦察

| 动作 | 具体内容 |
|------|---------|
| 列 | 基于外围侦察的发现，列出所有工程模块并标注一句话职责，写入 `architecture.md` |
| 画 | 内部依赖方向 + 外部系统（数据库、第三方 API、消息队列等），写入 `architecture.md`（Mermaid 或 ASCII）。每个节点标注证据来源：`✓` 多文件交叉验证、`？` 从目录名或导入关系推断、`○` 外部依赖未检查 |
| 找 | 核心抽象——traits、interfaces、abstract base classes、核心数据结构。描述概念本身（如"YAML 配置系统，三层合并优先级"），不标注文件路径 |
| 看 | 数据模型——ORM schema、migration 文件、DDL。列出核心实体和关系，写入 `architecture.md` |
| 量 | 文件数、模块规模分布 |

**完成标准**：读者能看着架构图回答：系统有几层？想了解的东西在哪个区域？它与周围区域的关系是什么？如果改某个东西，大概去哪里找？

#### 3. 用户旅程

| 动作 | 具体内容 |
|------|---------|
| 找 | 用户画像——谁在用这个系统？有几类用户？ |
| 找 | 证据来源：README 的使用示例/tutorial、CLI `--help` 输出、UI 截图/录制、集成测试文件、API 文档的 Endpoint 调用顺序 |
| 描 | 从首次接触到用出价值的完整路径——"导入资产 → 编辑内容 → 发布上线"。**每个步骤是用户推进价值的一个进展节点** |
| 标 | 每步标注**负责实体**——只用名称（如 → messages），不加类型标签、不写文件路径 |

**完成标准**：读者理解用户的典型使用路径——每步做什么、由哪些负责实体承载。知道用户最关心的价值交付顺序。**用户旅程中不应出现系统自动执行的内部步骤；每个旅程应覆盖一个连贯的端到端路径，而非多个独立入口的清单。**

**陷阱：**
- **内部步骤混入**：把系统自动执行的管线步骤写成用户旅程步骤。
- **多入口合一**：把多个独立入口的零散操作塞进一个旅程。
- **表面推断**：不要从 UI 界面文字推断用户实际上在做什么。
- **开发者操作混入**：把 git clone、npm install、配置 .env 等开发搭建步骤写成用户旅程。

**跳过规则**：单次调用的工具库可跳过。不要为没有用户旅程的项目编造。

#### 4. 开发回路

| 动作 | 具体内容 |
|------|---------|
| 找 | 构建设置、测试框架、lint 配置 |
| 找 | CI 配置（.github/workflows, .gitlab-ci.yml）——它告诉你哪些入口是"生产重要"的、构建流水线如何运转 |
| 录 | 写入 `.recon/dev-loop.md` |
| 验 | 确认已知构建+测试+lint 命令（不一定要跑，除非用户要求） |

**完成标准**：读者知道从拉代码到跑通测试的完整命令序列，可以据此在本机搭建开发环境。构建/测试/lint 各至少一条可执行命令，且命令前置依赖已说明。

**首轮产出后**：汇报成果，等待用户下一步指示。不自动接续用户之前的请求。

### 第二轮：区域侦察

用户首次指向一个区域时触发。产出边界由 RECON.md 中该区域的覆盖状态决定。

#### 1. 概念理解（basic）

| 动作 | 具体内容 |
|------|---------|
| 确认 | 确认用户指向的区域是真实实体——Agent 自行从代码或 architecture.md 推断，不确定则与用户确认 |
| 写 | 创建 `.recon/modules/{区域}.md`，依次产出三块——每块是理解的递进关系 |

**第一块（背景（按需） + 职责 + 生态位）**
回答"这个区域是干什么的"。读者能据此说清该区域在系统中的角色和边界。

**第二块（流转）**
回答"这个区域怎么干的（不看代码也能看懂）"。用领域语言描述该区域的角色——对外提供什么能力、接收什么输入、产出什么结果。聚焦于外部行为模式，不触及实现细节。写完删除所有实现语言词汇；如果剩余文本不成一个有意义的行为描述，说明抽象层不够。

**第三块（洞察，0-2条）**
回答"剥离了该项目选择后，这个区域在做的本质是什么"。换一种实现方式，这个"本质"仍然成立。没有则跳过，不凑数。

**停点**：三块产出后**停**，汇报摘要 + 问"是否深入看代码实现？"

#### 2. 代码深入（deep）

在用户确认深入时执行。产出写入 `modules/{区域}-深入.md`。

| 动作 | 具体内容 |
|------|---------|
| 读 | 该区域的核心文件（不是所有文件） |
| 拆 | 内部组件——子模块、层、抽象边界、连接关系 |
| 追 | 代码路径，精确到 `文件名::函数名` |
| 看 | 错误处理路径（不只 happy path） |
| 识 | 设计模式：描述观察到的结构、约定和抽象边界。有文档/注释支撑的 → 写设计理由并标注来源；纯从代码推断的 → 只描述「模式是什么 + 效果是什么」，不编造动机 |
| 验 | 校验对应 `{区域}.md` 中「洞察」节的已有结论——对照代码验证是否准确，如有偏差则更新 |

**完成标准**：读者读完能回答：该区域内部有哪些组件？各自职责和连接关系是什么？关键代码路径的走向和错误处理方式是什么？观察到了哪些设计模式？该区域中出现的项目术语已同步更新到 glossary.md。
