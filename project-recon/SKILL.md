# project-recon

**语言：所有 Recon Map 产出物使用中文。** 代码标识符保留原文。

对陌生项目做系统化**侦察**——目标不是"读代码"，而是持续产出一份可分享的 **Recon Map**。

核心假设：你不知道那里有什么。每次侦察都从外围到核心逐层推进，每层有明确完成标准阻止过早宣称"懂了"。

> Bold terms → 见 [GLOSSARY.md](GLOSSARY.md)

## 工作方式

Agent 每次调用先读取 `.recon/RECON.md`（如存在）确定已完成的工作，再根据用户意图确定本次范围。

| 状态 | 用户意图 | Agent 做什么 |
|------|---------|-------------|
| 无 RECON.md | — | 首次侦察：Step 1→2→3→4(仅用户旅程)→5，产出 RECON.md |
| 有 RECON.md | 指向具体模块（"看看X""X怎么设计的"） | 读该模块覆盖状态：Flow 未写 → Step 4(模块Flow)；Flow 已写、Deep 未写 → Step 6；全已写 → 告知已有深度，问是否重新审视 |
| 有 RECON.md | 指向具体模块，但模块不在 Coverage 表中 | 确认 X 是真实代码实体 → 补 Step 3（为 X 创建 `modules/{X}.md` 骨架、更新 `architecture.md`、RECON.md 增行）→ 按 Flow 未写处理，执行 Step 4(模块Flow) |
| 有 RECON.md | 无具体目标（"还有什么可看"） | 汇报覆盖状态，指出缺口，建议"想看哪个模块？" |
| 有 RECON.md | "怎么跑""怎么开发""搭建环境" | Step 5 |

Agent 通过 `.recon/RECON.md` 的 Module Coverage 表管理状态。**汇报时永远不向用户暴露分支名称、Coverage 数值或步骤编号**——用户只需要知道项目本身。

## 汇报规则

发现式汇报——汇报项目发现，不汇报侦察过程。

**中途定位**：首次侦察时 Step 1-2 完成后先给一句话定位（项目是什么 + 技术栈），不等全部跑完。

| 时机 | 汇报什么 | 长度 |
|------|---------|------|
| 首次侦察完成时 | ①定位 ②问题域 ③模块地图 ④关键架构决策 ⑤快速开始 ⑥推荐入口。同步落盘到 RECON.md Overview | 一段话，点序明确 |
| 深入模块完成时 | 该模块设计思路 + 关键 tradeoff + 意外发现 | 3-5 句 |
| 补漏完成时 | 补了哪些模块，每个一句核心职责 | 每个模块 1-2 句 |
| 发现关键信息时 | 实际发现（如"README 与实际代码不符"） | 2-3 句 |

所有汇报只谈项目本身。不 dump 文件内容、不说"已完成""执行完毕"、不使用内部术语。

**目的：** 让对话输出成为 **"我知道了你该知道什么"** 的指引，而不是"文件已生成"的通知。用户读完汇报后，对这个项目已经有方向感；细节去 `.recon/` 里找。

## 六大步

**产出原则：每份产出文件的抽象层次对齐文件标题。** `modules/X.md` 的所有内容（Flow、Deep）回答的都是"X 这个模块"的问题。发现自己在描述 X 内部某个更低层的实体——那段内容属于对应粒度的子文件。

### 1. Perimeter — 外围侦察

| 动作 | 具体内容 |
|------|---------|
| 读 | README, CONTRIBUTING, CHANGELOG, LICENSE |
| 读 | 项目元数据 (package.json, Cargo.toml, pyproject.toml, go.mod 等) |
| 扫 | 目录树 2 层深（`tree -L 2` 或等价操作） |
| 录 | 项目专用术语，按主题分组写入 `.recon/glossary.md`——每组独立二级标题，同类概念放一起。其它文件出现术语时用 `→ glossary.md#{主题}` 回链 |

**完成标准**：读者看完产出后能用一句话说清"这个项目做什么、用什么技术栈、顶层目录各负责什么"。关键术语已在 glossary.md 中解释，可随时查阅。

**陷阱**：README 可能过时。交叉验证 README 说的 vs 代码实际暴露的。

### 2. Entry — 入口侦察

| 动作 | 具体内容 |
|------|---------|
| 找 | main 函数、CLI 入口、HTTP 路由注册、导出 API |
| 列 | 所有入口点的清单，写入 `.recon/entry-points.md` |

**完成标准**：读者读完 entry-points.md 后能看到完整的入口清单，清楚启动流程从哪开始、哪些入口是生产关键路径。

### 3. Architecture — 架构侦察

| 动作 | 具体内容 |
|------|---------|
| 建 | 为每个模块创建 `.recon/modules/{module}.md` 骨架，写**职责**+**上下游**（格式见 OUTPUT.md） |
| 画 | 模块间的依赖方向，写入 `.recon/architecture.md`（Mermaid 或 ASCII） |
| 找 | 核心抽象——traits、interfaces、abstract base classes、核心数据结构 |
| 量 | 文件数、模块规模分布 |

**完成标准**：读者看完 architecture.md 后能看到模块依赖关系图和核心抽象清单——知道改一个模块会波及哪些地方。每个模块的职责和上下游在 `modules/{module}.md` 中（Step 3 创建）。

### 4. Flow — 数据流侦察

**最重要的步骤。** 分两个产出——用户旅程（全局视角）和模块 Flow 节（代码视角）。首次侦察时产出用户旅程；模块 Flow 节在用户指向具体模块时按需产出。

#### 用户旅程（User Journey）

产出全局 `user-journeys.md`，从用户视角追踪价值的交付顺序，每步标注涉及模块。

| 动作 | 具体内容 |
|------|---------|
| 找 | 用户画像——谁在用这个系统？有几类用户？ |
| 找 | 证据来源：README 的使用示例/tutorial、CLI `--help` 输出、UI 截图/录制、集成测试文件（e2e test 是最可靠的用户旅程编码）、API 文档的 Endpoint 调用顺序 |
| 列 | 从首次接触到用出价值的完整路径——"用户先 `init`，再 `add source`，再 `query`，最后 `export`" |
| 标 | 每步标注**涉及模块**（→ modules/{m}.md 的 Flow 节，architecture.md 次级），让读者知道每一步由哪些模块承载 |
| 标 | 路径中的前置依赖——哪个步骤必须在前一步之后才能做 |
| 记 | 退出点——用户什么情况下算"完成"了？ |

**完成标准**：读者看完后理解用户的典型使用路径——每步做什么、涉及哪些模块、前置依赖。知道用户最关心的价值交付顺序，能通过"涉及模块"链接跳转到 modules/{m}.md 查看模块职责与 Flow。

**陷阱**：不要从 UI 界面文字推断用户实际上在做什么。用户旅程需要**调用序列的证据**（CLI 文档，e2e 测试，README 示例），不是界面翻译。

**跳过规则**：单次调用的工具库可跳过——按需机制自然覆盖。不要为没有用户旅程的项目编造一个。

#### 模块 Flow 节

对**关键模块**（从用户旅程涉及模块中选取）写入 `.recon/modules/{module}.md` 的 Flow 节。首次侦察只做用户旅程不产出模块 Flow；后续用户指向具体模块时按需补充。Flow 的抽象层次遵守产出原则：文件标题是什么，Flow 就跟踪什么的流程。

Flow 节含两个子节，**按需写**——模块没有数据旅程或代码轨迹时，不写该子节；整个 Flow 节都不是必须存在的（通过评估后标记 `○` 即可）。

##### 数据旅程（Data Journey）

仅在该模块有**输入输出形态变化**或**副作用**时写。帮助读者从数据视角理解模块。

| 动作 | 具体内容 |
|------|---------|
| 述 | 输入形态 → 处理过程 → 输出形态 |
| 标 | 输出送入哪个模块（→ modules/{m}.md 如果存在，否则 → architecture.md） |

##### 代码轨迹（Code Trace）

仅在该模块有**可追踪的执行路径**时写。

| 动作 | 具体内容 |
|------|---------|
| 选 | 最简单的 primary feature（不是错误处理路径） |
| 跟 | 从入口到出口精确到 `文件名::函数名` |
| 记 | 沿途经过的每一层、每个文件、每个关键函数 |

**完成标准**：
- 首次侦察后：`user-journeys.md` 已产出（含涉及模块列）
- 深入模块后：目标模块的 Flow 节已产出（数据旅程或代码轨迹），代码轨迹精确到 `文件名::函数名`；无需 Flow 的模块已标记 `○`
- 重新审视后：回到代码验证，Flow + Deep 已更新

**如果写出来只有"用户发请求，然后数据被处理，然后保存到数据库"——不合格。** 代码轨迹必须精确到文件名和函数名。

Flow 节超过 100 行时模块可能太粗——拆成子模块，写入 `.recon/modules/{子系统}/{子模块}.md`。

### 5. Dev Loop — 开发回路侦察

| 动作 | 具体内容 |
|------|---------|
| 找 | 构建设置、测试框架、lint 配置 |
| 找 | CI 配置（.github/workflows, .gitlab-ci.yml）——它告诉你哪些入口是"生产重要"的、构建流水线如何运转 |
| 录 | 写入 `.recon/dev-loop.md` |
| 验 | 确认已知构建+测试+lint 命令（不一定要跑，除非用户要求） |

**完成标准**：读者读完 dev-loop.md 后知道从拉代码到跑通测试的完整命令序列，可以据此在本机搭建开发环境。构建/测试/lint 各至少列一条可执行命令，且命令前置依赖（如依赖安装步骤）已说明。

### 6. Deep — 纵深侦察

在用户要求深入了解指定模块时执行。Deep 的抽象层次遵守产出原则：文件标题是什么，Deep 就分析什么的设计决策。

| 动作 | 具体内容 |
|------|---------|
| 读 | 该模块的所有核心文件（不是全部，是核心） |
| 找 | 隐藏抽象——继承链、trait bounds、依赖注入 |
| 看 | 错误处理路径（不只 happy path） |
| 问 | 为什么这么设计 |
   | 写 | Deep 节到 `.recon/modules/{module}.md` + 设计决策追加到 `.recon/design-decisions.md` |

**完成标准**：读者读完 Deep 节能理解该模块的内部设计思路、关键 tradeoff 及其代码证据——知道"为什么这样设计"，不只是"怎么实现的"。模块中出现的项目术语已同步更新到 glossary.md。

## Recon Map

技能的核心产出。既是 Agent 的**状态数据库**，也是你的**回忆索引**。

## 失败模式

| 问题 | 症状 |
|------|------|
| 前置完成 | 读了 README 就说"懂了" |
| 幸存者偏差 | 只读了主流程，没看错误处理 |
| 地图≠疆域 | 代码结构和运行时架构不一致 |
| 分析瘫痪 | 在某个工具函数里深挖出不来 |
| 虚假熟悉 | 能复述 RECON.md 但不能从代码重新走 |
| 复盘变复述 | 重新审视模块时直接读 RECON.md 了事，没回到代码 |
