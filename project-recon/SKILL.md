# project-recon

**语言：所有 Recon Map 产出物使用中文。** 代码标识符保留原文。

对陌生项目做系统化**侦察**——目标不是"读代码"，而是持续产出一份可分享的 **Recon Map**。

核心假设：你不知道那里有什么。每次侦察都从外围到核心逐层推进，每层有明确完成标准阻止过早宣称"懂了"。

> Bold terms → 见 [GLOSSARY.md](GLOSSARY.md)

## 入口判断

调用时先做**分支判断**，决定走哪条 Branch：

```
用户说"看看这个项目"/第一次 → Full
用户说"看看其他模块"        → onCover
用户说"深入看下 X 模块"    → onDig
用户说"再看下 X 模块"      → onRevisit
无明确信号，项目已有 RECON.md → onCover
无明确信号，没有 RECON.md    → Full
```

`onDig` 和 `onRevisit` 的区别：depth < 3 是 onDig（深入新领域），depth ≥ 3 是 onRevisit（重新建立已遗忘的心智模型）。

## Branches

### Full — 首次全流程侦察

按 1→6 的顺序执行所有步骤，产出 Recon Map。

### onCover — 补漏未覆盖模块

1. 读 RECON.md，找 Step1-6 有任何 `-`（未完成）的模块
2. 如果全部 `✓` → 告知用户所有模块已覆盖，建议用 `onDig` 或 `onRevisit`
3. 如果有未完成模块 → 对每个 X 走对应步骤（通常是 Step 4 Flow），创建或更新 `.recon/modules/{module}.md` 的 Flow 节，完成后更新里程碑

### onDig — 深入指定模块

1. 读 RECON.md 确认目标模块
2. 如果 depth ≥ 3 → 告知用户这个模块已深度覆盖，建议用 `onRevisit` 如果遗忘了
3. 执行 Step 6（Deep Recon）于该模块：
   - 创建或更新 `.recon/modules/{module}.md` 的 Deep 节
   - 追加设计决策到 `.recon/design-decisions.md`
4. 更新 RECON.md 里程碑 + `Last Visit` 时间

### onRevisit — 复盘指定模块

1. 读 RECON.md 中该模块的现有记录（作为钩子）
2. 读完之后**必须回到代码重新执行 Step 4 + 6**，不允许直接从 RECON.md 复述
3. 更新 `.recon/modules/{module}.md`（Flow 节 + Deep 节）
4. 更新 RECON.md 的 `Last Visit` 时间

## 汇报规则

**汇报的触发信号是"有发现"，不是"做完一步"。** Agent 内部照常 1→6 推进，但在有发现时开口，而不是在步骤边界打报告。

| 时机 | 汇报什么 | 长度 |
|------|---------|------|
| 发现关键信息时 | 项目的实际发现——"这是一个支付系统，Go 语言，三个主要模块…" | 2-3 句 |
| Full 分支结束时 | 按以下顺序组织汇报：①一句话定位（这是什么项目）②模块地图（模块各自干什么）③关键架构决策（最值得注意的设计选择，不是全部）④快速开始（如果想跑起来，做什么）⑤如果只想读一个文件——推荐从哪个模块/哪份产出开始 | 一段话，点序明确 |
| onCover 结束时 | 这次补漏了哪些模块 + 一句总结每块的核心职责 | 每个模块 1-2 句 |
| onDig / onRevisit 结束时 | 该模块的内部设计一句话概括 + 关键 tradeoff + 是否有令人意外的东西 | 3-5 句 |

**不要做：**
- 不要 dump `.recon/` 文件内容到对话中（文件就是为了存完整内容才存在的）
- 不要只说"已完成"或"已写入 .recon/glossary.md"——说发现了什么
- **不要以技能概念组织语言**——不要说"Step 1"、"第 X 步"、"执行 onDig 分支"、"depth N/6"。说项目本身的事："先看了外围——项目是……""这个模块主要流程跟踪过了，但内部细节还没深入"

**目的：** 让对话输出成为 **"我知道了你该知道什么"** 的指引，而不是"文件已生成"的通知。用户读完汇报后，对这个项目已经有方向感；细节去 `.recon/` 里找。

## 六大步

### 1. Perimeter — 外围侦察

| 动作 | 具体内容 |
|------|---------|
| 读 | README, CONTRIBUTING, CHANGELOG, LICENSE |
| 读 | 项目元数据 (package.json, Cargo.toml, pyproject.toml, go.mod 等) |
| 扫 | 目录树 2 层深（`tree -L 2` 或等价操作） |
| 录 | 项目专用术语，按主题分组写入 `.recon/glossary.md`——每组独立二级标题，同类概念放一起 |

**完成标准**：读者看完产出后能用一句话说清"这个项目做什么、用什么技术栈、顶层目录各负责什么"。关键术语已在 glossary.md 中解释，可随时查阅。

**陷阱**：README 可能过时。交叉验证 README 说的 vs 代码实际暴露的。

### 2. Entry — 入口侦察

| 动作 | 具体内容 |
|------|---------|
| 找 | main 函数、CLI 入口、HTTP 路由注册、导出 API |
| 找 | CI 配置（.github/workflows, .gitlab-ci.yml）——它告诉你哪些入口是"生产重要"的 |
| 列 | 所有入口点的清单，写入 `.recon/entry-points.md` |

**完成标准**：读者读完 entry-points.md 后能看到完整的入口清单，清楚启动流程从哪开始、哪些入口是生产关键路径。

### 3. Architecture — 架构侦察

| 动作 | 具体内容 |
|------|---------|
| 给 | 每个目录/模块贴上"职责标签" |
| 画 | 模块间的依赖方向，写入 `.recon/architecture.md`（Mermaid 或 ASCII） |
| 找 | 核心抽象——traits、interfaces、abstract base classes、核心数据结构 |
| 量 | 文件数、模块规模分布 |

**完成标准**：读者看完 architecture.md 后能看到模块依赖关系图、每个模块的核心职责、以及核心抽象清单——知道改一个模块会波及哪些地方。

### 4. Flow — 数据流侦察

**最重要的步骤。** 分两条子追踪，视角不同但同层抽象。

#### Feature Flow（代码视角）

选一个主要功能，**从入口到出口完整追踪**，产出到 `.recon/modules/{module}.md` 的 Flow 节。

| 动作 | 具体内容 |
|------|---------|
| 述 | 写模块概述：职责 + 上下游（让读者在看 Flow 之前先知道这是什么、在哪一环） |
| 选 | 最简单的 primary feature（不是错误处理路径） |
| 跟 | 从用户输入/HTTP 请求/CLI 参数到最终输出/数据库写入 |
| 记 | 沿途经过的每一层、每个文件、每个关键函数 |

**完成标准**：读者沿着 Flow 节能走通一条主线功能——从入口到出口，每层跳转精确到 `文件名::函数名`。读者可以拿着这段追踪去对应代码位置验证。

**如果写出来只有"用户发请求，然后数据被处理，然后保存到数据库"——不合格。** 产出必须精确到文件名和函数名。

#### User Journey Flow（用户视角）

从用户视角追踪价值的交付顺序，产出到 `.recon/user-journeys.md`。

| 动作 | 具体内容 |
|------|---------|
| 找 | 用户画像——谁在用这个系统？有几类用户？ |
| 找 | 证据来源：README 的使用示例/tutorial、CLI `--help` 输出、UI 截图/录制、集成测试文件（e2e test 是最可靠的用户旅程编码）、API 文档的 Endpoint 调用顺序 |
| 列 | 从首次接触到用出价值的完整路径——"用户先 `init`，再 `add source`，再 `query`，最后 `export`" |
| 标 | 路径中的前置依赖——哪个步骤必须在前一步之后才能做 |
| 记 | 退出点——用户什么情况下算"完成"了？ |

**完成标准**：读者读完 user-journeys.md 后能理解用户的典型使用路径、每一步的前置依赖、以及"完成"的标志——知道用户最关心的价值交付顺序。

**跳过规则**：如果项目是单次调用的工具库（无 CLI/无多步 API/无 UI），标注 `trivial — single call` 并跳过。不要为没有用户旅程的项目编造一个。

**陷阱**：不要从 UI 界面文字推断用户实际上在做什么。用户旅程需要**调用序列的证据**（CLI 文档，e2e 测试，README 示例），不是界面翻译。

### 5. Dev Loop — 开发回路侦察

| 动作 | 具体内容 |
|------|---------|
| 找 | 构建设置、测试框架、lint 配置 |
| 录 | 写入 `.recon/dev-loop.md` |
| 验 | 确认已知构建+测试+lint 命令（不一定要跑，除非用户要求） |

**完成标准**：读者读完 dev-loop.md 后知道从拉代码到跑通测试的完整命令序列，可以据此在本机搭建开发环境。

### 6. Deep — 纵深侦察

只在指定模块上执行（`onDig` / `onRevisit`）：

| 动作 | 具体内容 |
|------|---------|
| 读 | 该模块的所有核心文件（不是全部，是核心） |
| 找 | 隐藏抽象——继承链、trait bounds、依赖注入 |
| 看 | 错误处理路径（不只 happy path） |
| 问 | 为什么这么设计 |
   | 写 | Deep 节到 `.recon/modules/{module}.md` + 设计决策追加到 `.recon/design-decisions.md` |

**完成标准**：读者读完 Deep 节能理解该模块的内部设计思路、关键 tradeoff 及其代码证据——知道"为什么这样设计"，不只是"怎么实现的"。模块中出现的项目术语已同步更新到 glossary.md。

## 失败模式

| 问题 | 症状 | 防御 |
|------|------|------|
| 前置完成 | 读了 README 就说"懂了" | 每步有可检查的完成标准（见 [GLOSSARY.md](GLOSSARY.md)） |
| 幸存者偏差 | 只读了主流程，没看错误处理 | Step 4 强约束"每一层都要精确到文件名" |
| 地图≠疆域 | 代码结构和运行时架构不一致 | Step 3 + 4 一起防：架构图 + 运行时追踪 |
| 分析瘫痪 | 在某个工具函数里深挖出不来 | 时间盒约束，Step 6 只对指定模块 |
| 虚假熟悉 | 能复述 RECON.md 但不能从代码重新走 | Step 4 完成标准精确到文件::函数，不能泛泛描述 |
| 复盘变复述 | onRevisit 时直接读 RECON.md 了事 | onRevisit 明确规则：读完钩子后必须重新 trace 代码 |
