# article-retelling

将原始材料通过自己的风格「复述」成一篇可发表的博客或文章

---

## 写作基调

这是一篇可发表的文章。不是翻译稿，不是笔记整理。

读起来像人写的——不是"AI 替你写了中文版"，是"你用中文重新表达了一遍"。

* 有节奏，有态度，每一个词都承担信息。遮掉原文，它独立成立；
* 去掉"原文提到"、"作者指出"这类嵌套，它自成一个叙事。
* 正文中不出现"刚刚""近日""最近""不久前"等模糊时间锚点。
* 不硬撑：一个意思说完了就收，不需要总是补充"总结/升华"让它看起来更完整
* 语言口语化：像向朋友解释复杂概念，使用自然的停顿和过渡词，避免书面化长句
* 节奏感强：长短句交替，有顿挫和重点，不单调铺叙

**检验标准**：
  - 如果这篇文章要发在你的公众号或博客上，别人读起来不会觉得这是AI写的，而是*像人类经过思考后写下的*。

---

## 整体流程

```
输入 → 理解消化 → 跨源交叉分析 → 声音锚定 → 复述锚定 → 解构提取 → 轮廓提案 → 重建叙事 → 品味检查 → 机械门禁 → 输出
```

---

## 📥 输入阶段

接受三类输入方式：
- **URL(s)**：单个或多个 URL，使用 webfetch 获取并解析全文
- **文件路径**：单个或多个本地文件路径
- **文本粘贴**：直接处理粘贴内容

多个来源时自动分配 ID（S1, S2, S3...），便于后续跨源分析阶段引用。

解析内容包括：标题、正文、所有图片及其上下文、所有外链及其上下文。

---

## 🧠 理解消化

对原始材料做全面理解，输出一份「理解摘要」供后续阶段使用。不在这个阶段做结构决策。

### 1. 整体理解

读完原文后，用一段话回答：原文在说什么？核心问题是什么？作者试图说服读者什么？

输出格式：
```
核心命题：原文试图回答什么核心问题 / 传递什么核心观点
受众定位：原文写给谁看的？→ 转述时面向的受众是否有变化？
信息密度：密集 / 适中 / 稀疏
```

### 2. 类型识别

根据原文特征判断文档类型，决定后续处理倾向：

| 类型 | 检测特征 | 叙事倾向 |
|------|----------|----------|
| **教程类** | 步骤编号、序列词（first/next/最后）、代码块占比高 | 按步骤递进，每步带一段展开 |
| **观点/论述类** | 第一人称论述、论据支撑结构、主观判断 | 论点开路，论据递进，收束到立场 |
| **技术论文类** | 摘要/引言/方法论/结论结构、引用标记 | 先给结论，再拆原理，最后摆证据 |
| **图文混合类** | 大量图片/截图、图片与文字交错排列 | 图文交错，文字不重复图里的内容 |
| **新闻/报道类** | 倒金字塔结构、导语先行、时间线推进 | 钩子(导语) → 支撑细节 → 背景与延展 |
| **评测/比较类** | 对比结构（A vs B）、优缺点分列、推荐/评分 | 并列展开对比维度 → 优劣势 → 结论 |
| **访谈/对话类** | Q&A 流转、多名说话人交替、话题切换 | 结构由问题驱动，保留对话节奏，归拢核心论点 |
| **一般内容** | 无明显类型特征 | 钩子开头 → 展开 → 收束到可带走的东西 |

类型识别结果影响解构粒度、图片处理优先级，但不限制最终组织方向。

### 3. 难点与模糊点标记

标记原文中以下内容：
- **不清晰或矛盾处**：原文表述模糊、逻辑跳跃、前后矛盾的位置
- **需要核实的事实**：数据、日期、引用——如有疑问标记 `[需核实]`
- **偏误风险**：原文可能过于片面或过时的观点

这些标记在解构和重建阶段需要特别处理。

### 4. 复述目标确认

确认转述目标（这一步只是记录，不在此决策——决策在后续锚定阶段）：
- 还原式转述：保留原文核心信息，仅改变表达
- 压缩式转述：提炼精华，删减
- 扩展式转述：在原文基础上补充背景或相关观点

### 5. 多源处理指引（输入含≥2个来源时启用）

每篇来源独立完成 §1.1–§1.4 的理解消化，产出各自的「理解摘要」。
多份摘要进入下一阶段「跨源交叉分析」进行合并与冲突标记。

---

## 🔄 跨源交叉分析

当输入包含≥2个来源时执行此阶段。将多份理解摘要的信息点合并、去重、标记冲突，产出统一的信息清单供重建使用。

### 1. 信息点合并

将各源的信息点汇总，分配来源标签 `[S1]`/`[S2]`/`[S3]`。对内容重叠的信息点做去重处理：
- **完全重叠**：保留表述更优的那一条，标记多源印证（`[S1+S2]`）
- **部分重叠**：合并为一条完整信息，保留各源自有的独特细节
- **矛盾**：保留双方，标记 `[冲突: S1 说 X, S2 说 Y]`，在重建时处理

### 2. 互补与盲区识别

识别跨源间的互补关系及各源的盲区：
- 何处互为补充（S1 给结论，S2 给数据）
- 何处存在空白（所有来源均未提及的内容不做猜测）

### 3. 产出跨源信息清单

合并后的信息点清单，每条带 `[Sx]` 标签，进入后续的声音锚定、复述锚定、解构与轮廓提案阶段。

---

## 🎙️ 声音锚定

在决定「复述锚定」前，先明确这篇文章要对齐的声音。没有声音锚定，后续的“像你自己写的”只能靠猜。

### 1. 声音来源

- **用户样文**：优先使用用户已发表、愿意作为参照的文章
- **本次对话偏好**：当用户明确描述了语气、节奏或禁用表达时，作为补充约束
- **通用作者风格**：当没有样文也没有足够偏好信息时，降级为稳定、自然、非翻译腔的通用作者风格

### 2. Voice DNA

为当前文章生成一份简短的声音摘要，至少包括：
- 句长：偏短 / 中等 / 长短交替
- 开头方式：直切 / 设问 / 叙事 / 判断先行
- 过渡习惯：少连接词自然推进 / 允许少量显式连接 / 依赖小标题切段
- 口语度：克制 / 自然 / 强口语
- 判断力度：克制描述 / 明确判断 / 带锋芒
- 结尾习惯：落观点 / 落行动 / 留余味 / 点到即止
- 禁用表达：列出 3-5 类不应出现的句式、语气或套话

### 3. 降级原则

- 有样文时，以样文为准，不拿“更标准”的写法覆盖用户声音
- 无样文时，不默认已经对齐用户本人，只要求稳定、自然、可发表
- 后续「复述锚定」「重建叙事」「品味检查」都应以这里的声音锚定为准

---

## 🔍 复述锚定

在充分理解原始材料后，决策一个「复述锚定」，并呈现给用户，按用户要求修改，由用户决定是否继续。

多源场景下，核心命题为「这些材料共同指向什么」，而非某单篇的主旨。锚定输出中需体现跨源的汇聚视角。

### 输出格式

```
读感：[像一篇分享 / 像一篇教程 / 像一份文档 / 像笔记整理]
叙事：[有核心命题 / 有观点递进 / 平铺陈列 / 无叙事]
视角：[面向读者讲一个发现 / 面向用户介绍一个工具 / 无明确视角 / 作者个人叙事]
结构：[层层推进 / 分节阐述 / 章节编号平铺 / 碎片化]
受众：xxx
声音：用一句话概括这篇文章应呈现的整体声音
约束：xxx

是否认可上述「复述锚定」？如果不认可，请优先指出要改的是：读感、叙事、视角、结构、受众、声音，还是约束。
```

### 用户反馈处理

输出复述锚定后，按以下情况处理：

- **用户未反馈或回应模棱两可**：主动给出 2–3 个具体调整方向的建议，引导用户做选择。例如："可以将视角从'介绍工具'转为'解决什么问题'，或者将结构从'分节阐述'改为'层层推进'——你倾向哪个方向？"
- **用户不同意但未给出方向**：预设 2–3 个替代方案供选择。例如："这篇按教程组织还是按观点组织更接近你的预期？"
- **用户给出调整方向**：按反馈修改锚定，再次输出供确认
- **用户明确反对并要求停止**：终止流程

## 🔍 解构提取

### 0. 前置元数据剥离

在信息提取之前，扫描标题与正文之间、以及正文最末的所有行。满足以下任一特征的视为元数据，**全部直接删除**：

- 包含日期格式（2026年/May 26等）
- 包含分隔符 `·`、`|`、`/`（如 `2026年5月·PrismML`）
- 匹配署名词头：`作者`、`By`、`Published`、`From`、`来源`、`原文`
- 匹配签名/介绍栏：`作者介绍`、`About the`、署名行

> **原则**：不维护模式列表，只按位置+格式特征判断。标题与正文第一段之间、以及正文末尾最后 3 段中满足上述特征的，一律删除。

### 1. 信息点提取

将原文打散为独立信息点，不保留原文结构。每一条信息点 = 一个不可再分的事实/论点/步骤：

- 以"一条信息一句话"为粒度
- 编号，便于结构阶段引用
- 每点标注关联：
  - **原文段落**（便于还原上下文核实）
  - **关联图片**（该信息点对应的图片 ID）
  - **关联外链**（该信息点涉及的引用链接）
  - **类型标签**：事实 / 论点 / 步骤 / 示例 / 引用 / 结论
  - **保留级别**：必须 / 建议 / 可选

> **解构原则**：只提取"原文说了什么"，不判断"应该放在哪"——那是结构阶段的工作。

### 1.2 人称归属标记

对每一条含"我"的信息点做归属分类：

- **[可吸收]**：通用经验、观点、方法、论断——重建时自然融入你的"我"
- **[不可吸收]**：独有的具体经历、时间线、署名、个人背景信息——重建时切换为第三人称或零人称

### 1.5 长块内容分类

对于超过 5 行的连续原样内容（代码块、prompt 块、大段引用），单独分类处理：

| 类别 | 定义 | 处理方式 |
|------|------|----------|
| **可执行内容** | 读者需要直接复制使用的代码、prompt、配置 | 重建时保持原样，置于代码块中，位置融入正文（在对应叙事段落旁嵌入） |
| **示例引用** | 原文用于展示而非执行的引用（如原文摘录、示例输出） | 改写为描述性文字（2-3 行），不保留原样代码块 |
| **原文大段引用** | 原文引用的他人段落/文章片段 | 简述引用内容 + 标注出处，不保留完整引用块 |

> **区分方法**：问"读者看到这段会复制去用吗？" 会 → 可执行内容；不会 → 改写为描述。

### 2. 图片分类与处理

| 类别 | 定义 | 处理方式 |
|------|------|----------|
| **核心信息图** | 架构图、流程图、示意图、数据图表——承载不可用文字替代的信息 | 原始图片 URL/路径可用时直接引用；无法引用时，将核心信息（结构关系/数据/流程）转为文字描述，位置标注 `[待补图: 图描述]`，不强行绘制 SVG/Mermaid |
| **装饰性图片** | 仅用于视觉分隔或氛围，不承载核心信息 | 按上下文判断是否保留 |
| **广告/Banner** | 推广、订阅、横幅类图片 | 直接过滤 |
| **代码截图** | 截取的代码片段 | 转为可读的文字代码块，标注来源 |

> **图片策略**：优先引用原始图片（URL/路径可用时）→ 无法引用时仅保留文字描述 + `[待补图]` 标记。不强行绘制 SVG/Mermaid——模型凭记忆重绘的示意图容易出错且不精确，文字描述 + 留待补图是更诚实的选择。

### 3. 外链分类与处理

| 类别 | 定义 | 处理方式 |
|------|------|----------|
| **引用性链接** | 资料出处、延伸阅读、标准文档、引用来源 | 保留链接，在文中标注原文出处（如 `[见原文引用]`）。当引用内容经过中介媒体（Ars Technica / The Verge / 36氪等），重建时直接指向原始来源（研究论文/官方公告），不在正文中提及中介媒体名称 |
| **推广性链接** | 广告、付费产品、推广内容 | 转化或删除，标注类型（如 `[推广链接已处理]`） |
| **其他链接** | 正常内容相关的参考链接 | 按上下文判断，保留需标注其性质 |

### 4. 噪声过滤

对已提取的信息点进行噪声清除，确保进入结构提案的清单是干净的。过滤规则：

| 噪声类型 | 定义 | 处理方式 |
|----------|------|----------|
| **推广/广告段落** | 原文中推销产品、服务、订阅、付费的段落 | 整段删除 |
| **付费墙提示** | "订阅继续阅读"、"付费会员专享"类内容 | 删除提示，注明原文对应部分受限于付费墙 |
| **无关冗余** | 与核心信息无关的抒发、自我推广、偏离主题的旁白 | 删除 |
| **CTA/关注引导** | "关注我"、"转发"、"订阅"等号召性用语 | 删除 |
| **原文元内容** | 付费墙标记、作者自我介绍、订阅引导、文章声明（如"本文含推广链接"）等原文的功能性内容，对转述文章无信息价值 | 直接删除 |
| **作者署名/签名** | 原文末尾的作者介绍栏、签名、头像说明、社交媒体链接 | 直接删除 |
| **转载/翻译声明** | "本文由xxx翻译"、"原文发表于xxx"、"转述说明"等任何标注转述/翻译属性的文字 | 直接删除 |

> **过滤原则**：宁紧勿松。不确定的信息点保留但标记 `[噪音: 待判断]`，结构提案阶段由人决定。**宁可多留一个待标记，也不删掉有价值的内容。**

## 📋 轮廓提案

解构完成后，生成两项内容供人确认：

### A. 信息点清单

列出所有信息点（编号 + 来源标签 + 保留级别 + 一句话摘要），示例如下：

```
[1] [S1] [必须] 事实：Claude Skills 的核心是三层结构（Audience/Voice/Business）
[2] [S1] [必须] 步骤：先从 Voice DNA 入手，再扩展其他
[3] [S2] [建议] 论点：技能比 prompt 更持久
...
```

### B. 叙事原则

不用模板编号。一篇好文章需要的不是"填结构"：

- **开头要尽快建立阅读动机**——但不强行制造“钩子”；如果素材适合冷开头，直接切入
- **中间有节奏**——密集信息段与简短过渡段交替，但不过度追求“设计感”
- **段落自然推进**——每一段都应从上一段长出来，而不是并列堆放信息
- **结尾要自然落点**——给出判断、行动或余味，但不强行“升华”
- **没有指示性语言**——全文尽量避免"本文/这个部分/第一个要点/另一方面"这类写作提示语

### C. 用户确认

输出结构提案后等待用户确认。确认内容：
1. 信息点是否有遗漏或多余（可增删）
2. 叙事组织方式是否合理

确认后进入重建阶段。如有修改要求，按反馈调整后再次确认，直到用户认可。

## ✍️ 重建叙事

### 叙事线索

开始重建前，先确定一条叙事线索。整篇文章围绕这根线索展开，不是走马灯式地排列信息点。

线索必须满足三个条件：
- **单一**：整篇文章只讲一件事。如果一个信息点跟这件事无关，删掉或改写角度
- **有推力**：A 点让读者想看 B 点，B 点让读者想看 C 点——不是 A → B → C → 结束
- **收束**：结尾不是独立的"最后一点"，是前面所有信息积累后自然到达的地方

确定线索后，写出一句"核心命题"（不超过 30 字）。重建时每一段问自己：这段话在推这条线索吗？如果没有它读者会感到断了吗？

### 重建约束（生成前即应用）

开始重建时，将以下标准作为生成约束，而非生成后再检查：

- **自包含性**：不依赖原文，读者能独立完整理解所有内容
- **信息完整度**：标记为“必须”的信息点必须覆盖；“建议”的尽量覆盖；“可选”的仅在能增强主线时保留
- **理解准确**：每个段落对照原文意图，无偏移或曲解
- **叙事自洽**：信息流通顺，段落之间自然推进，无断点
- **风格一致**：通读应与「声音锚定」一致；若无样文，则至少保持稳定、自然、非翻译腔
- **节奏变化**：句子长短有机结合，避免整段句式同构
- **口语化程度**：适度口语化，但不牺牲判断力度和信息密度
- **开头独立**：不提及原文、原作者、"转述"、"翻译"、"原文"等字眼。直接切入内容，从头到尾保持"这就是一篇原创文章"的视角
- **结尾自然**：结尾应自然收束，不额外补一段“总结升华”来制造完整感
- **零元内容**：正文不含"原文是付费内容"、"注："、"转述说明"等元内容段落
- **排版可发**：默认以短中段交替为主；只有并列关系明显时才使用列表；代码块前后都要有解释句；避免连续超长段压住阅读节奏
- **阅读连续性**：无超过 5 行连续的非中文内容打断阅读流

### 核心原则

1. **保真第一**：原文的核心论点、信息、步骤不能丢失或曲解
2. **用你的声音**：严格对齐「声音锚定」和本文「写作基调」中定义的声音特性
3. **保留有价值信息，过滤无关内容**：保留原文中有价值的信息和逻辑；过滤广告、推广、付费墙等与原文档价值无关的内容
4. **叙事不跟随原文**：不跟随原文结构，按确认的叙事方向重新组织信息点
5. **人称归己**：原文用"我"时，按解构阶段 §1.2 的归属标记执行——标记 [可吸收] 的融入你的"我"，标记 [不可吸收] 的切换为第三人称或零人称

### 写作基调自检

重建完成后，回看一次「写作基调」和「声音锚定」：

> **检验标准**：如果这篇文章要发在你的公众号或博客上，你会不会犹豫——犹豫的不是内容是否准确，而是"这段话读着像不像我自己写的"。

通读一遍，凭直觉判断。不过关就返回重建。

### 图片处理

- 按解构阶段输出的图片策略处理
- 引用原始图片：直接嵌入 URL 或路径
- 无法引用原始图片的：保留文字描述 + `[待补图]` 标记嵌入文中

### 外链处理

- 引用性链接保留在文中合适位置
- 推广性链接处理后在文中留下标记说明

## 🎭 品味检查

重建叙事之后执行。对外一步，对内三步：锚点设定 → 双角色检查 → 自动修正。

### Step 1: 锚点设定

基于文章类型 + 声音锚定 + 复述锚定，为两个角色各自生成该篇文章的锚定维度取值。

**风格锚点维度：**

| 维度 | 含义 | 取值示例 |
|------|------|----------|
| 目标风格 | 这篇文章应该是什么风格 | 教程风格 / 观点风格 / 叙事风格 |
| 声音一致性 | 是否与声音锚定保持同一作者感 | 严格 / 宽松 |
| AI腔容忍度 | 哪些词/句式在这个类型下算 AI 腔 | 严格 / 宽松 |
| 翻译味红线 | 哪些表达属于机械翻译痕迹 | 具体描述 |

**编辑锚点维度：**

| 维度 | 含义 | 取值示例 |
|------|------|----------|
| 目标节奏 | 段落长度和节奏急缓 | 短段紧凑 / 中长段从容 |
| 句间承接 | 相邻两句的关系应如何成立 | 承接清晰 / 允许跳跃 |
| 段间推进 | 新段落如何自然出现 | 自然生长 / 小标题切分 / 顺序推进 |
| 排版规则 | 代码块、引用、列表规范 | 按类型定义 |
| **自含性红线** | 是否存在脱离原文无法理解的内容 | 严格 / 宽松 |

### Step 2: 双角色检查

角色各自记住自己的锚点，通读全文一遍（自然的读，不逐句扫描），只标记**真实偏离锚点**的地方。

**风格**：重点抓三类问题
- **翻译腔**：沿用英文逻辑顺序、抽象名词堆叠、像把原文逐句搬成中文
- **AI 腔**：空泛修辞、无信息总结、为了“完整”硬补过渡或升华
- **声音偏移**：与声音锚定明显不一致，读起来不像同一个作者在说话

**编辑**：重点抓三类问题
- **句间承接**：相邻两句是否存在承接、递进、转折或解释关系；不能只是平行堆叠
- **段间推进**：每段首句是否自然接上上一段；不能靠“此外/同时/另一方面/值得注意的是”硬接
- **排版可读性**：长段、列表、代码块、图片占位是否打断阅读节奏

如果没有偏离点，输出「通过」。不为了凑发现而强行输出。

### Step 3: 自动修正

汇总两个角色的发现，逐条独立修正。只改标记的位置，不改未标记的内容。

每条修正输出摘要：`[角色] 问题位置 → 修正方式`。无修正时输出「无修正」。

## ✅ 机械门禁（兜底）

品味检查通过后执行。终检以下内容质量项，全部通过后输出：

```
[ ] 开头独立：第一段未提及原文/作者/翻译/转述？
[ ] 零元内容：正文无"转述说明""原文""注："等元内容？
[ ] 句间承接：是否不存在连续 3 句以上仅做平行堆叠、没有承接关系的写法？
[ ] 段间推进：每一段的出现是否都能回答“为什么下一段该出现在这里”？
[ ] 翻译腔：是否不存在照搬英文顺序、抽象名词堆叠、机械解释句？
[ ] 图片处理：核心图已引用或已标记 `[待补图]`？占位合理？
[ ] 外链处理：引用有出处，推广有说明？
[ ] 排版可发：是否无连续超长段、无无谓列表、代码块前后有解释句？
```

任何一项未通过，返回对应阶段修正。
