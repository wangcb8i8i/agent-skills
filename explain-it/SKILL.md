# 解释技能

将代码或文档转化为用户可以复述的心智模型。

本技能的设计受以下原则驱动：

1. **存在先行** — 解释的起点是"它为了解决什么痛苦而生"，而非结构
2. **认知降维** — 复杂度由 AI 消化，用户每屏只看到一层深度
3. **可运行思维** — 理解 = 能在思维中运行模型并预测行为
4. **叙事黏性** — 抽象真理需要类比、反直觉点和微型叙事才能固化
5. **双向校准** — 解释是认知对齐的对话，而非独白

## 工作流

```
用户问题
  │
意图理解    ← 用户想问什么类型的问题？需要什么信息？
  │
信息收集    ← 根据意图确定材料范围，只读必要部分
  │
初始回答    ← 输出形状由问题类型决定，不多不少
  │
  │  用户：「X 部分还不太清楚」
  │              ↑
  └── 追问循环 ┘ 只补缺口，可无限递归
```

---

### 阶段一 · 意图理解

#### 输入

用户的原始问题。

#### 步骤

**① 分类问题类型（6 选 1）**

| 用户说 | 类型 |
|--------|------|
| "这是什么" "XX 是做什么的" | 识别型 |
| "怎么运作" "流程" "数据怎么流" "调用链" | 流程型 |
| "由什么组成" "结构" "架构" "组件" "模块" | 结构型 |
| "为什么这样" "设计理由" "设计依据" | 理由型 |
| "A和B什么区别" "vs" "优缺点" | 对比型 |
| 已有一轮解释 + "X部分再细点"/"那Y呢" | 追问型 |

**② 提取目标概念**

从问题中提取用户想理解的核心事物，不锚定当前打开的文件。

| 用户说 | 目标概念 |
|--------|----------|
| "我想了解工作流全貌" | 工作流（跨文件，非当前文件结构） |
| "这个函数怎么用" | 这个函数 |
| "为什么用Redis不用Memcached" | Redis vs Memcached 选型理由 |

**③ 确定信息源**

由目标概念决定搜索范围：

| 目标概念 | 可能的材料源 |
|----------|-------------|
| 工作流 | pipeline 定义 / orchestrator / handler 链 / 配置 |
| 函数功能 | 函数定义 + 调用处（如需） |
| 项目架构 | 目录结构 / 依赖图 / 入口 / 路由 |
| 技术选型 | ADR / 设计文档 / issue |

**④ 范围确认**

- 问题明确 → 直接走
- 模糊且上下文不足 → 只问一个澄清问题
- 范围超大（"解释整个系统"）→ 先出目录，让用户选

---

### 阶段一.5 · 前置检查

在进入信息收集前，快速确认用户对必要前置概念的理解程度。

决策：用户对目标概念的前置依赖掌握到何种程度？

```
  ├─ 已掌握（session 内已有证据 / 用户明确表示知道）→ 跳过
  ├─ 未掌握（用户表示不知道 / 材料中出现的术语超出用户认知）→ 先输出 1-2 句铺垫解释，再进入阶段二
  ├─ 有误解（用户表达的理解与已读材料确认的事实不符）→ 先澄清误解再继续
  ├─ 模糊且无用户基线信息 → 用一个轻量询问辅助判断（"你目前对[主题]了解到什么程度？"），不超过 1 次
  └─ 无法判断 / 通用常识 → 默认掌握，不检查
```

**不检查的内容**：
- 跟当前问题无关的背景
- 用户不可能预先知道的内部实现细节
- 通用的编程或领域常识
- 解释性或判断性差异（如"这个设计不好"）——仅检查可客观验证的事实性错误

---

### 阶段二 · 信息收集

#### 原则

- **读最少材料回答问题**，不是读完整文件再总结
- 当前上下文不够 → 搜索定位需要的段落，不猜

#### 按类型的读取边界

| 类型 | 读取范围 | 跳过什么 |
|------|----------|----------|
| 识别型 | 入口/标题/摘要。≤ 10 行 | 所有实现细节、分支、配置 |
| 流程型 | 执行路径上的关键函数、转手点、分支点 | setup/import/error handler 等非主线 |
| 结构型 | 目录结构、模块边界、接口/签名 | 函数体所有实现 |
| 理由型 | 注释、docstring、决策记录 | 实现代码 |
| 对比型 | 双方的定义和关键差异点 | 共同之处 |
| 追问型 | 仅用户指的那一块 | 一切已覆盖过的内容 |

#### 降级

需要读 >3 个文件才能回答且是横向搜索（不同模块、不同层次的组件）→ 先出地图，让用户选方向。
纵向流程追踪不触发此降级（跨 N 个文件的调用链是正常需求）。

---

### 阶段三 · 初始回答

#### 输出形状由类型决定

| 类型 | 输出结构 | 推荐视觉模式 |
|------|----------|-------------|
| 识别型 | 一句话存在理由 + 一句话定位 + 2-3 个关键事实 | 无图，或树形图（有嵌套时） |
| 流程型 | 一句话存在理由 + 端到端流程图 + 每步标注：谁做的、什么进、什么出 | Call flow / Sequence / State/Transform |
| 结构型 | 一句话存在理由 + 结构图 + 组件职责表（每组件一行） | Structure / Hierarchy / Compact table |
| 理由型 | 决策表或论证链 | Decision table / Reasoning chain |
| 对比型 | 对比表：准则列 / 选项列 / 突出决定性差异 | Compact table / Options-compared |
| 追问型 | 只输出缺口，不加前后缀 | 由缺口实质决定 |

#### 格式约束

- 对话输出 ASCII，禁 mermaid
- 每张图通过可视化决策流独立判断
- 一张图最多，特殊情况两张
- 不加内部流程标签
- 不加自检过程
- 不加"根据我的分析""我注意到"等填充语
- **输出规模约束**：单个 ASCII 图超过 35 行 → 自动拆为多层
  上层粗粒度总图（≤ 35 行），标注"各步骤详情见下方"
  下层分段可全部输出，用户自主选择精读段
- **术语一致性**：首次出现陌生技术概念/缩写时，用 `「X ≡ Y」` 行内定义锚定。同 session 后续引用同一术语直接使用，不再重定义。术语来源必须来自已读材料，不发明定义
- **核心骨架先行**：需展开 3 个以上节点的复杂解释，先输出一屏可概览的骨架（≤5 个要点），标注"下方逐点展开"，再分段深入
- **类比 + 失效声明**：使用类比时，必须附带一句类比失效声明，格式：`这个类比在[映射点]上成立，但在[失效处]不成立，因为…`
- **反直觉触发器**：识别型和理由型回答中，至少暴露一个反直觉事实或常见误解。用"常犯的一个错误是…"或"令人意外的是…"开头
- **口语化转述**：含 3 个以上技术术语的解释，末尾输出一段无术语的口语化复述，开头为"如果用大白话说，…"，≤3 句

#### 不多不少原则

用户问"这个函数流程"，回答应包含的是调用链 + 参数传递。不应包含：
- 这个文件的目录结构
- 同模块其他函数
- 设计理由（除非用户明确问）
- 建议修改方向

#### 可选自测

- 回答后可选附 1-2 个精加工自测问题，开头为`想验证是否理解了？试着回答：…`
- 严禁使用`清楚了吗``理解了吗``有问题吗`等封闭式追问
- 自测是提供给用户的工具，不是要求——用户可忽略

### 阶段四 · 追问循环

#### 循环起始

用户发起任意形式的缺口信号：

```
"那Y呢"                 → 范围迁移，目标概念从X换成Y
"X部分再细点"           → 同概念深度请求，类型继承
"为什么那步要这样处理"   → 同一概念的纵深，进理由
"你刚说的XX是什么"      → 术语解释，进识别型
"继续"/"还有呢"         → 给下一个有意义的块，不是"你想看哪里"
```

#### 处理

- **重走意图分类**：只做①分类，不重走②③④。分类结果可能改变
  例：初始流程型 → 追问"为什么" → 重新分类为理由型
- **无输出模板约束**：只回答缺口，不匹配阶段三的形状表
- **无深度上限**："再细点"可无限递归。第1次追问进实现细节，第3次进边界条件和错误路径
- **风格适应**：用户在追问中显式表达的解释偏好（粒度、图/文、组织方式），同 session 后续回答自动沿用。不主动询问偏好
- **推演测试**：对流程型和结构型的第 2 次以上递归追问，末尾附一个"如果…会怎样"极端条件设问，帮助用户检验心智模型的边界

#### 结束

- 用户沉默 = 结束。不主动问"还有问题吗""要进入收尾吗"
- 用户说"明白了""好的" → 结束

## 可视化决策流

每张图独立判断：

```
信息点包含 ≥2 个元素之间的结构关系？
  ├─ 否 → 纯文字
  └─ 是 → 关系能用一句话说清？
       ├─ 能 → 纯文字
       └─ 不能 → 图能让读者一眼看到关键信息？
            ├─ 能 → 画图
            └─ 不能 → 纯文字
```

**最低图形门槛**：
- 流程型、结构型初始回答必须至少包含一张图
- 除非关系简单到三句话内能说清

**锚点约定**：
- 代码引用：用 `文件路径:行号` 标记关键位置
- 文档引用：用 `§章节号` 标记关键章节

---

## 自检（每次回答前执行，不输出）

**发现小漏** → 修正后继续。
**发现大漏或误读** → 回到信息收集阶段补读，修正后重新输出。
**无法确认又无法放弃的断言** → 以低确定性表述输出。
**无法确认的内容过多** → 缩小回答范围，只输出能确认的部分。

---

## 产物输出

默认不写文件，解释留在对话中。仅在用户明确要求持久化时执行：

- 写入用户指定路径；未指定则建议 `docs/read-notes/<文件名>.md`
- 写 markdown 时使用文件环境下的可视化格式（mermaid 图、全宽表格），而非终端 ASCII
- **对话直接输出解释时，必须使用 visual-patterns.md 定义的终端原生 ASCII 视觉模式（`├─→` `└─→` 等）；严禁在对话中输出 mermaid 代码块。仅用户明确要求持久化写 markdown 文件时，才转用 mermaid。**
- 文件头部注明源文件/文档路径、日期和解释范围
- 写入后确认路径即可，不在对话中重讲一遍

### 多轮解释产物策略

保存时输出结构化文档，非原始对话拼接。文档结构：
1. **结构图** — 用 mermaid 或层级图呈现全结构，标注已探索区域
2. **已探索部分摘要** — 每个已探索部分的核心内容，各一至两段
3. **完整链路** — 端到端逻辑链

不包含：用户追问记录、原始对话拼接。读者拿到文件后应能独立理解全部内容。

## 硬边界

不得：
- 修改、创建或删除任何文件（产物输出中用户明确要求时除外）
- 逐行复述代码或逐章复述文档
- 解释超出用户问题所需范围的内容
- 将推断呈现为已确认的事实
- 从局部检查中暗示已掌握全路径/全文
- 将解释与实现建议、批评、审查或合著混在一起，除非用户要求
- 添加不能阐明结构、流程或关系的图表
- **文档场景补充**：空白只标注不填补——不得补充文档"本应说"的内容
- 在输出中使用"意图理解""信息收集""初始回答""追问循环"等内部流程标签——用户看到的是解释内容，不是流程的阶段名
- 输出自检过程——自检是内部闸门，用户只看到校验后的结果
- 在未读取必要材料的情况下作答——所有输出必须有已确认的信息源支撑
- 回答后**不主动评价**（"理解了吗""清楚了吗"），不强迫用户表态。允许提供可选的自测工具（见"可选自测"），但必须是工具性的、非评价性的、可忽略的
- **允许提供选择型延伸**（"主路径之外有个错误恢复路径，要看吗？""还有个相关概念要了解吗？"）——这由具体上下文决定，不是固定模板
- 延伸提示**一次最多一个**，避免变成目录
