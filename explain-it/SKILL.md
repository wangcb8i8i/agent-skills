# 解释技能

将代码或文档转化为用户可以复述的心智模型。

## 工作流

```
场景判定
  │
层1 · 定向    ← 这是什么？核心结论/动机是什么？
  │              ↓ 停靠确认
层2 · 结构化  ← 由什么组成？各部分如何关联？
  │              ↓ 停靠确认
层3 · 深挖    ← 关键部分如何运作？（可选，可重复）
  │
收尾 · 固化   ← 我能从头到尾说清楚了
```

### 场景判定

从用户消息和上下文自动判定目标对象类型：

- `.py` / `.ts` / `.js` / `.rs` / `.go` / `.c` / `.h` / `.java` / `.kt` 等代码扩展名
- 消息中包含 `def ` / `function` / `class` / `interface` / `const` / `fn` 等代码语言信号
- 当前编辑器中打开的文件扩展名为代码类型 → **代码**
- `.md` / `.pdf` 扩展名
- 消息中包含 `提案` / `规格` / `决策` / `报告` / `proposal` / `spec` / `ADR` / `RFC` 等文档信号 → **文档**
- 不明确时，问一个澄清问题

判定后，各层标注 **→ Code / → Doc** 处需参考对应场景的参考卡片。如果用户明确指定了回答深度（如"一句话总结""审视"），用它来限制或跳过某些层。

---

### 层1 · 定向

用户在此层结束时能回答：**这是什么？核心结论/动机是什么？**

#### 准备

读刚好够定位的材料——入口/标题/摘要/结论区。不要读完整份材料。不需要理解全部细节。

→ **Code**：见 `references/code.md §1`
→ **Doc**：见 `references/doc.md §1`

#### 输出

1. **一句话定位**：核心功能/主张。从"动机"出发——"为什么有这段代码/文档？它解决什么问题？"
2. **全局轮廓**：高层结构一览，每个部分/章节一句话角色

格式约束：
- 一段话（结论）+ 一张图（全局结构）
- 图必须独立通过可视化决策流（见下）

**可视化决策流**（每张图独立判断）：

```
信息点包含 ≥2 个元素之间的结构关系？
  ├─ 否 → 纯文字
  └─ 是 → 关系能用一句话说清？
       ├─ 能 → 纯文字
       └─ 不能 → 图能让读者一眼看到关键信息？
            ├─ 能 → 画图
            └─ 不能 → 纯文字
```

**最低图形门槛**（所有场景通用）：
- 层1必须至少包含一张全局结构图
- 除非结构关系简单到三句话内能说清

**锚点约定**：
- Code：用 `文件路径:行号` 标记关键位置
- Doc：用 `§章节号` 标记关键章节

#### 停靠

用户可在此选择是否继续。如果继续，进入层2。

> 用户停在此处应能复述："这是关于 XX 的东西，它主要由 A、B、C 组成"

---

### 层2 · 结构化

用户在此层结束时能回答：**由什么组成？各部分如何关联和协作？**

#### 准备

读组件/章节的正文内容，收集关系信息。这一层的阅读量取决于规模——够识别组件和连接即可。

→ **Code**：见 `references/code.md §2`
→ **Doc**：见 `references/doc.md §2`

#### 输出

1. **组件/章节拆解**：每个组成部分的一句话职责
2. **关系网络**：各组件/章节之间的连接关系
   - Code：调用链 / 数据流
   - Doc：论证链 / 引用关系
3. **视觉图**：一张展示结构和关系流向的图

格式约束：
- 职责表 + 关系图，两者缺一不可
- 图同样通过可视化决策流判断

#### 停靠

用户可在此选择是否需要深挖某个具体部分。每选中一个部分进入一次层3。

> 用户停在此处应能复述："A 做 XX，传给 B 做 YY，C 把结果做了 ZZ"

---

### 层3 · 深挖

用户在此层结束时能回答：**这个具体部分是如何运作的？**

可选，可重复。每次深挖一个独立部分。

#### 准备

读选中部分的详细内容——函数体、分支、异常路径，或具体章节的完整论证。

→ **Code**：见 `references/code.md §3`
→ **Doc**：见 `references/doc.md §3`

#### 输出

1. **机制描述**：该部分的详细运作方式
   - Code：实现逻辑 + 有意义的分支 + 边界情况
   - Doc：论证过程 + 支撑证据 + 假设
2. **设计依据**：为什么这样设计/论证（如果有可说明的）

#### 停靠

用户可选择继续深挖其他部分，或进入收尾。

> 用户停在此处应能复述："A 内部是这样运作的，特殊情况下会……"

---

### 收尾 · 认知固化

用户在此阶段结束时能回答：**我能从头到尾说清楚全部了。**

#### 准备

无额外阅读。所有信息在之前各层中已获取。

#### 输出

必须输出三个部分，缺一不可：

1. **完整链路重建**：从动机到最终输出/结论的完整路径。把层1到层3的认知拼接为一条逻辑链。
   - Code：输入→各阶段处理→输出，每步谁做的、怎么转手的
   - Doc：前提→各层推理→结论，每步的证据和推理

2. **核心关系回顾**：点出关键连接点——"A 的输出决定了 B 的输入"、"§3 的假设支撑了 §4 的结论"

3. **一句话浓缩**：一段话，用户可以带走复述。覆盖"它是什么、由什么组成、如何运作/论证、最终得出什么"

输出后确认结束。用户未读完这三个部分之前，不离开交互。

---

## 自检（每层交付前执行，不输出）

每层输出前做一次快速内部确认：

→ **Code**：见 `references/code.md §4`
→ **Doc**：见 `references/doc.md §4`

**发现小漏** → 修正后继续本层输出。
**发现大漏或误读** → 回到本层准备阶段补读，修正后重新输出。不能放有问题的解释出去。
**无法确认又无法放弃的断言** → 以低确定性表述输出。
**无法确认的内容过多** → 缩小本层解释范围，只输出能确认的部分。

---

## 产物输出

默认不写文件，解释留在对话中。仅在用户明确要求持久化时执行：

- 写入用户指定路径；未指定则建议 `docs/read-notes/<文件名>.md`
- 写 markdown 时使用文件环境下的可视化格式（mermaid 图、全宽表格），而非终端 ASCII
- 文件头部注明源文件/文档路径、日期和解释范围
- 写入后确认路径即可，不在对话中重讲一遍

### 多轮解释产物策略

保存时输出结构化文档，非原始对话拼接。文档结构：
1. **结构图**（来自层2）— 用 mermaid 或层级图呈现全结构，标注已探索区域
2. **已探索部分摘要**（来自层3）— 每个已探索部分的核心内容，各一至两段
3. **完整链路**（来自收尾）— 端到端逻辑链

不包含：用户停靠选择记录、原始对话的逐层记录。读者拿到文件后应能独立理解全部内容。

---

## 硬边界

不得：
- 修改、创建或删除任何文件（产物输出中用户明确要求时除外）
- 逐行复述代码或逐章复述文档
- 解释超出用户问题所需范围的内容
- 将推断呈现为已确认的事实
- 从局部检查中暗示已掌握全路径/全文
- 将解释与实现建议、批评、审查或合著混在一起，除非用户要求
- 添加不能阐明结构、流程或关系的图表
- **文档场景补充**：空白只标注不填补——不得补充文档"本应说"的内容
- 跳过层1（定向）直接进入层2或层3
- 在一层中倾泻全部分析——每层只输出该层约定密度的内容
- 在输出中使用"层1""定向""结构化"等内部流程标签——用户看到的是解释内容，不是流程的阶段名
- 输出自检过程——自检是内部闸门，用户只看到校验后的结果
- 在收尾前离开交互——收尾的三个部分必须全部输出
