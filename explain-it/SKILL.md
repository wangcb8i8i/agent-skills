# explain

将代码或文档转化为用户可以**回音**（复述并预测）的心智模型。只解释，不实现、不审查。

## 原则

1. **存在先行** — 起点是"它为了解决什么痛苦而生"，而非结构
2. **手术刀** — 精准切口，只切必要材料，只答问题形状
3. **骨架** — 先出可概览的骨架，再逐点展开
4. **缺口** — 识别用户理解中的具体空洞，缺哪补哪

## 工作流

### 1. 识别缺口

完成：缺口类型（6选1）+ 目标概念 + 材料范围 + 前置掌握程度 已确定

**① 分类（6 选 1）**

| 用户说 | 类型 |
|--------|------|
| "这是什么""XX是做什么的" | 识别型 |
| "怎么运作""流程""调用链" | 流程型 |
| "由什么组成""结构""架构" | 结构型 |
| "为什么这样""设计理由" | 理由型 |
| "A和B什么区别""vs" | 对比型 |
| 已有解释+"X部分再细点"/"那Y呢" | 追问型 |

**② 提取目标概念 + 确定信息源**

| 目标概念 | 可能的材料源 |
|----------|-------------|
| 工作流 | pipeline 定义 / orchestrator / handler 链 / 配置 |
| 函数功能 | 函数定义 + 调用处（如需） |
| 项目架构 | 目录结构 / 依赖图 / 入口 / 路由 |
| 技术选型 | ADR / 设计文档 / issue |

**③ 前置掌握度检查**

- 已掌握 → 跳过
- 未掌握 → 先1-2句铺垫
- 有误解 → 先澄清
- 模糊 → 轻量询问一次

范围：模糊且上下文不足 → 只问一个澄清问题；范围超大 → 先出目录，让用户选。

### 2. 手术刀切入

完成：按缺口类型的读取边界，必要材料已读取

按类型读取边界 → `types.md`

>3个非连续文件 → 先出地图让用户选方向。

### 3. 构建骨架

完成：输出匹配缺口类型的形状约束，含骨架视图

按类型输出形状 → `types.md`
需要时查阅视觉模式 → `patterns.md`

### 4. 缺口填补循环

完成：用户沉默或说"明白了"

用户追问 → 重分类（可能变类型） → 只补缺口，不重读已覆盖内容。

- 无深度上限，第3层起附加方向提示：`继续往下是 [边界条件/…]，要继续吗？`
- 用户说"继续" → 解除引导；说"不用"→ 结束；沉默 → 结束
- 用户表达的解释偏好（粒度、图文、组织）自动沿用同 session 后续回答
- 不主动问"还有问题吗""要进入收尾吗"

可选自测：附1-2个精加工自测问题，开头为`想验证是否理解了？试着回答：…`。严禁`清楚了吗`等封闭式追问。

## 格式约束

- 对话输出 ASCII，禁 mermaid。写文档时才转 mermaid
- 使用类比时附带失效声明
- ≥3技术术语时末尾加白话复述
- 识别型和理由型至少暴露一个反直觉事实
- 代码引用用 `文件:行号`，文档引用用 `§章节`

## 硬边界

- 不修改文件（用户要求持久化时除外）
- 不逐行复述代码
- 不将解释与实现/审查混在一起
- 不输出自检过程
- 不主动评价，不强迫表态
- 所有输出必须有已读材料支撑

持久化产物输出 → `output.md`
