# 结构型

用户想知道"由什么组成"——需要理解模块职责和组织关系。

---

## 读取边界

读取范围：目录结构、模块边界、接口/类型签名。

跳过：所有函数体实现、分支逻辑、配置值。

---

## 输出形状

```
结构存在的理由 → [ASCII 结构图] → 各模块职责一句话
```

不要一上来就给图。先回答"为什么这样组织"，再给图。

---

## 视觉模式

### Nesting（层级图）

显示包含关系和职责，不画时间线，不画调用方向。

```
[entry]
  └─→ [orchestrator]
        ├─→ [domain service]
        ├─→ [cache]
        └─→ [repository]
```

每个节点标注一句话职责，不要只标名字。

示例：

```
[controller]          ← 接 HTTP 请求，参数校验
  └─→ [search service]  ← 编排搜索流程
        ├─→ [query builder]   ← 把参数拼成查询语句
        └─→ [result serializer] ← 把结果转成响应格式
```

### Compact table（结构中的角色对比）

Nesting 不够用时，用表格明确各模块的差异化职责：

```
| 模块 | 职责 | 谁依赖它 |
|------|------|----------|
| ...  | ...  | ...      |
```

---

## 完成标志

用户能复述"系统分几层，每层管什么"。
