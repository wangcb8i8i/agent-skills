# 冷启动

在 SKILL.md 步骤 A 中使用。为"还不知道要问什么"的用户建立第一张心智地图。

---

## A1. 动机先行 — 信号衰减链

| 优先级 | 信号源 | 读什么 | 适用 |
|--------|--------|--------|------|
| 1 | README / 项目首页 / 包简介 | 第 1 段 + 标题 | 项目级 |
| 2 | 包元数据 | package.json description / Cargo.toml 注释 / go.mod module line | 有包管理 |
| 3 | 目录 + 入口 imports | 目录结构 + 入口文件 import 行 → 推断领域 | 项目/模块 |
| 4 | 构建/部署配置 | Dockerfile CMD / Makefile targets / CI 配置文件 | 项目/模块 |
| 5 | 模块公开 API | 导出函数/类型签名 → 推断职责 | 无文档模块 |
| 6 | 调用上下文 | 谁 import 这个模块 / 父级调用链路 | 被内嵌模块 |
| 7 | 问用户 | "这个项目/模块是做什么的？" | 以上全不可用 |

从优先级 1 开始，读不到信号就降一级，直到能回答"它为了解决什么痛苦而生"。
一旦能回答，立即停止，不继续降级。

## A2. 地形压缩 — 输出格式

```
动机：______________________

骨架（3 块）：
  1. [最大块：用途/目标]     ← 这一块是动机的展开
  2. [第二块：关键机制]       ← 它靠什么实现（1 的支撑）
  3. [第三块：边界/约束]      ← 它不做什么（2 的约束）
```

规则：
- 每块 ≤ 15 字
- 块之间有因果链：2 支撑 1，3 框住 2
- 不列目录结构（那是信息，不是心智模型）
- 不要在骨架后追加"实际上还有很多细节"——骨架就是骨架，满了就停

示例（Kubernetes）：

```
动机：自动管理容器化应用的部署和扩缩

骨架（3 块）：
  1. 声明式期望状态 —— 你说"跑3个副本"，系统会维持3个
  2. 控制循环 —— 控制面不断对比期望和实际，调平差异
  3. 调度约束 —— 节点资源不够时就排队等待，不杀死已有容器
```

## A3. 切口引导 — 输出格式

输出 2-3 个探索方向，让用户从"无头绪"过渡到"能问出第一个问题"。

```
你可能可以了解：
  - [方向 A —— 用一句话描述这个方向解决什么困惑]
  - [方向 B]
  - 或者你有自己的方向？
```

方向来源：
- 从地形骨架中选取最可能让人困惑的一块
- 从阅读材料中遇到的"常见误解"或"最易误用点"
- 不要列所有可能方向，只列 2-3 个最有入口感的

示例：

```
你可能可以了解：
  - Pod 是怎么被调度到节点上的 —— 控制循环中最核心的一步
  - 声明式配置怎么写 —— 你会接触到最多的部分
  - 或者你有自己的方向？
```
