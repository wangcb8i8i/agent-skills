# 代码场景参考卡片

## §1 定向准备：入口定位 + 核心功能提取

从代码中定位：
- 真正的入口点（main / handler / route / entry function）
- 核心输入输出（什么进来，什么经过处理，什么出去）
- 一句话功能描述

检查足够的代码以识别 — 四个要素：
- **入口点**：实际被调用的起点，不是看到的第一个函数
- **转手点**：职责切换的边界（controller → service → repository）
- **有意义的分支**：改变行为走向的条件
- **出口或效果**：返回、写库、发消息、渲染页面

当含义依赖周边流程时，不要仅凭孤立代码片段定位。

## §2 结构化准备：组件识别 + 调用/数据流追踪

- 文件和模块的结构关系（目录层次）
- 关键转手点和中间层
- 调用链（入口 → 每一步 → 出口）
- 数据变换（输入 → 中间形态 → 输出）

识别用户可能关心的关注点以供后续深挖：
- **职责**——这段代码的每个部分各自负责什么
- **执行流程**——控制流如何贯穿各组件
- **数据流**——输入如何逐步变成输出
- **结构**——组件之间的依赖和组织关系
- **设计理由**——为什么按这种方式组织

## §3 深挖准备：实现细节 + 分支/异常路径

- 具体函数的实现逻辑
- 有意义的分支和条件（那些改变行为的）
- 异常处理和边界情况
- 设计理由和取舍

哪些不讲：与用户问题无关的代码面、无关的分支、无关的边界情况。

## §4 分层自检清单

每层输出前执行对应条目。

### 定向自检
- 核心功能定位是否正确？（重新确认入口和输出）
- 一句话能否说清？说不清 → 定位不够准确

### 结构化自检
- 组件划分和调用关系是否正确？（跟踪一条路径确认）
- 关键转手点是否说清楚了？
- 有没有只说了"是什么"没说"为什么这样组织"的地方？

### 深挖自检
- 分支路径和边界情况是否覆盖了有意义的路径？
- 实现细节是否与代码一致？（快速重读关键行）
- 是否把推断当成了已确认的行为？

### 收尾自检
- 用户能否从输入到最终结果复述完整链路？
- 问题范围内的内容是否全部覆盖？
- 是否有未检查但可能影响理解的代码路径？
