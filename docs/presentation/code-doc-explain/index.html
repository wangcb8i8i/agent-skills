code-explain & doc-explain — 将复杂信息转化为可复述的心智模型

2026.05 agent-skills 解读 · 比较 · 设计哲学

agent-skills 项目中有两个深度互补的技能：code-explain 解读代码，doc-explain 解读文档。它们各自面向不同的输入类型，但共享同一个核心使命——将复杂、陌生的信息空间，转化为用户可以理解、记忆和复述的心智模型。

两者遵循同一套四阶段流水线架构，使用「地图+导航」的多轮交互策略处理大型任务，坚守「最小可理解单元」和「用户掌控节奏」的设计原则。在可视化策略、自检机制和输出约定上各有侧重，但硬边界高度一致。

1

共同使命

code-explain 和 doc-explain 的核心使命高度一致：将复杂信息转化为用户可以复述的心智模型。区别在于输入——代码是可执行的逻辑结构，文档是修辞性的论证推理。

代码解释需要追踪控制流、数据变换和关键转手点；文档解释需要理清前提、推理链和结论。但用户离开时能做的事相同：用自己的话向第三方解释清楚。

2

四阶段流水线

3

code-explain：代码解释引擎

六种关注点

code-explain 定义了六种解读关注点：职责（代码是干什么的）、执行流程（控制流如何贯穿）、数据流（输入如何变成输出）、结构（组件之间的关系）、设计理由（为什么这么写）和 细微之处（容易误读的地方）。这六种关注点不是固定模板，而是根据用户问题按需组合。

三种叙事结构

根据代码规模有三种选择。小型单文件用直答式：一句话结论加关键要点。跨文件流程用追踪式：从入口沿调用链逐段展开。大型代码用地图+导航式：三轮交互，第一轮画地图，第二轮深入分支，第三回合拢。每轮都是硬停，用户掌控节奏。

地图视角体系

地图不是单张图，而是一组视角的组合。默认输出结构+职责两个视角，其余按需添加。选择规则是「不自洽才加倍，不自证才加视角」——每个额外视角必须有明确的读者需求依据。五种视角覆盖：结构（层级/组成）、职责（角色/重要性）、流程（控制/数据流）、依赖（耦合关系）、变化（diff 标注）。

终端原生可视化

code-explain 坚持终端优先——使用 ASCII 图和符号矩阵。图形不是文字的降级替代，而是承载不同信息密度的平行通道。每张图只承载一个信息层，多关注点时用多个小图而非一张大图。三道决策闸门：认知负担来自哪？文字需来回翻阅？去图会打折扣？

4

doc-explain：文档解释引擎

文档类型与深度层级

doc-explain 面对七种文档类型（提案、规格、决策记录、指南、参考、报告、战略愿景），定义了四个深度层级。一句话只给结论；要点提供结构化关键信息；完整展示前提到结论的整条论证链；审视标注证据强弱和空白。信号模糊时默认选完整——因为降级比升级容易感知。

单轮与多轮输出

小型文档走单轮输出，直接回答。大型文档走多轮模式：子阶段 1 画文档地图，子阶段 2 用户选章节深入追踪，子阶段 3 将已探索章节编织为端到端推理链。每轮是硬边界——禁止在一轮中合并多个子阶段。

四重校对自检

每个输出前经过四道闸门。归属校对：结论来自正确章节（最常见的错误是把 B 章节的观点归到 A）。方向校对：检查否定词和限定词——忽略一个「不」字就能反转结论。范围校对：不暗示未读内容。确定性校对：区分文档原文与推断。通过四关才能输出——宁可范围小但准确，不要范围大但可疑。

6

设计哲学

最小可理解单元

只回答用户所需，不多解释一个函数或一个章节。首轮只输出够用但不过载的关键链路，根据用户反应决定展开还是收拢。

用户控制节奏

多轮交互每轮硬停，由用户决定深入哪条分支还是合拢。信息主权始终在用户手上。

不自洽才加倍

「不自洽才加倍，不自证才加视角」——这是对抗过度解释的纪律。每个额外信息层必须有明确的读者需求依据。

自检不输出

自检是内部质量闸门，推理过程不对用户输出。用户只看到校验后的结果和收束信息。

7

硬边界

不修改文件
除非用户明确要求持久化，否则不创建、修改或删除任何文件。
不逐行/逐章复述
根据问题组织有选择、有结构的解读，而非顺序罗列。
不超出范围
只回答用户所需，不多解释。
不混淆推断与事实
只有经直接验证的内容才能作为事实。推断必须标注。
不混合角色
解释与批评、评审、创作严格分离——除非用户要求。
不自动触发
最关键的一条：只在用户明确要求理解代码或文档时调用。
✦

一句话结论

code-explain 和 doc-explain 不是解释器，是认知转化器——把别人的逻辑，变成你自己的理解。
