工作流方法论

Research-Plan-Driven
AI 辅助开发的精度工作流

从无结构的 ad-hoc 提示，到有审批门的纪律化交付——一个为 AI 与人类协作设计的六阶段方法论。

6 个阶段 4 道审批门 12 张幻灯片 持续验证

Research-Plan-Driven 工作流将 AI 辅助开发从自由形式的聊天对话转变为规范的工程流程。它由六个依次执行的阶段组成——Task Clarify、Research、Planning、Todo List、Implementation、Verification——每个阶段之间由明确的审批门隔开。每行代码都必须经过经过验证的研究和已批准的计划来证明其合理性。这套方法论能有效减少返工、防止 scope creep，并在每个环节产出可审计的 artifact。

这并非一个关于 AI"应该怎么做"的建议。它是一个被明确编码的技能（codified skill），带有不可协商的规则（non-negotiable rules）、证据标准（evidence standards）和验证协议（verification protocols），全部经过真实使用场景的打磨。每个阶段都有清晰定义的目标、具体的交付物、完成标准，以及旨在防止错误累积的约束条件。

第一部分

Ad-Hoc AI 开发的困境

当今大多数 AI 辅助开发遵循一条直线路径：用户描述任务，AI 立即生成代码。这跳过了理解、研究和规划这三个人类工程师在写代码前通常会投入精力的步骤。后果是什么？是 hallucinated 的架构根本不匹配现有代码库、被遗漏的约束条件、悄无声息的 scope creep，以及被误认为是进展的返工循环。

当 AI agent 直接跳到代码生成时，几种典型的失败模式就会出现：

上下文缺失（Missing context）
Agent 不了解代码库真实的约定、现有的 pattern 和架构约束。它在猜测，而猜测的修正成本很高。
虚假信心（False confidence）
生成的代码看上去合理，但遗漏了边界情况（edge cases）、错误处理和集成点。没有研究环节，这些缺口在运行之前是不可见的。
范围蔓延（Scope creep）
没有书面计划和明确的审批，实现过程会不断漂移。每一轮对话都在"顺便加一个功能"，直到最初的意图被埋没。
缺乏审计轨迹（No audit trail）
当问题出现时，没有 artifact 轨迹可以追溯——什么被决定了、为什么、谁批准的。调试变成了考古。

Research-Plan-Driven 工作流用一个可预测的 pipeline 取代了这些失败模式。每个阶段都有进入条件（entry condition）、具体交付物（concrete deliverable）和完成标准（completion criterion）。没有假设，一切都被验证。

第二部分

六阶段 Pipeline 总览

该工作流是一个线性的六阶段序列，每个阶段产出一个 artifact，作为下一个阶段的契约。前四个阶段之间有显式的审批门（approval gates）：用户必须审阅并批准每个 artifact，工作才能继续推进。

Clarify ➔ 审批门 ➔ Research ➔ 审批门 ➔ Planning ➔ 审批门 ➔ Todo List ➔ 审批门 ➔ Implementation ➔ Verification

前四个阶段——Clarify、Research、Planning、Todo List——每个阶段都以 review-revise cycle 结束。用户阅读 artifact、给出意见、agent 修订、重复直到显式批准。只有在那之后，Implementation 和 Verification 阶段才以无需中途批准的方式开始执行。

第三部分

Task Clarify：行动前的共享理解

工作流不是从 research 开始的。它从结构化的对话开始。目标是通过有纪律的提问，在调研开始之前将原始想法转化为清晰、准确、明确的共享理解。

问题一次只问一个，遵循特定的方法：优先使用选择题，只在必要时使用开放式问题，避免让用户感到被淹没。每个问题针对以下五个维度之一：

目的（Purpose）
为什么要做这个？它解决什么问题？
具体交付物（Concrete deliverables）
输出应该是什么样的？我们怎么知道它完成了？
约束（Constraints）
我们必须在什么边界内工作？技术、时间、资源、依赖？
边界（Boundaries）
什么在范围内？什么与什么交互？系统边界在哪里？
非目标（Non-goals）
我们明确不做什么？说明非目标与说明目标同等重要。

这一阶段有意采用对话形式而非 artifact 驱动。输出是一个共享心智模型，记录在对话本身中，它指导所有后续阶段。

第四部分

Research：基于证据的 grounding

Research 阶段是这个工作流区分于肤浅方法的关键。它不是关于产生想法或提出方案。它是关于建立对代码库现实的经过验证的理解——任何计划都必须尊重这种现实。

研究过程遵循一个"委托-验证"（delegation-and-verify）模式。协调人将任务分解为聚焦的、不重叠的问题或子系统，然后并行派遣 1–3 个子 agent 调查每个领域。每个子 agent 收到详细的 prompt，指定要调查什么、为什么调查、以及需要返回何种类型的发现。目标是全面理解抽象、架构和控制流。

返回的发现被视为未经验证。协调人综合审查它们、解决冲突、并直接读取代码以验证关键主张或填补实质性缺口。冲突的发现不会被静默合并——模棱两可之处被显式提出。只有经过这种交叉验证后，协调人才撰写 research artifact。

每个非显而易见的发现必须包含源引用，格式为 [src: path:line]，让审阅者无需重新搜索代码库就能验证主张。Artifact 应包含相关系统概览、关键文件及其职责、数据流和控制流、现有 pattern 和约定、依赖关系、以及潜在的陷阱。

Research phase 只有在 artifact 存在、基于代码库证据、影响计划的实质性问题已被解决或提出、且用户已显式批准其用于 planning 时，才算完成。

什么不能算作充分的研究：没有行为理解的文件列表、没有综合的 grep 输出、被当作事实呈现的设计方案、未经过交叉验证的单一来源发现、或会实质性影响计划却未得到回答的问题。

第五部分

Planning：渐进披露架构

Planning artifact 建立在一个简单的洞察之上：审阅者应该在任意详细程度上都能停下来并做出有依据的判断。这一点通过一个包含七个必需部分的渐进披露结构实现，每个层次只提供前一个层次无法传达的内容。

层级部分目的
1Title & Summary我们要构建什么以及为什么，引用 research artifact。2–3 句话。
2Change Inventory要创建/修改的确切文件路径以及每个变更的意图。清单列表。
3Top-Level Approach高层策略——各部分如何组装在一起。3–5 句话。
4Resulting Shape执行后代码库的样子。必须包含结构图。
5Key Decisions重要的技术决策及其理由。每个决策 2–4 句话。
6Detailed Approach具体的实现细节。优先展示新颖或有风险的部分。
7Scope & Risks非目标、范围边界、风险、边界情况、未解决的决策。

"Resulting Shape"部分是计划中最常被遗漏的部分，而且是强制性的。它通过结构图、模块图或关系示意来展示最终状态，让审阅者一目了然地看到结果，而不必从一大堆 diff 中推断。

在 Detailed Approach 部分，指导原则是"权重优先于顺序，而非顺序优先于权重"。先展示最新颖、最 risky 或影响最重大的实现细节。常规的 wiring 和样板代码放在最后。这确保审阅者的注意力花在最重要的事情上。

第六部分

Pre-Review：多角色评审门

在用户审阅计划之前，可以进行一次可选的自动化 pre-review。这不是走形式——它是由 1–3 个专门审阅者角色进行的、选择性的、基于证据的评审，只有在计划中有明确的强匹配信号时才会触发。

可用的审阅者角色包括 architect、test engineer、software engineer、performance engineer、security engineer、API designer、frontend engineer、product reviewer 和 documentation engineer。每个角色都有定义的关注领域。一个角色只有在计划明确触及该角色的关注领域、并且该角色的视角很可能暴露出实质性问题时才会被选中。

每个审阅者阅读已获批准的 research 作为背景语境，然后从自己被分配的角色视角审阅计划。发现必须同时锚定在计划和研究两个文件中。没有证据的主张会被丢弃。重要性标准（materiality standard）很严格——一个发现必须切实影响实现可靠性、验证清晰度、架构安全性、兼容性信心或用户审批信心。

审阅者的输出被综合处理：重叠的问题去重、保留最强证据支持的表述、保留有意义的分歧。综合后的 notes 写入 pre-review artifact，控制权返回到正常的用户审阅流程。最终决定始终由用户做出。

第七部分

Todo List：执行契约

计划获批后，工作流会在 plan artifact 中追加一个 ## Todo List 部分。这不是粗略的任务分解——它是一份精确的执行契约，完整覆盖已获批的实现工作，按真实执行顺序和依赖关系拆分为客观上可完成的步骤。

每个 todo item 使用 checkbox markdown 格式，有清晰的主题和对计划中相应部分的引用。Todo list 不仅包含实现任务，还包含显式的验证任务：typecheck、lint、运行相关测试套件。列表后面跟着一个 ## Things to note 部分，记录实现阶段必须遵守的关键约束。

Todo list 必须被显式批准后才能开始实现。这防止了一种常见的失败模式：计划被批准了，但执行者对它的解释与计划者的意图不同。Checklist 使解释变得具体和可审阅。

第八部分

Implementation：忠诚度优先于创造力

Implementation 阶段执行已获批的 todo list，而不是重新定义它。这是前面所有 artifact 投资获得回报的阶段：有了经过验证的研究基础、已获批的计划以及具体的 todo items，执行者可以完全专注于正确实现，而无需临时做出架构决策。

一套清晰的变更策略（change policy）规定了当现实与计划出现偏差时该怎么做。小的 wiring 改动、imports、测试调整和次要的支持性编辑可以接受，无需重开早期阶段。但任何构成新 scope、未获批的行为变更、静默重构或机会主义重构的行为，都需要回退到最早的无效阶段。如果已批准的计划不再符合实际代码库，或者新发现的需求改变了 scope 或设计，工作流就会回退。

这一策略为深入的研究和规划创造了强大的激励。它也确保当意外出现时——它们一定会出现——这些意外被有意识地处理，而不是被静默吸收到一份越来越不准确的实现中。

第九部分

Verification：双向忠诚度审计

最后一个阶段是一个 fidelity audit（忠诚度审计），而不是功能测试。它回答一个问题：我们做了我们承诺要做的事，而且只做了我们承诺要做的事吗？审计同时从两个方向进行。

正向验证（Forward verification）捕捉 false positives。对于 todo list 中的每个 [x] 项，验证所声称的交付物是否确实存在、是否符合描述。任何标记为完成但实际上未完成的内容都会被标记。

反向验证（Reverse verification）捕捉 false negatives。它将实际的代码变更与 todo list 进行交叉核对。每个创建或修改的文件都被追溯到一个具体的 todo item。无法映射到任何 todo item 的变更会被作为"可能意料之外的"提出。微小的支撑代码——imports、脚手架、错误处理——被接受。任何改变了 todos 范围之外行为或功能的变更都会被标记为 unexpected change。

偏差按来源分类：implementation-level（工作已做但与 todo 不匹配）、plan-level（todo list 不充分）、research-level（计划正确但研究错误）。这种分类确保修复进入正确的阶段，而不是将错误累积在错误之上。

第十部分

Review-Revise 循环：持续对齐

该工作流的一个决定性特征是在前四个阶段内部运行的 review-revise 循环。每个阶段产出一个 artifact。用户审阅它并用标准格式标注意见（COMMENT: <text> [on: "<anchor text>"]）。Agent 全面分析每条意见，修订 artifact 以解决该意见，然后呈现更新版本供再次审阅。这一过程重复进行直到显式批准。

这个循环有严格的规则。意见不被视为批准。没有意见被忽略、静默丢弃或部分解决。Agent 不会机械地解决问题——每条意见都需要批判性思考，以维护 artifact 上下文的逻辑一致性。而且不能在没有用户明确批准的情况下退出循环。

这个看似缓慢的过程正是快速执行阶段成为可能的原因。在 review-revise 上投入的每一小时，都能节省未来三小时调试基于未经验证的假设构建的代码的时间。

核心规则

不可协商的规则

该工作流由一套适用于所有阶段的规则治理。这些不是指南——它们是界定该工作流为何物的约束。

规则适用阶段
进入下一阶段前必须获得显式批准Research, Planning, Todo List
将每个已批准的 artifact 视为下一阶段的合同Research, Planning, Todo List

除非用户明确要求，否则不读取无关文件所有阶段

问题必须聚焦于打磨真正的需求Task Clarify

前四个阶段和最后两个阶段之间的区别是刻意的。Clarify、Research、Planning 和 Todo List 关乎共享理解和对齐——它们受益于审视和迭代。Implementation 和 Verification 关乎执行一个已经达成一致的合同——它们受益于速度和专注。

总览

六阶段 Pipeline

每个阶段产出一个 artifact，作为下一阶段的契约。前四个阶段之间有 4 道显式审批门。

Clarify ➔ 审批 ➔ Research ➔ 审批 ➔ Plan ➔ 审批 ➔ Todo ➔ 审批 ➔ Implement ➔ Verify

前四个阶段内有 review-revise 循环，直到用户显式批准。Implementation 和 Verification 无需中途批准即可自动执行。

阶段 1

Task Clarify

通过结构化的对话，在调研开始之前将原始想法转化为清晰、明确、具体的共享理解。一次一个问题，优先选选择题。

目的 & 交付物

为什么做？输出应该是什么样的？怎么才算完成？

约束 & 边界

在什么范围内工作？什么与什么交互？我们不做什么？

阶段 2

Research

建立对代码库现实的经过验证的理解。委托-验证模式：1–3 个子 agent 并行调查，发现结果再与实际代码库交叉验证。

并行调查

按问题或子系统拆分。每个子 agent 获得详细的上下文和明确的约束。

交叉验证

所有发现视为未经验证。协调人解决冲突、填补缺口、直接读取代码验证关键主张。

源引用

每个非显而易见的发现包含 [src: path:line]，审阅者无需重新搜索即可验证。

阶段 3

Planning

渐进披露结构：审阅者可以在任意详细程度上停下来做出有依据的判断。7 个部分，每个部分只提供前一层无法传达的细节。

1. Title & Summary

2–3 句话。做什么和为什么。

2. Change Inventory

确切文件 + 每个变更的意图。

3. Top-Level Approach

策略。3–5 句话。

4. Resulting Shape

最终状态的结构图。

5. Key Decisions

权衡和理由。

6. Detailed Approach

从新颖/有风险的开始，而非常规内容。

7. Scope & Risks

非目标、风险、边界情况。

门控流程

多角色 Pre-Review

在用户审阅前，1–3 个专门审阅者角色评审计划——仅当计划中有该角色的强匹配信号时才触发。

证据锚定

每个发现必须同时引用计划和研究。没有证据的主张被丢弃。

重要性标准

只报告会实际影响可靠性、安全性、兼容性或审批信心的内容。不吹毛求疵。

综合而非汇总

重叠问题去重，保留最强证据支持的表述，保留有意义的分歧。

Todo List

追加到 plan artifact 中的执行契约。客观上可完成的步骤，按真实依赖关系排序，包含显式的验证任务。

## Things to note — 实现阶段必须遵守的约束

阶段 5

Implementation

执行已获批的 todo list，而不是重新定义它。前期 artifact 的投资让执行者可以完全专注于正确的代码，而非架构决策。

无需回退

小的 wiring、imports、测试调整、次要支持编辑。不影响 scope 或设计的日常变更。

需要回退

新 scope、未获批的行为变更、静默重构、机会主义重构。当计划不再匹配现实时，回到正确的阶段。

阶段 6

Verification

双向忠诚度审计：我们做了承诺要做的事，而且只做了承诺要做的事吗？

正向验证（Forward）

对每个 [x] 项：交付物真的存在吗？匹配描述吗？捕捉 false positives——标记为完成但实际未完成。

反向验证（Reverse）

将每个文件变更追溯到 todo item。超出 todos 的变更被提出为意料之外的。捕捉 false negatives。

偏差按来源分类：implementation mismatch、plan gap、research error——确保修复进入正确的阶段。

核心机制

Review-Revise 循环

前四个阶段中的每一个都通过结构化的迭代循环运行，直到显式批准。

撰写 Artifact ➔ 用户审阅 ➔ 修订 ➔ 批准?

批判性响应

每条意见在修订前都被全面分析。不做机械式的问题解决。

不静默丢弃

意见不会被忽略、部分处理，或被当作批准。

显式退出

没有用户的明确批准，不能退出循环。没有捷径。

总结

核心原则

审批门（Approval Gates）

没有用户对 artifact 的显式签字，阶段不能推进。每个 artifact 都是下一阶段的合同。

证据锚定（Evidence Grounding）

Research 来自代码库，而非假设。每个主张都必须通过源引用可验证。

渐进披露（Progressive Disclosure）

计划的结构使审阅者能在任意深度停下来并做出有依据的判断。

忠诚度优先（Fidelity Over Creativity）

Implementation 执行合同。现实偏离时，回退到正确的阶段——不要吸收偏差。

Clarify → Research → Plan → Todo → Implement → Verify。每个阶段、每道门、每个 artifact——都为一个目的服务：一次性交付正确的代码。
