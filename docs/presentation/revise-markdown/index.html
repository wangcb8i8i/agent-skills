文档修订方法论

Revise Markdown
结构化审阅-修订循环

一个专注的 review-revise loop：在 markdown artifact 中找到、分析、处理嵌入的评审意见，迭代直到用户显式批准。

1 个工作流 4 个步骤 10 张幻灯片 显式批准退出

Revise Markdown 是一个单一用途的 skill，专门实现 review-revise 循环。它不处理多阶段 pipeline，而是聚焦于一个精确的闭环：读取 artifact、发现所有嵌入的评审意见、理解每个意见在要求什么、修订文档以妥善处理每条意见、然后请用户再次审阅——重复直到用户说"批准"。

这是更大型工作流（如 Research-Plan-Driven）中 review-revise 循环的原子化版本。它解决了一个简单但棘手的问题：当文档中有评注意见时，如何确保每条意见都被认真对待，而不是被忽略或被机械地处理。

第一部分

为什么需要 Review-Revise 循环

直接编辑（one-shot edit）是最常见的文档修订方式，但它在面对多条评审意见时几乎总是失败。原因很直接：评论之间可能相互冲突，上下文可能在修订中丢失，某些意见可能被无意中忽略。更常见的情况是——修订者看到几条简单的评论就快速改完，但忽略了其中一条评论背后隐藏的深层问题。

Revise Markdown 将这个过程结构化为一个可重复的循环，而不是一次性的修改。它强制在修改之前进行分析，在提交之前进行综合评估，并且不允许在用户明确批准之前退出。

一次修改遗漏一条意见
当文档中有 5 条评论时，一次性修改几乎总会遗漏或部分处理其中一条。循环机制迫使每条评论都单独被处理。
评论间存在冲突
两位审阅者可能给出相反的建议。不先综合评估就直接修改，会导致改到一半发现前后矛盾。
模糊的评论需要澄清
"把这个改得更好"——更好的什么？没有回退机制时，修订者只能猜测。循环允许在不确定时请求澄清。
没有明确的"完成"标志
缺少显式批准机制时，修订可能在仍有未解决问题的状态下被认为"做完了"，留下隐患。
第二部分

工作流总览

Revise Markdown 的工作流是一个简单的循环，包含 4 个核心步骤：

读取 Artifact ➔ 发现 & 分析评论 ➔ 综合修订 ➔ 用户审阅 ➔ 批准?

如果用户审阅后有新的评论，循环回到"发现 & 分析"步骤。只有用户显式批准时循环才结束。

这个工作流的关键特征在于它不是一个线性序列——它是一个真正的循环。每次迭代都从重新读取 artifact 开始，这确保了修订者始终在处理最新版本，而不是基于过时的上下文进行操作。

第三部分

评论格式

评论必须遵循标准格式，这样才能被可靠地识别和处理：

COMMENT: 这里需要更明确的错误处理说明，特别是在网络超时的情况下。用户需要知道重试策略。
 [on: "Error handling section"]

评论可以出现在两个位置：

内联（Inline）
评论直接放在被评论内容的下方或旁边。这种方式让评论的上下文自然可见。
文档末尾的评论专区
在一个专门的 ## Review Comments 部分中列出评论，每条评论通过 on: 字段指向被评论的内容。

COMMENT: 是标准的 marker。但也接受其他明显的变体，如 REVIEW COMMENT:。关键是 marker 必须明确可识别——这是避免将普通文本误认为评论的要求。

on: 字段是可选的，但强烈推荐。它指向评论所针对的具体内容。在修订时，用这个字段来定位对应的部分，而不是依赖猜测。

第四部分

发现与分析

循环的每一步都从重新发现所有评论开始。这不是简单的 grep——它需要理解每条评论的内容和意图。

发现阶段检查两个地方：各个部分中的内联评论，以及文档末尾的 ## Review Comments 部分。遗漏任何一条评论都是不允许的。

分析阶段要求对每条评论进行深入理解，然后才制定修订方案：

理解被请求的变更
这条评论在要求什么？是内容修改、结构调整、措辞优化，还是补充遗漏的信息？
识别评论之间的相互作用
多条评论可能指向同一段落的不同方面。它们可能相互补充，也可能相互冲突。在修改之前需要综合评估所有这些关系。
评估评论的有效性
偶尔评论可能基于过时的信息或对文档的误解。在修订前需要判断是否需要澄清。

分析阶段是整个循环中最关键的步骤。跳过它直接修改，几乎必然导致部分处理或不一致的修订结果。

第五部分

综合修订

分析完成后，进入修订阶段。这里的核心原则是：每一条评论都被充分处理，没有评论被静默丢弃。

修订的基本流程：

  1. 定位—使用 on: 锚点或上下文找到评论所指的部分
  2. 修订—修改文档以解决评论中提出的问题
  3. 移除—从 artifact 中移除已解决的评论（内联和总结部分都移除）
  4. 验证—确认修订没有引入新的问题或与其他评论冲突

注意第三步：已解决的评论必须从文档中物理移除。这不仅仅是为了整洁——它确保在下一轮审阅中，已解决的问题不会再次被误认为未解决。一条评论只应该保留在文档中，当它仍然是一个开放问题时。

如果一条评论与其他评论冲突，需要解决冲突后再进行修订。如果一条评论不清晰，应该先请求澄清，而不是猜测意图进行修改。

第六部分

处理模糊评论

并非所有评论都是清晰明确的。当一条评论的意图模糊不清时，直接猜测并修改是错误的选择。正确的做法是显式地提出模糊性。

处理流程：

COMMENT: make this better [on: "Conclusion"]
→ 这条评论说"make this better"——我应该从哪个维度改进：
  清晰度、准确性、完整性还是简洁性？我倾向于认为这篇文章
  在完整性上有所欠缺，建议补充一个具体的案例。这样可以吗？

这种做法有三个好处：它让用户知道你注意到了评论；它提供了具体的改进方向供确认；它避免了在错误的方向上浪费时间。在评论被澄清之前，它保留在文档中，不会被视为已解决。

第七部分

约束与规则

这个 skill 由一套不可协商的约束来定义。这些规则不是建议——它们是工作流之所以有效的结构性保证。

规则理由
不将评论视为批准评论是反馈，不是 sign-off。用户在仍开放评论时说"看起来不错"不算批准。
不静默丢弃或部分处理评论每一条评论都必须被记录在案。部分处理等于没有处理。
不机械化处理——批判性思考每条评论评论可能相互作用、冲突或遗漏上下文。机械化的解决方式会遗漏这些细微之处。
没有显式批准不能退出循环"Looks good"但仍有未解决评论，不算批准。
评论不清晰时先请求澄清猜测浪费迭代次数。一次澄清可以节省多次错误修订。
解决后从 artifact 中移除评论块保持 artifact 整洁，使下一轮审阅可以专注于真正未解决的问题。

这些规则的共同主题是：拒绝走捷径。循环设计的原则是"慢在分析，快在确认"——在分析和修订上投入时间，但确保用户不需要反复审阅同样的问题。

第八部分

完成标准

循环只在满足以下两个条件时结束：

所有评审意见已被处理并从 artifact 中移除
文档中不应再有任何 COMMENT marker。已解决的评论已被吸收到正文中；未解决的评论如果仍然开放，则循环不能结束。
用户已显式批准 artifact
批准信号必须是显式的：用户说"批准"、"LGTM"、"proceed"，或类似的明确表述。默许、部分同意、或有条件的接受都不算批准。

这两个条件是"与"的关系——只满足其中一个是不够的。一个没有评论但用户没有批准的文档仍然是开放状态；一个被用户"批准"但还有未解决评论的文档实际上没有被批准。

核心原则

设计哲学

Revise Markdown 虽然简单，但它体现了一些重要的设计原则，这些原则也适用于更广泛的工作流：

评论是资产，不是负担

每条评论都代表一个改进机会。处理评论不是通关打怪，而是协作提升质量的体现。

分析先于行动

在看到完整的评论图景之前不要动手修改。单独处理每条评论会导致不一致的修订。

显式优于隐式

批准必须是显式的，模糊必须被指出，评论必须被物理移除。不留下模糊地带。

不能跳过步骤

循环的每一步都是强制性的。发现、分析、综合、修订、审阅——顺序不能乱，步骤不能省。

这个 skill 的价值不在于复杂——恰恰相反。它的价值在于简单和严谨：把一个被广泛实践但通常做得很随意的事情，变成一个可重复、可审计、可靠的过程。

方法

修订四步法

每一条评论都经过定位→修订→移除→验证的完整处理流程。
