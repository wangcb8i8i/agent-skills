# inspiration

**语言：所有产出物使用中文。**

## 工作方式

勘察 → 发散 → 汇报。任一阶段判定素材不足以产出灵感时，直接进入汇报如实告知。

## 牵引

用户提供的注意力方向——话题关键词、目标领域、追问角度。同一方向在勘察阶段决定看什么，在发散阶段决定往哪想。

用户可在对话中任意时刻给出牵引，也可中途切换。方向变化会改变勘察的重点——从新方向重新勘察后进入发散。

## 三个阶段

### 1. 勘察

深入理解上下文，筛出能追问的点。

**上下文判定**：
- 用户消息中直接提供了 → 轻量确认（一段话确认理解后继续）
- 来自文件/代码/recon → 完整勘察
- 无上下文 → 检查 `.recon/`，无则按牵引方向自行查阅

**筛选方法**。对每个潜在点问：
- 这里反直觉的是什么？
- 限制条件是什么？
- 如果拿掉这个会怎样？

只答得上"很有趣"的点过不了。

上下文不足时主动通过 web 搜索补充领域认知。搜索结果是原料，不是结论。

**完成标准**：每个通过筛选的点有明确来源 + 至少有 1 个可追问方向。

**不足时**：所有点筛不过 → 直接汇报。

### 2. 发散

对筛选过的点做系统化联想，产出并写下灵感。

**联想方向**（参考表，随用随查）：

| 方向 | 追问 |
|------|------|
| 提炼 | 这个做法可以抽象成什么模式？ |
| 迁移 | 这个模式搬到相邻领域能解决什么？ |
| 反转 | 如果约束反过来？他们刻意不做什么？ |
| 跃迁 | 完全跳出这个领域还能变成什么？ |
| 组合 | 与一个无关的东西强行杂交能长出新品类吗？ |
| 对抗 | 最好的反对理由是什么？哪个环节最脆弱？ |

不要求六个方向都覆盖。

每条灵感锚定在勘察上——能指出源头→联想→为什么有意义。

**完成标准（深度优先于条数）**：每条能说清"源头→想法→意义"，可追溯回上下文具体位置。只能产出浅层联想时，即使达到条数门槛也不退出——如实汇报。

### 3. 汇报

直接向用户呈现灵感，不走文件。

**约束**：
- 不出现内部术语（勘察/发散/对抗等）
- 不汇报过程（禁止"已完成勘察"等）
- 每条说清：想法、来源、延伸方向

**格式建议**：

```
简述，一句话点出核心想法

从哪来的：上下文中的什么触发了这个联想

展开——这个想法是什么、为什么有意思、跳跃链条

→ 延伸方向（可选）
```

骨架是"简述·源自·展开·延伸"，写法自然。不足时如实告知缺了什么、为什么不足以产生有意义的联想。

汇报完即结束，不追问。

## 分支处理

**重入**（同一上下文再次调用）：
- 上下文不变 → 跳过勘察，发散只走对抗，施压已有灵感
- 上下文变化 → 从勘察重新执行
- 不确定 → 重新勘察

**牵引切换**：从勘察重新进入，用新方向勘察。

## 失败模式

| 问题 | 防御 |
|------|------|
| 勘察不足就发散 | 勘察完成标准——每个点能指明可追问方向 |
| 发散变闲聊 | 勘察时收紧筛选——只放过真能追问出东西的点 |
| 只有表层联想 | 发散完成标准——必须说清"源头→想法→意义" |
| 素材贫瘠强行灵感 | 三步都有"不足如实告知"的出口 |
