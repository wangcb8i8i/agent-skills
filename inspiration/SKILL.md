# inspiration

**语言：所有产出物使用中文。**

## 工作方式

```
上下文（文档 / 代码 / recon / 对话 / 任意）
         │
         ▼
  ① 勘察 —— 捕捉值得注意的点
         │
         ▼
  ② 圈点 —— 筛出能撑起追问的点
         │
         ▼
  ③ 发散 —— 以牵引方向为锚做系统化联想
         │
         ▼
  ④ 捕捉 —— 把涌现的灵感显性化
         │
         ▼
  ⑤ 汇报 —— 向用户呈现灵感

  勘察／圈点／发散中任意阶段判定素材不足以产出灵感
         │
         ▼
  直接进入汇报，如实告知
```

## 牵引

用户提供的注意力方向。同一个方向在流水线不同阶段产生不同效果：

- **勘察阶段**：决定 Agent 在上下文里重点捕捉什么
- **发散阶段**：决定 Agent 往哪个领域联想

用户可能在对话中任意时刻给出牵引——可能是话题关键词、目标领域、或一个追问角度。

### 方向调整

产出灵感后，用户可能在相同上下文上切换牵引方向。从圈点阶段重新进入——用新方向重新圈点，发散，捕捉。

## 五个阶段

### 1. 勘察

深入理解用户提供的上下文。上下文可以是任意形态：文件路径、文档片段、代码、recon 产出、对话中已讨论过的事物。

用户未提供上下文时，检查当前工作目录下是否有 `.recon/`——有则自动作为上下文使用。没有则根据用户抛出的牵引方向，自行查阅项目文件、搜索相关代码、获取所需信息，建立理解。

上下文就绪后，判断勘察模式：
- 上下文来自用户对话（用户直接在消息中提供了材料，或本次是同一上下文的重入）→ **轻量确认**：用一段话总结"我理解的内容是……"，确认无误后直接进圈点
- 上下文来自外部读取（文件、代码、recon）→ **完整勘察**：按下方标准执行

勘察的目标不是穷尽所有细节，而是捕捉到**值得产生灵感的东西**：反直觉的设计取舍、独特的实现路径、被刻意简化的地方、一个做法背后的约束链条。

勘察过程中，如果当前上下文不足以识别有价值的张力点或牵引方向——主动通过 web 搜索等方式补充领域认知（趋势、争议、常见盲区等）。搜索结果是上下文原料，不是结论。搜索结果纳入勘察范围后，重新执行勘察流程。

**完成标准**：能用自己的话讲清上下文中值得注意的东西，每一项指向前述四类目标之一。不要求多少条，但每条必须是"这个点确实有点意思"。如果上下文本身太薄找不出任何值得注意的点——如实告知，不强行发散，进入汇报。

### 2. 圈点

从勘察的"值得注意的点"中圈出能撑起追问的。不是所有"有点意思"都有东西可追——有些只能停在表面描述，有些能引出深层联想。

对每个点问：**从这里出发，能追问出什么？** 如果唯一的回答是"这个设计很有趣"，它过不了圈点。

**完成标准**：每个被圈出的点撑得起追问——"从这里能追出什么？"有明确的回答。如果所有的点都撑不起追问——如实告知"素材不足以产生灵感"，进入汇报，不继续执行后续阶段。

**重入约束**：如果是同一上下文、同牵引方向的再次调用（用户对前一轮结果不满意，重试触发），圈点阶段识别前一轮已产出的灵感，只圈被遗漏的点。如所有值得注意的点都已覆盖 → 跳过圈点，发散阶段只走"对抗"方向，对前一轮灵感做施压检验，不再从头发散新灵感。

### 3. 发散

以用户牵引方向为锚，对圈点后保留的点做系统化发散。六个追问方向从两个轴上驱动探索：

| 方向 | 追问 | 探索轴 |
|------|------|-------|
| 提炼 | 这个做法可以抽象成什么模式？关键要素是什么？ | 认知距离：近——从具体到抽象 |
| 迁移 | 这个模式搬到相邻领域，能解决什么？ | 认知距离：中——跨领域平移 |
| 反转 | 如果约束反过来？他们刻意不做什么？这说明什么？ | 认知距离：中——从负面空间找启发 |
| 跃迁 | 完全跳出这个领域，这个思路还能变成什么？ | 认知距离：远——到达新品类 |
| 组合 | 把这个思路和一个完全无关的东西强行杂交，能长出新品类吗？ | 交叉——双源碰撞 |
| 对抗 | 如果反对这个灵感，最好的反对理由是什么？哪个环节最脆弱？ | 验证——给灵感施压 |

不要求六个方向都覆盖。

**有牵引时**：以用户指定的方向为主，其他方向为辅。

**无牵引时**，判断圈点产出是否包含张力点：
- 圈点产出以张力点和争议点为主（常见于勘察经过搜索补充的场景）→ 按 **张力驱动策略** 执行
  1. 发散以"对抗"方向为主，对每个张力点/争议点追问："最常见的做法是什么？与之相反的做法是什么？主流方案忽略了什么？"
  2. 对抗完成后，选出最具张力的 1-2 条做跃迁或组合
  3. 如产出达到 2-3 条灵感候选，结束张力驱动，进入捕捉
- 圈点产出以常规注意点为主 → 按 **广度优先策略** 执行
  1. 对所有圈出的点，每个只做 1 个最自然的联想（提炼或迁移，任选一个方向）
  2. 从中选出最有潜力的 1-2 条联想，深入做反转/跃迁/组合（不覆盖所有方向）
  3. 如产出达到 3 条灵感候选，结束广度优先，进入捕捉

每一步跳跃都锚定在勘察上——能指出灵感和源头的具体关联。

**完成标准**：每个被发散的方向都能说清"源头是什么 → 联想到了什么 → 为什么有意义"。空泛的"这个可以做 X"不算完成。素材确实不足以支撑任何有意义的发散时，如实报告，不编造，进入汇报。

### 4. 捕捉

把发散中涌现的灵感显性化。写之前确认两件事：核心想法能一句话说清、能追溯回上下文的具体位置。写完之后，如果有自然延伸的探索方向，顺手标注，但不强求。

**完成标准**：灵感文本已就绪，每条有清晰的"来源→想法"链条。

### 5. 汇报

把捕捉到的灵感直接向用户呈现，不走文件。向用户说明想法是什么、从哪里来、可以往哪走。不逐字照念文字、不提及工作文件。发现式汇报——汇报灵感本身，不汇报过程。

**禁止**："已完成勘察""已发散 3 个方向""已捕捉 2 条灵感"——汇报过程而非发现。

**语言约束**：汇报使用用户自然语言，不出现内部流程术语（勘察／圈点／发散／捕捉／牵引／对抗等）。把"这是跃迁方向想到的"转译为"这个思路完全跳出原领域后联想到"。

**建议格式**：每条灵感的结构相对固定，但语言流动——不填空、不模板化。

```
简述，一句话点出核心想法

从哪来的：上下文中的什么触发了这个联想

展开——这个想法是什么、为什么有意思、跳跃链条

→ 延伸方向（可选）
```

骨架是"简述·源自·展开·延伸"，写法自然就好。"从哪来的"在正文里可以是"注意到……""这让我想到……"等自然表达。不要求每行前写标题。

**完成标准**：
- 灵感有产出时：每条灵感已清晰呈现，说清来源→想法链条
- 素材不足以支撑灵感时：如实告知缺了什么、为什么不足以产生有意义的联想

汇报完即结束，不追问"要不要继续深挖"。如果某个灵感确实有值得继续的方向，在捕捉时标注"还可挖"，由用户决定是否追问。

## 失败模式

| 问题 | 症状 |
|------|------|
| 勘察不足就发散 | 灵感浮于表面，与上下文脱节。防御：勘察阶段完成标准——必须能说清值得产生灵感的东西，而非复述事实 |
| 发散变闲聊 | 联想越跑越远，失去与源头的锚定。防御：每条灵感必须能追溯到上下文中的具体依据 |
| 只有表层联想 | "这个可以做 X"但说不清 X 和源头的具体关联。防御：发散阶段完成标准——必须说清"源头→想法→为什么有意义" |
| 素材贫瘠强行灵感 | 上下文本身不够丰富，强行产出"灵感"。防御：勘察、圈点、发散三步都有"不足则如实告知"的出口 |
