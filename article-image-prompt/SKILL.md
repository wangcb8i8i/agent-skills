# 文章配图 Prompt 生成

一个**视觉翻译器** — 读取文章（文件/URL/粘贴/上下文），产出它需要的配图，不需要用户做自己不会的设计决策。

**翻译器** 是核心概念：忠实读取原文，找到需要翻译的部分，用另一种语言（图像）交付。用户不需要告诉它*怎么*翻译 — 只需要确认目录。

---

## 阶段总览

| # | 阶段 | 做什么 | 用户操作 |
|---|------|-------|---------|
| 1 | **INGEST** | 读取原文 | 无 |
| 2 | **AUDIT** | 扫描配图候选 | 无 |
| 3 | **ALIGN** | 确认目录：TRIAGE 选方案 → STYLIZE 定风格 → REFINE 定构图 | 选择 + 确认 |
| 4 | **TRANSLATE** | 执行翻译：生成 Prompt | 无 |
| 5 | **RENDER** | 交付译本 | 无 |

---

### 1. INGEST — 读取文章

从文件路径、URL、粘贴文本或对话上下文中获取文章。识别格式（markdown/HTML/纯文本/PDF）。

**完成标准：** 全文已在上下文中，并标注了来源格式。

---

### 2. AUDIT — 扫描配图候选

对全文进行**松扫描** — 标记所有值得配图的内容：

- **抽象概念**（一致性、背压、收敛）
- **系统/架构**（分层、组件、服务）
- **流程**（阶段、状态转换、决策分支）
- **对比或权衡**（vs.、与…相反、对比）
- **量化论断**（X% 的、N×、增长曲线）
- **空间/因果关系**（导致、依赖于、位于…内）
- **数据结构**（树、环、队列、缓冲区）
- **文章核心论点**（整篇文章的主题 — 封面候选）

每个候选按**理解增益**评估——这张配图对读者理解文章有多大帮助：

| 等级 | 标签 | 含义 |
|------|------|------|
| A | 高增益 · 强烈推荐 | 没有这张配图读者会错过文章的关键论点，或难以理解核心机制 |
| B | 中增益 · 可接受 | 配图能让读者更快理解，但不配图也不影响理解正确性 |
| C | 低增益 · 存疑 | 强行配图的效果不如让读者直接读文字 |

**无最低频次门槛：** 一个概念只出现一次但承载了文章论点，比出现十次但无关紧要的概念更值得配图。

排序后，排名最高的核心论点候选标记为**封面候选**。接下来的 N 个（默认：3 个）为**内联候选**。根据文章长度重载 N：<500 字 → N=1，500-3000 → N=3，>3000 → N=5。

输出中文候选列表，包含理解增益等级和一句推荐理由。候选名称使用文章原词（中文）。

**完成标准：** 已生成并展示排序后的候选列表。

---

### 3. ALIGN — 确认目录

ALIGN 确认"翻译目录"：先选候选、再定视觉风格取向、最后定构图细节。拆为三步 — **TRIAGE**（选配图方案）、**STYLIZE**（定视觉风格取向）、**REFINE**（定每张图构图与细节）。

---

#### TRIAGE — 挑选配图方案

展示 AUDIT 输出的全部候选项。根据评分和文章结构推断 2–3 个**组合包**作为选项，选项组合规则：

| 选项 | 内容 | 推断规则 |
|------|------|---------|
| 推荐方案 | 封面 + 若干内联候选 | 候选间构成连贯叙事（流程/对比/因果链）。按清晰度×叙事权重排序后，优先纳-入不同叙事维度的候选，避免同一维度堆叠多张 |
| 精简方案 | 封面 + 置信度最高的 1 个内联候选 | 文章核心论点 + 最强论据 |
| 完整方案 | 所有候选 | AUDIT 输出的全部 |
| 其他 | 用户自由描述 | 自然语言说明要哪些图——具体候选、文章段落、或概念描述 |

向用户提供 3 个预设方案选项 + 自由输入让用户选择，**等待用户响应**后再进入下一阶段。按预设选项的内容组织方案描述，不额外输出候选列表。

**错误处理：**
- 用户说"候选全不对"或"再扫一次" → 回 AUDIT 重新扫描全文
- 用户要求增删某张图 → 接受修改后直接进入下一阶段

**完成标准：** 用户已选择方案或描述了需要的配图集。不允许空集。

---

#### STYLIZE — 定视觉风格取向

用户在进入构图细节前，先确定整个配图批次的**视觉风格取向**。风格决定了选哪套色板、渲染氛围和细节密度基调。

根据文章类型推荐最适合的 1 种风格（默认选中），同时提供全部 4 套供用户选择：

| 风格 | 色板标识 | 氛围 |
|------|---------|------|
| tech-handdrawn · 技术手绘风 | `#3D7EA6` 主色 | 严谨·手绘·白板感 |
| analysis-contrast · 高对比分析风 | `#FF3D00` 主色 | 戏剧化·高张力·分析感 |
| grayscale-structural · 灰度结构风 | `#4A5568` 主色 | 克制·结构·工程感 |
| blog-handdrawn · 手记博客风 | `#7986CB` 主色 | 温暖·有机·手写感 |

推荐与文章类型最相关的 4 种风格让用户选择（默认选中推荐方案对应的风格），用户也可自由描述。**等待用户响应**后再进入下一阶段。

每个选项需包含：**风格名** + **一句话色调说明**（深底色/浅底色 + 主色）+ **一句话视觉参考**（像什么场景）。不需要 hex 值、不需要风格 slug、不需要抽象氛围词。

确定风格后，本次批次的全部插图统一使用该色板体系（单图可在 REFINE adjust 时单独调整）。

**完成标准：** 风格已选定，色板已确定。

---

#### REFINE — 确定每张图的构图与细节

对 TRIAGE 保留的每张候选图，展示推荐**构图**和**图中文字策略**。风格已在 STYLIZE 阶段全局确定。

每张图的推荐方案包含：
- 插图类型（概念示意图/架构图/流程图/信息图/装饰配图）
- 构图描述 — 内容锚定的自然语言说明，括号注图形结构
- 图中文字策略

```text
🖼 封面: "两阶段提交共识"
   类型: 概念示意图
   构图: 协调器和参与者分屏对比，绿/红状态指示（中心连接线）
   文字: 尽量无字 · 位于 § 开头

📄 1. "投票阶段"
   类型: 流程图
   构图: 投票请求→授予→提交 三步流程（从左到右泳道布局）
   文字: 仅标签 · 位于 § "准备阶段" 后
📄 2. "崩溃恢复"
   类型: 时间线图
   构图: 失败检测→切换→集群恢复（T→B 时间轴，两个分支状态）
   文字: 仅标签 · 位于 § "故障场景" 后
```

展示方案后，等待用户响应：

- 如果用户表示接受（"可以""ok""挺好的""没问题""就这样"等明显肯定语） → 直接进入 TRANSLATE 阶段生成 Prompt
- 如果用户提出任何修改请求（含接受词但带修改也可） → 进入 adjust 循环

**错误处理：** 用户说"等一下"/"等等"说要改 → 返回 REFINE 重新展示当前方案供调整。

##### adjust 循环

调整 → 展示新方案 → 等待用户响应：

- 表示接受 → 退出 adjust，进入 TRANSLATE
- 继续描述修改 → 继续 adjust 循环

用户无需学习特定命令词——所有交互用自然语言完成。技能根据用户的修改描述自行映射到对应调整项（替换单图色板、修改构图方向、调整文字策略、更换插图类型等）。


**完成标准：** 用户已表示接受，或 adjust 循环中已确认。每张保留候选的构图和文字策略已确定。

---

### 4. TRANSLATE — 执行翻译

把目录确认后的方案"翻译"为图片模型能理解的语言。每张已确认的插图生成两项内容：

1. **Visual Thesis** — 一句话说明读者只看到这张图（不读文章）时应该理解什么。必须包含可验证的主张，不只是话题标签（用中文）：
   - ✓ "读者应该理解 **为什么社交 Feed 选用 AP 优于 CP**" — 包含主张 + 立场
   - ✗ "读者应该理解 **AP 和 CP 的区别**" — 只有话题，没有主张

2. **Prompt** — 图片生成 Prompt，首句锚定在 Visual Thesis 上。

Visual Thesis 先写，锚定 Prompt，包含在输出文件的 Prompt 之前。这样 `.prompt` 文件里有一条可追溯的意图声明 — 可以用它来检验生成的图片是否符合预期。

**语言策略：** 在 `.prompt` 文件中，只有 `## Prompt` 段是英文（实际的图片生成文本）。其他所有段使用中文 — 用户通过中文来理解和验证意图。

**输出文件格式：** 单个 `{article-dir}/illustration-prompt/{slug}.prompt` 文件，所有插图 Prompt 写在此文件内，用 `# <diagram-slug>` 分隔。`slug` 从文章主题推断（kebab-case）。如果文件已存在，追加 `-2`、`-3` 等。输入为文件路径时，`article-dir` 为该文件所在目录；输入为 URL 或粘贴文本时，`article-dir` 为当前工作目录。如目录不存在则创建。

**单文件内每张插图的段结构** — 完整字段列表见 Reference → Prompt spec：

```markdown
# <diagram-slug>

## Visual Thesis
[一句话：含可验证主张，不只是话题标签]

## Context
[位置、插图类型、目标模型、图中文字策略]

## Style Anchor
[色板、渲染风格、氛围、细节密度]

## Prompt
[English — narrative declaration + Visual Thesis anchoring + colors + layout + exclusions]
```

> 不同配图的Prompt spec 之间 使用 `---` 分隔

参考 `references/color-science.md` 获取色板 hex 值、`references/metaphor-guide.md` 获取概念→视觉隐喻映射、`references/type-strategies.md` 获取各类文章的措辞模式。

**完成标准：** 每张已确认插图都有一个符合规范和质检要求的 Prompt。文件结构已校验。

---

### 5. RENDER — 交付译本

保存 `.prompt` 文件（TRANSLATE 阶段已逐步生成）。向用户报告文件路径。

**完成标准：** 已报告文件路径。

---

## Reference

### Prompt spec

```markdown
# <diagram-slug>

## Visual Thesis

[一句话：读者从这张图应该理解什么。必须包含可验证的主张，不只是话题标签。]

## Context

- 位置: [所在章节 / 段落]
- 插图类型: [cover / diagram / flowchart / infographic / decorative]
- 目标模型: [Generic / GPT Image / Midjourney / Flux / SDXL / Ideogram]
- 图中文字: [none / labels only / numbers only / exact text / minimal]
- 文字语言: 中文（默认）

## Style Anchor

- 色板: <slug>
- 渲染风格: [vector flat / gradient / hand-drawn / realistic / etc.]
- 氛围: [2-4 个氛围关键词]
- 细节密度: [rich / moderate / minimal]

## Prompt

[English only — 按 构图七层 逐层编写：叙事声明 → 画布 → 约束 → 实体 → 关系 → 视线引导 → 语义绑定]
```

### 构图七层

1. **叙事声明** — 媒介/风格 + Visual Thesis 锚定。首句必须回答"这张图表达了什么主张？"，而不是"这张图展示了什么主题？"（例如：`A split-screen comparison: left side shows user waiting with loading spinner (CP — consistent but slow), right side shows smooth scrolling with stale-data badge (AP — fast but eventually consistent). The AP side is visually favored — brighter, larger, warmer.`）
2. **画布** — 背景色（hex）+ 宽高比
3. **约束** — 排除项，以 `No...` 语句表达（至少 2 条）
4. **实体** — 每个实体一句话：位置、形状、`#hex`、大小/状态
5. **关系** — 连接类型、方向、端点
6. **视线引导** — 阅读顺序指导（`Reading order: start at..., then...`）
7. **语义绑定（可选）** — `representing`/`indicating`/`depicting` 写在结尾单独一段，决不可嵌入实体或关系句中

### 核心规则

- **Visual Thesis 先行**：每张图必须先写 `## Visual Thesis`。主张必须包含可验证论断 — 不只是话题标签（"AP 和 CP 的权衡" ✗ → "为什么社交 Feed 更适合 AP" ✓）。
- **叙事锚定**：Prompt 首句必须与 Visual Thesis 对齐。如果 thesis 说"AP 更优"，首句必须在视觉上表达这种偏好（更亮/更大/更暖），而非保持中立。
- **Hex 色彩精准**：所有颜色值使用所选色板的精确 hex。不允许同义词替换（`#FF6B35` 就是 `#FF6B35`，不是 `amber` 或 `orange`）。
- **原文依据**：每个视觉元素必须追溯到原文。三层溯源：原文依据（事实）→ 合理推导 → 视觉转化方案。
- **视线引导**：必须明确引导视线方向。不允许省略。
- **叙事焦点**：视觉上最突出的元素必须是核心概念的"主角"。不允许"核心元素放角落、装饰居中"。
- **图中文字**：默认为中文，按每张图已确认的策略执行。
- **Block representing**：`representing`/`indicating`/`depicting` 只写在尾部语义绑定段。不可嵌入实体或关系句。
- **中英分界**：在 `.prompt` 文件中，`Visual Thesis`、`Context`、`Style Anchor` 等元数据段使用中文。只有 `## Prompt` 正文是英文（发给图片模型的生成文本）。

### 质检清单

交付前检查：

- [ ] **Thesis 检查**：每张图都有 `## Visual Thesis`，且包含可验证主张
- [ ] **叙事对齐**：Prompt 首句与 Visual Thesis 一致（如 thesis 认为"AP 更优"，首句不保持中立）
- [ ] 构图注中每个实体都出现在 Prompt 中
- [ ] 所有 hex 值来自所选色板（无同义词替换）
- [ ] 视线引导已明确指定
- [ ] 约束已应用（"禁止添加"等排除项）
- [ ] 视觉元素追溯到原文
- [ ] `representing`/`indicating` 只出现在尾部，不嵌入中间
- [ ] 叙事焦点对齐（最突出元素 = 核心概念）
- [ ] Prompt 为一个连贯段落，非列表
- [ ] 图中文字遵循已确认的策略和语言（默认中文）
- [ ] 跨图一致性：同一概念在所有图中使用相同的视觉描述
- [ ] **语言检查**：`## Visual Thesis`、`## Context`、`## Style Anchor` 使用中文；仅 `## Prompt` 正文为英文
