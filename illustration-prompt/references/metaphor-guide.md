# 概念→视觉隐喻指南

抽象概念必须翻译为图像生成模型可理解的视觉元素。本文件提供“概念 -> 候选隐喻”的参考映射，并说明这些隐喻更适合什么图类型、有哪些 grounding 风险、何时不该使用。

本文件是参考层，不替代 `Source Grounding Ledger`、`Visual Glossary`、`Target Model Anchor` 或 `On-image Text Policy`。

## 使用顺序

选择视觉隐喻时，按以下顺序判断：

1. 优先使用原文已经提供的比喻或类比。
2. 若原文没有自带比喻，再从本表中选择最贴合领域语义的隐喻。
3. 结合配图类型筛选：概念示意图、架构图、流程图、信息图、装饰配图适用的隐喻不同。
4. 最后检查是否符合 `Source Grounding Ledger`：
   - 是否只是 `Visual metaphor`
   - 是否覆盖或扭曲了 `Direct facts`
   - 是否违反 `Must not invent`

## 使用规则

- 隐喻只用于帮助图像模型理解抽象概念，不等于原文事实本身。
- 结构关系、数量、方向、组件边界等信息优先来自 `Direct facts`，不能被隐喻替代。
- 当同一概念跨图反复出现时，应进入 `Visual Glossary`，避免每张图都换一个隐喻。
- 当图中文字策略为 `无文字` 或 `尽量无字` 时，优先选择一眼可读的形状型隐喻。
- 当目标模型排字能力较弱时，不要依赖“写在图上的词”来解释隐喻。
- 如果某个隐喻会引入错误时代感、错误行业语义、错误物理结构，就不要使用。
- 如果原文有自带比喻，如 `think of it as a highway`，优先使用原文比喻，不强行覆盖为表中隐喻。
- 如果没有合适隐喻，宁可直接使用结构化示意，也不要硬套隐喻。

## 概念→视觉隐喻矩阵

| 抽象概念 | 候选隐喻 | 更适合的图类型 | Grounding 风险 | 图中文字建议 | 避免事项 |
|---------|---------|---------------|----------------|-------------|---------|
| 分布式 / 一致性 | 互连节点、投票箱、绿色确认标识、环形拓扑 | 概念示意图、流程图 | 容易把逻辑协议误画成物理设备 | 无文字 / 仅标签 | 不要凭空加入机房、机柜、真实服务器品牌 |
| 分区 / 隔离 | 玻璃屏障、虚线隔离框、不同颜色区域、水区 | 架构图、概念示意图 | 容易把逻辑边界误读成物理墙体 | 无文字 / 仅标签 | 不要把“隔离”画成完全断连，除非原文明确如此 |
| 加密 / 安全 | 半透明盾牌层、锁/钥匙符号、二进制流、发光轮廓 | 概念示意图、装饰配图 | 容易落入陈词滥调，掩盖真实机制 | 尽量无字 | 不要只剩锁头而没有真实对象关系 |
| 延迟 / 性能 | 沙漏、秒表、速度线、赛车道、流量密度渐变 | 信息图、概念示意图 | 容易过度戏剧化，丢失可比较性 | 仅数字 / 仅标签 | 不要伪造精确耗时数据 |
| 数据流 / 管道 | 发光粒子管、河流分支、管道截面、波浪带 | 流程图、概念示意图 | 容易把方向和阶段画模糊 | 无文字 / 仅标签 | 不要只画“流动感”而没有起点终点 |
| 微服务 / 模块化 | 船运集装箱、乐高块、独立舱室、连接桥 | 架构图、概念示意图 | 容易把“模块”误画成物理容器系统 | 仅标签 | 不要让隐喻压过真实层级和边界 |
| 可观测性 / 监控 | 仪表盘、波形图、雷达图、信号灯 | 信息图、装饰配图、概念示意图 | 容易误导为真实 UI 截图 | 仅数字 / 尽量无字 | 不要捏造具体告警指标或产品界面 |
| CI/CD / 部署 | 流水线、传送带、绿灯/红灯状态、登台区 | 流程图、概念示意图 | 容易把流程简化得过头，遗漏关键阶段 | 仅标签 | 不要凭空加入未提到的审批、环境或工具 |
| 缓存 / 存储 | 堆叠层、蜂巢格、书架、存储单元阵列 | 架构图、概念示意图 | 容易混淆缓存和持久化存储 | 仅标签 | 不要把抽象缓存误画成硬盘阵列，除非原文明确 |
| 机器学习 / 训练 | 神经网络节点、数据集方块、训练→推理路径 | 概念示意图、流程图 | 容易引入过时“脑神经”刻板视觉 | 仅标签 / 仅数字 | 不要无依据加入机器人、人脸、真实照片素材 |

各图类型对隐喻的具体约束和写法细节见 `references/type-strategies.md`。

## 反模式

- 把隐喻当成事实画出来，导致结构失真。
- 一张图同时混用多个不相容的隐喻系统。
- 只因为某个隐喻“常见”就套用，而不看原文语境。
- 用隐喻替代数量、边界、方向、顺序等应来自 `Direct facts` 的信息。
- 用需要大量图中文字解释的隐喻，却没有匹配的文字策略或模型能力支持。
