这技能存在的唯一理由是：产出足够**完整、深度、可信**的研究报告——好到用户能据此做决策。

所有内部设计决策，当两条原则冲突时，由这条裁决：哪条让报告更**完整**、**深度**或**可信**？这三个质量支柱决定了每一条规则的存在理由。

### 完整 — 知识空间被充分探索

锚定点定义边界，递归饱和确认不再有新发现。搜索覆盖每个锚定维度；分析覆盖每个锚定维度；报告不遗留信息缺口。

- Scope 阶段确认后，自动进入研究执行，不再询问用户确认
- 研究执行期间，每完成一个内部阶段给出 1 句话进度更新（非中断式）

### 深度 — 穿透到结构性约束

因果链追问贯穿始终——不只是描述事实，而是挖掘事实背后的驱动因素。不是「X 在增长」，而是「因为 Y 所以 X 在增长；Y 又受 Z 驱动」。追问到触及结构性约束（资源/政策/技术瓶颈/激励结构/人口/地理/法律框架）时停止——那就是根因。

输出语言与用户输入语言一致（中文输入→中文输出）。

### 可信 — 可追溯、已检验、有边界

每份材料经过筛选和可信度评估，每条发现标注置信度，每条建议标注反转条件。/CHALLENGE 对顶部主张做压力测试。当事实不足以支撑结论时，承认不可知。

- 所有事实性主张必须标注来源。无法确认的明确标注「假设」或「待验证」
- 如果某维度的置信度不足以支撑决策且无法补搜 → 输出「无法形成可行动结论」并列出什么信息能翻盘，不凑一份报告

## Phase 1: Scope — 定边界

**目标：** 把模糊的研究兴趣转化为有明确边界的任务。**完成条件：** Scope 确认卡片已通过用户确认。

这是唯一需要用户深度参与的阶段。此阶段确定的锚定点和研究类型将约束后续所有步骤。

### Step 1: 问题澄清

用户的输入就是起点，不强行追问。根据输入的具体程度做适应性引导，核心是搞清「想知道什么」和「从什么视角」。

- 如果用户输入是短语，引导明确：
  - 想问的具体是什么？（市场格局？投资机会？技术成熟度？）
  - 从哪个视角？（创业者/投资人/企业采购/研究者？）
  - 这个研究最终要支持什么决策？（**选型决策** A vs B / **判断决策** 做不做 / **理解决策** 了解全景 / **监控决策** 关注什么信号）
- 如果用户输入已经具体，确认即可，不强行追问
- 识别**研究类型**（投资判断/技术趋势/产品决策/行业调研/生活经验/知识储备），后续流程据此自适应

### Step 2: 锚定点确认

**锚定点**是用户约束研究方向的唯一机制——中间不再有机会调整。锚定点决定了「完整」的边界。

- 确认用户的锚定点——有什么具体要求、关注角度、排除项？
  - "重点关注中美对比"、"只考虑 2025 年以后的数据"、"不看以太坊生态"
- 锚定点不足时，引导补充边界：时间范围、地域范围、排除项
- 确认决策约束条件：什么时间窗口内需要做这个决策？做错和被误导相比，哪种代价更高？

### Step 3: 已有知识注入（可选）

用户已有的信息是研究的跳板——不要重复已知，聚焦补充未知。

- 用户可提供已掌握的信息或材料——跳过不影响流程
- 如用户提供了原始材料，在常规研究之前首先扫描这些材料，定位相关段落作为背景锚点

### Step 4: 格式选择

产出格式在 Scope 阶段决定，统一输出节奏。

- **Full Report** —— 完整报告（默认）
- **Executive Brief** —— 1 页简报（核心结论 + 建议）
- **Slide Deck** —— 逐页要点（Mermaid 提纲）
- **Mind Map** —— Mermaid 思维导图
- **Decision Memo** —— 1 页，含推荐结论 / 关键证据（3条） / 关键不确定性 / 什么新信息会反转本推荐

格式影响研究侧重：Decision Memo → 优先确认 3 条关键证据的置信度；Slide Deck → 优先确保 5-7 个核心 claim 个个站得住；Executive Brief → 深度优先于广度，聚焦 1 个核心判断。

### Step 5: Scope 确认

输出研究任务确认卡片，用户确认后进入 Auto-Research：

```
研究问题: {一句话核心问题}
研究类型: {投资判断/技术趋势/产品决策/行业调研/生活经验/知识储备}
决策类型: {选型/判断/理解/监控}
锚定点: {用户指定的约束条件}
范围: 时间={} / 地域={} / 排除={}
格式: {Full Report / Executive Brief / Slide Deck / Mind Map}
已有材料: {有，共 N 份 / 无}
```

**Scope 质量门：**
- [ ] 核心问题可操作——不是兴趣短语，而是可回答的问题
- [ ] **锚定点**明确——知道查什么，也知道不查什么
- [ ] 研究类型已识别，分析框架可确定
- [ ] 决策类型已识别
- [ ] 用户知道得到答案后要做什么

## Phase 2: Research — 全自动研究

**目标：** 基于 Scope 确认的锚定点，全自动完成搜索、筛选、分析、综合、饱和——不需要用户介入。**完成条件：** 递归饱和关闭（C1+C2+C3 满足，或 C4 熔断），进入 Phase 3。

### 1. 搜索 — 覆盖锚定维度

- 生成多组搜索关键词，确保每个锚定维度至少被一组关键词覆盖
- 信源多元化：行业报告 + 一手信息 + 新闻报道 + 社区讨论 + 学术/技术文献，至少覆盖 3 类
- 去重后保留核心材料（至少 5 篇以上优质材料），每份生成 passport（标题/来源/日期/标签/摘要/可信度初评）
- 主动检查信息缺口——对照锚定点的每个维度，如果某个关键角度没有覆盖到，补搜

### 2. 筛选 — 评估材料可信度

对每份材料做二维评分，锚定点作为加权条件：

- **相关度**：★★★ 直接相关 / ★★ 间接相关 / ★ 弱相关
- **决策影响力**：◉ 会改变推荐 / ◎ 支撑现有推荐 / ○ 不改变推荐
- 双高材料（★★★ + ◉）优先深度阅读。高相关低影响力 → 摘要即可
- 对 ★★★ 和 ★★ 材料做四维可信度评估，见 [references/FALLBACK.md](references/FALLBACK.md)
- 标注来源间的矛盾点，留到分析阶段处理

### 3. 分析 — 穿透到根因

研究类型决定了分析框架的自然结构。分析视角参考 [references/LENS.md](references/LENS.md)。

- 先想清楚：这个问题的本质是什么？需要从哪几个角度才能把它说透？
- 根据研究类型确定最小分析角度数：
  - 投资判断 / 技术趋势 → 至少 3 个角度
  - 行业调研 → 至少 4 个角度
  - 产品决策 / 生活经验 → 至少 2 个角度
  - 知识储备 → 至少 3 个角度
- **锚定点**的每个维度必须至少被一个分析角度覆盖，而非仅作为分析边界
- **因果链**追问贯穿——对每个关键发现递归追问 Why，直到触及**结构性约束**（资源/政策/技术瓶颈/激励结构/人口/地理/法律框架）。区分**近因**和**根因**——当回答触及结构性约束时停止，那就是根因

### 4. 综合 — 提炼可信结论

- 汇总所有分析角度的核心发现，每条标注来源和**置信度**
- **交叉验证**：多来源确认 → 升级置信度；单来源 → 标注待验证；矛盾 → 分析原因并裁决或保留双方立场
- 按「置信度 × 对决策的影响力」排序所有发现，**锚定点**指向的维度给予更高权重
- 生成可执行建议（3-8 条），每条包含：行动 + 预期效果 + 主要风险 + 前置条件
- 每条建议标注**反转条件**——什么新发现会让本推荐不成立
- 每条建议标注回应了哪个**锚定**约束

### 5. 降级策略

信息不足时，按 [references/FALLBACK.md](references/FALLBACK.md) 的降级策略处理。如果核心决策维度降级后置信度仍 < 50% → 执行决策门。

### 6. 递归饱和 — 确认完整

综合完成后执行**逐层饱和**：覆盖度评估 → 不满足关闭标准 → 补搜 → 增量合并 → 重新综合 → 再评估。直到饱和（关闭标准和补搜策略见 [references/SATURATION.md](references/SATURATION.md)）。

## Phase 3: Report — 输出

**目标：** 产出可交付的研究报告——完整呈现发现，可信地支撑结论。**完成条件：** 报告文件已写入。

### Step 0: 入口检查

进入 Report 前，重新检查所有核心决策维度的置信度。如果任一维度 < 50% → 执行决策门（见 [references/FALLBACK.md](references/FALLBACK.md)）。

### Step 1: 确定叙事主线

报告结构由核心问题的本质决定。根据 Scope 阶段确定的研究类型、用户角色、问题粒度、决策类型，路由到对应路径：

```
研究类型 = {type} → {type} 主线路径（见 references/STRUCTURE.md）
用户角色 = {role} → {role} 视角前置（见 references/STRUCTURE.md）
问题粒度 = {granularity} → 展开度（见 references/STRUCTURE.md）
决策类型 = {decision_type} → 收尾方式（见 references/STRUCTURE.md）
```

叙事主线记录为一句话——这份报告要讲一个什么故事、按什么顺序展开。

### Step 2: 撰写报告章节

叙事递进通用模式：背景/现状 → 核心发现 → 深层分析 → 含义/行动。以下章节根据叙事结构选择组合。

**决策摘要**（必须）
- 推荐结论（1 句）
- 关键证据（3 条以内，每条标注置信度和来源）
- 如果只能记得一件事——the one thing that should influence the decision
- 主要**反转条件**——什么新信息会推翻推荐

**正文**（必须）
- 正文按 Step 1 确定的叙事主线展开
- 每个主题节标题以读者最关心的维度命名（如「竞争格局」「技术成熟度评估」「情景分析」），而非分析方法名称
- 每个关键结论呈现**因果链**——不只说「X 在增长」，还要说「因为 Y，所以 X 在增长；而 Y 又受 Z 驱动」
- 遇到反面证据或争议点，主动呈现，不回避
- **分析方法不暴露**——所有内部方法、分析框架、工具代号不出现报告中

**行动建议 / 趋势判断 / 战略含义**（按类型选择，见 [references/STRUCTURE.md](references/STRUCTURE.md)）

**不确定性评估**（必须）
- 关键不确定性清单
- 每条标注：如果被消除，是否改变推荐？→ 会改变 → 高优先级（建议用户优先查证）；不会改变 → 低优先级（可接受）

**参考来源**（必须）
- 来源列表，按贡献度或主题组织

**报告质量门**（见 [references/STRUCTURE.md](references/STRUCTURE.md)）

### Step 3: 追问入口

报告输出后默认自动保存。用户可对任意发现聚焦深挖，无需重新发起完整研究。

- 基于已有研究上下文，生成 2-3 组定向搜索关键词
- 优先搜索可能证明被追问发现错误的证据（**决策反转搜索**）
- 仅对追问点做窄化深挖，不展开全部分析维度
- 增量输出至已有报告末尾，不覆盖原文
