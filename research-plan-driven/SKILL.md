## 工作流变体

### 完整流程（默认）
所有 5 个阶段（调研 → 规划 → 待办清单 → 实现 → 验证），含审批关口。适用于任务涉及重大未知、跨模块变更或重要设计决策时。

### 快捷流程

适用于任务范围狭窄、充分理解、模型已有信息足以支撑规划的场景。流程：任务澄清 → 调研 → 规划 → 待办清单 → 实现 → 验证。

**核心原则：投入匹配任务。** 两种流程的调研产物都是"描述存在"——区别在于获得这些描述需要多大的调研投入，而非产物内容范围或格式上的差异。

**调研投入规则：**
- 模型对任务涉及的部分已有准确认识，无需探索代码库即可为规划提供足够的存在描述 → 免于代码库探索、子 agent 委派、深度证据锚定
- 模型缺乏足够认知的维度 → 仍需针对性地调查该维度。快捷不是自动化减负，是评估后决定不查无关部分
- 产物仍产出 `docs/<task-slug>.research.md`，在任务范围内描述代码库现状、接口签名、约束等存在事实。产物格式和审批关口与完整流程一致
- 规划阶段发现调研深度不足 → 切换回完整流程补调研。尝试直接规划是快捷流程唯一的失败模式

**何时启用：**
- 仅在任务澄清阶段、理解了任务范围后，由你判断并明确批准
- 判断依据：任务涉及的代码库部分是否在模型能可靠回答的已知范围内？如果有超出已知范围的关键模块，建议选完整流程

## 关键规则

| 不可协商规则 | 适用阶段 |
| ---- | ----------------- |
| 推进到下一阶段前需获得明确批准 | 调研, 规划, 待办清单 |
| 将每个已批准的产物视为下一阶段的契约 | 调研, 规划, 待办清单 |
| 每次审查修订都需进行批判性思考，保持上下文和逻辑一致性 | 调研, 规划, 待办清单 |
| 审查发现上游产物存在实质性问题时，回退到最早无效阶段，而非修订当前产物绕过 | 调研, 规划, 待办清单 |
| 除非用户明确要求，不读取 `docs/` 下无关文件 | 所有阶段 |
| 用户批准进入下一阶段后，先阅读对应的参考文件 | 所有阶段 |
| 问题必须聚焦于精确化真实需求 | 任务澄清 |
| 苏格拉底式提问，每个问题附带你的推荐答案 | 任务澄清 |
| 执行阶段无需停下等待批准 | 实现, 验证 |

## 批准标准

推进到下一阶段前必须获得明确批准。

**算作批准：**
- "approved" / "approve" / "可以" / "同意" / "批准"
- 任何明确表明产物可以继续推进的表述

**不算作批准：**
- 模糊的一词确认（"好"、"OK"、"嗯"、"LGTM"、"looks good"），未附带明确的推进信号
- 沉默或未提出反对意见
- 提出修改或质疑的评论

如果回复有歧义——请确认。措辞：*"确认一下：上面的算 approve 吗？还是需要再改一轮？"*

## 任务命名

为任务生成一个简短、稳定、可读的 `task-slug`，并在整个流程中使用。

> 如果从用户提供的产物恢复，复用其 `task-slug`。

`task-slug` 用于命名必要的产物：

- `docs/<task-slug>.research.md`

- `docs/<task-slug>.plan.md`

**规则：** 使用项目根目录下的 `docs` 目录，不存在则创建

## 任务澄清

在调研开始之前，通过结构化对话将原始任务/想法转化为**清晰、准确、具体**的共同理解。

**规则：**

- 每条消息提 **1-3 个问题**；将相关问题分组
- 尽可能使用**多选题**
- 仅在必要时才使用开放性问题
- 如果任务的某个方面需要深入，拆分为多个问题

**聚焦于理解：**

- 目的
- 具体的交付物或成功标准
- 约束
- 边界
- 明确的非目标

这一步对后续调研和规划的质量至关重要。

**产出物：** 任务澄清结束时，将确认的目的、交付物、约束、边界、非目标写入调研产物 `docs/<task-slug>.research.md` 开头，作为边界性信息。调研据此展开——人审调研时，开头即可确认范围是否对齐。

| 提供的产物 | 恢复阶段 |

## 审查-修订 循环

**目的**：多轮打磨产物至用户明确批准，避免前期错误导致后续返工。

### 批注格式

产物 `.md` 文件中的审查批注采用以下格式：

### {TYPE} [(on: "...") | (re: "...")] [Line N]

> 批注正文

_— Reviewer, YYYY/MM/DD HH:MM_

`TYPE` 取值：`COMMENT` / `SUGGESTION` / `QUESTION` / `APPROVE` / `REJECT`

查找方式：搜索 `###` 后紧跟上述 `TYPE` 之一的段落（`### COMMENT`、`### SUGGESTION` 等），及其后的 `>` 引用块和 `_— Reviewer` 时间戳行。

**核心约束：** 修订循环只修改当前阶段的产物。不在此直接修改上游产物。如果批注涉及上游问题，按回退协议处理。

### 执行步骤

1. **分类所有审查评论：** 每条评论先判断——问题根因在当前产物，还是上游某个已批准产物？
   - 根因在上游 → 走回退协议（见下文）。此当前产物保留不变，不尝试绕过或修补上游问题。
   - 根因在当前产物 → 走修订流程（步骤 2-4）。
2. 逐一分析当前产物的评论，修订产物，移除已解决的评论区块
3. **重新提交前自检：** 逐条读取行号确认修复已存在（非记忆）、通读全文检查未引入新问题、对未完全解决的评论如实说明
4. 请用户再审，重复直到明确批准

**卡住时：** 2+ 轮仍未解决核心问题 → 停止逐条修、问用户、下一轮只聚焦这一问题。

## 回退协议（通用）

审查中如果识别到问题根因在上游已批准产物，不在此修补或绕过——回退到最早无效阶段。

### 触发条件

审查评论指向的缺陷满足以下任一条件：

- **事实依赖崩塌：** 当前产物的某个结论基于上游产物的错误事实
- **边界失效：** 上游产物约定的范围/约束/非目标已不足以支撑正确规划
- **遗漏关键信息：** 上游产物缺少当前阶段作出正确决策所必需的信息
- **用户明确提出修订上游物：** 用户说"调研这里不对"而非"规划这里不对"

### 执行步骤

1. **识别最早无效的阶段。** 哪个产物的信息或决策错了，导致后续连锁偏离？
2. **告知用户。** 说明回退理由、目标阶段、将丢弃哪些下游产物。等待确认后再执行。
3. **产物处置：**
   - 无效阶段及其所有下游阶段的产物 → 丢弃
   - 所有上游（之前已批准的）产物 → 保留
   - 示例：规划无效 → 丢弃 `plan.md` 及其待办清单；保留 `research.md`
4. **加载。** 阅读回退到的阶段的参考文件
5. **重新进入。** 从回退阶段开始一个新的产物，走该阶段的标准流程（审查-修订循环），直到批准。上游保留的产物保持已批准状态。

### 与实现阶段回退的关系

实现阶段的回退遵循相同协议（已整合在 `references/phase-4-implementation.md` 中），是此通用协议在实现阶段的具象化。两者的触发场景和判定逻辑一致。

## 工作流完成

仅当所有实现均已验证、反馈并得到用户确认后，本技能才算完成。
