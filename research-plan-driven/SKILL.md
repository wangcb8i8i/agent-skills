## 工作流变体

### 完整流程（默认）
所有 5 个阶段（调研 → 规划 → 待办清单 → 实现 → 验证），含审批关口。适用于任务涉及重大未知、跨模块变更或重要设计决策时。

### 快捷流程
调研阶段以轻量形式执行（产物精简，无需代码库探索），适用于任务范围狭窄、理解充分、用户已有足够上下文。流程：任务澄清 → 调研（轻量） → 规划 → 待办清单 → 实现 → 验证。

**规则：**
- 仅在任务澄清阶段、理解了任务范围后，才建议使用快捷流程
- 用户必须明确批准使用快捷流程
- 调研阶段仍需产出 `docs/<task-slug>.research.md`，但内容仅需覆盖：任务澄清确认的范围、边界、已知约束、关键假设
- 免于代码库探索、子 agent 委派、深度证据锚定——但产物格式和审批关口不变
- 如果在规划阶段出现不确定性，建议切换回完整流程补调研深度

## 关键规则

| 不可协商规则 | 适用阶段 |
| ---- | ----------------- |
| 推进到下一阶段前需获得明确批准 | 调研, 规划, 待办清单 |
| 将每个已批准的产物视为下一阶段的契约 | 调研, 规划, 待办清单 |
| 每次审查修订都需进行批判性思考，保持上下文和逻辑一致性 | 调研, 规划, 待办清单 |
| 除非用户明确要求，不读取 `docs/` 下无关文件 | 所有阶段 |
| 用户批准进入下一阶段后，先阅读对应的参考文件 | 所有阶段 |
| 问题必须聚焦于精确化真实需求 | 任务澄清 |
| 苏格拉底式提问，每个问题附带你的推荐答案 | 任务澄清 |
| 执行阶段无需停下等待批准 | 实现, 验证 |

## 批准标准

推进到下一阶段前必须获得明确批准。

**算作批准：**
- "approved" / "approve" / "可以" / "同意" / "批准"
- 任何明确表明产物可以继续推进的表述

**不算作批准：**
- 模糊的一词确认（"好"、"OK"、"嗯"、"LGTM"、"looks good"），未附带明确的推进信号
- 沉默或未提出反对意见
- 提出修改或质疑的评论

如果回复有歧义——请确认。措辞：*"确认一下：上面的算 approve 吗？还是需要再改一轮？"*

## 任务命名

为任务生成一个简短、稳定、可读的 `task-slug`，并在整个流程中使用。

> 如果从用户提供的产物恢复，复用其 `task-slug`。

`task-slug` 用于命名必要的产物：

- `docs/<task-slug>.research.md`

- `docs/<task-slug>.plan.md`

**规则：** 使用项目根目录下的 `docs` 目录，不存在则创建

## 任务澄清

在调研开始之前，通过结构化对话将原始任务/想法转化为**清晰、准确、具体**的共同理解。

**规则：**

- 每条消息提 **1-3 个问题**；将相关问题分组
- 尽可能使用**多选题**
- 仅在必要时才使用开放性问题
- 如果任务的某个方面需要深入，拆分为多个问题

**聚焦于理解：**

- 目的
- 具体的交付物或成功标准
- 约束
- 边界
- 明确的非目标

这一步对后续调研和规划的质量至关重要。

## 审查-修订 循环

**目的**：多轮打磨产物至用户明确批准，避免前期错误导致后续返工。

### 执行步骤

1. 找出所有审查评论，逐一分析后修订产物，移除已解决的评论区块
2. **重新提交前自检：** 逐条读取行号确认修复已存在（非记忆）、通读全文检查未引入新问题、对未完全解决的评论如实说明
3. 请用户再审，重复直到明确批准

**卡住时：** 2+ 轮仍未解决核心问题 → 停止逐条修、问用户、下一轮只聚焦这一问题。

## 工作流完成

仅当所有实现均已验证、反馈并得到用户确认后，本技能才算完成。
