# 实现

## 目的

精确执行已批准的待办清单，以保持审批关口工作流的完整性。

实现遵循已批准的规划和已批准的待办清单。它不重新定义它们。

## 执行步骤

1. 重新读取 `docs/<task-slug>.plan.md` 中已批准的规划和已批准的待办清单。
2. **忠实执行。** 按顺序执行已批准的待办项，从头到尾。对每个项：
   - 完成所述工作。
   - 当工作真正完成时，更新 checkbox：`- [ ]` 变为 `- [x]`。
   - 待办清单已包含验证项（如运行测试、检查输出）。在执行序列中一并完成它们。
3. **停。** 如果已批准的产物不再支持正确的执行，立即停止。
4. 当所有待办项都被勾选后，进入验证和反馈阶段。

#### 回退协议

此阶段遵循 SKILL.md 定义的通用回退协议。实现阶段的特殊性在于：触发回退的信号是"已批准的产物不再支持正确的执行"（而非审查评论），且回退发生在编码途中。

**实现阶段触发条件：**
- 代码结构与已批准产物的描述不一致，且非细微接线差异
- 已批准的待办清单缺少必需步骤
- 新发现的需求改变了范围或设计
- 已批准规划不再反映实际代码库状态

**执行（同通用协议）：**
1. **立即停止。** 不要继续实现。
2. **识别最早无效的阶段。** 哪个产物（调研、规划或待办清单）出了问题？
3. **告知用户。** 说明回退理由、目标阶段、将丢弃的产物。等待确认。
4. **产物处置：** 无效阶段及其所有下游阶段的产物丢弃，上游保留。
5. **加载。** 阅读回退到的阶段的参考文件。
6. **重新进入。** 从回退阶段开始新产物，走标准审查-批准循环。上游保留的产物保持已批准状态。

## 变更策略

| 变更类型 | 处理方式 |
|---|---|
| 小型接线、imports、测试调整、次要支持性修改 | 无需重开前期阶段 |
| 新范围、未批准的变更行为、静默重新设计、机会性重构 | 回退到最早无效阶段 |
| 已批准规划不再符合实际代码库 | 回退 |
| 新发现的改变范围或设计的需求 | 回退 |

如果已批准的待办清单缺少必要步骤，停下来回到最早无效阶段。一旦规划失效，不要继续编码。

## 约束

- 不要在已批准的工作之外增加范围、行为变更或架构变更。
- 不要实现未包含在已批准待办清单中的工作。
- 不要将未完成的任务标记为完成。
- 不要做出本应在已批准产物中捕获的新设计决策。

## 完成标准

**忠实执行。** 仅在以下条件满足时才能离开此阶段：

- 每个已批准的待办项都已标记为完成（`- [ ]` 已改为 `- [x]`）
- 实现保持在已批准范围内
- 没有需要回退到更早阶段的未解决的阻塞点
