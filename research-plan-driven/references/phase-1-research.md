# 调研

## 目的

基于用户提供的材料和代码库中与任务相关的部分，建立对所有任务相关信息的深度理解，形成一份关于当前现状的独立事实记录。

## 执行步骤

1. 以具体的、可调研的术语定义任务，并将调查拆分为聚焦的问题或子系统，确保各方向不重叠。
2. 界定调研范围——倒推需要确认哪些现状以支撑安全规划：
   - 问自己：*"涉及的文件或模块？"*、*"关键未知因素？"*、*"用户已提供的上下文？"*
   - 只调研必要项，已知或与决策无关的维度可跳过
   - 够用即止
   - 按以下维度组织检查（可跳过但须注明原因）：
     - 数据在受影响路径的流转方式
     - 相关代码库模式和约定
     - 影响规划的陷阱或约束
     - 已有的可复用功能
     - 查了但不存在的内容
   - **边界守护：** 所有发现通过"描述存在"检验（见调研边界）。发现规划性质的内容（推荐、设计、伪代码）标记或移除。
3. 默认将代码库探索或调查委派给子 agent 处理，仅当明确判断"自行检查比委派更窄更快"时才亲自处理。
4. 将所有委派结果视为未经验证的输入，然后合起来审查，解决冲突，仅在需要验证关键声明或填补重大缺口时亲自阅读代码。
5. 仅当代码库和委派调查都无法回答时，才向用户询问决策、需求或事实。
6. 编写可读/可审的调研产物，包含经过验证的、与任务相关的代码库现状和约束，以及任何明确的未解决的代码库外问题。
7. 提交前，对照步骤 2 的范围定义逐项自检。未覆盖或跳过的维度需注明原因。

## 调研边界

**描述存在。** 调研是对"已经存在什么"的发现，而不是对"应该构建什么"的设计。检验标准：*"这条描述的是已经存在的东西吗？"*

### 属于调研范畴
- 当前代码库结构、数据流和行为
- 可用选项及其特征（能力、成本、局限）
- 现有模式、约定和约束
- 已知问题、缺口及其后果
- 可验证的事实，附带来源引用

### 不属于调研范畴
- 推荐或决策（如"使用源 X"、"选择方法 Y"）
- 架构或流程重新设计
- 新功能的伪代码、代码片段或类型定义
- 实施阶段、路线图或分步计划
- 推荐某一选择的决策表（相对于比较现有选项）

### 边界判断

不确定时，问："这条描述的是已经存在的东西吗？"
- 是 → 属于调研
- 否 → 不属于调研

## 产物组织

调研产物不预设节标题和顺序。产物结构由你在编写过程中对每条信息执行判断而涌现，而非匹配一个预定义的填空清单。

### 信息分类

对产物中的每条信息，判定其类型：

- **边界性信息**：定义"能做什么、不能做什么"——范围、非目标、约束、必须遵守的约定。**漏读则对现状的理解根本性错误。**
- **关键性信息**：对理解现状最关键的事实——核心发现、已知盲区、未解决问题、重大风险。**漏读则对现状的整体判断可能偏颇。**
- **支撑性信息**：提供证据、背景、细节——代码库具体行为、数据流实际路径、现有模式的表现。**漏读则理解不够精确，但不影响大方向判断。**

### 排序规则

按优先级自然地组织产物，判断顺序是"先类型，再同一类型内按重要程度"：

1. **边界性信息必须出现在所有关键性信息之前。** 先框定能做什么、不能做什么，再进入具体内容。
2. **关键性信息必须出现在所有支撑性信息之前。** 先建立对现状最关键的理解，再展开证据细节。
3. **同类型内，按"如果被漏读或后读会导致后续内容被误解"的程度降序排列。**

### 判断链（用于自检）

产物的组织过程不是一个模板填空。每次输出一个内容块时，执行以下推理：

> "我为什么要写这一段？它属于什么类型？如果它排在后面，读者在读到它之前会基于什么假设——这个假设对吗？"
> "**描述存在。** 这条信息描述的是已经存在的东西吗？"

如果某段信息放哪里都成立，说明它还可以，但这不是重点。重点是对排在前面的每段信息问：

> "这一段如果不在这里，后续内容会产生什么误解？"

如果你觉得"放哪里都没有问题"，说明这一段可能是背景噪音，考虑是否保留。

### 产物组织不是章节规划

这种方式产出的结构每次都可能不同：

- 任务只有一个边界约束和一条关键发现 → 产物可能只有对应的两段。
- 任务跨多个子系统且约定繁多 → 产物自然按"约定 → 关键性发现 → 逐子系统细节"展开。

没有"写着写着发现少了哪个节就补上"的操作。如果有多个子系统逐个展开是合理的，那是因为几个子系统各自产生了关键性信息——不是因为"模板中有详细发现这一节"。

**重要——信息分类不是章节标题：** 边界性 / 关键性 / 支撑性 是编写时判断信息类型和排序的标签，不是产物的章节名称。产物标题应使用领域实际名称（模块名、数据流名、问题域术语）——读者读到的是调研结论本身，不是"我在按方法论写作"。

## 调研委派

当委托可以提高覆盖率或速度时，并行启动 1-3 个聚焦的子 agent。按聚焦的问题或子系统拆分工作，避免任务重叠。

给每个子 agent 的 prompt 必须包含：

* 调查什么以及为什么（任务上下文）
* 需要查看的特定文件或模式
* 需要返回什么样的发现（实际的实现、模式、数据流、约定、边界情况）
* 约束：仅调查事实——不提议解决方案、架构或功能想法

一个好的子 agent prompt 应该：

* 要求全面、结构化地追踪代码，专注于理解抽象、架构和控制流（理解事物如何工作）
* 明确约束和边界
* 提供足够的上下文，使 agent 能做出判断，而非仅遵循狭窄指令

协调者审查子 agent 的发现，填补重要缺口，从代码库证据中解决冲突，并综合生成 `docs/<task-slug>.research.md`。

不要静默合并冲突主张。明确暴露未解决的歧义。始终验证：
- 冲突的发现
- 低置信度或模糊的发现

## 审查原则

产物必须能让审查者自信地回答：*这份调研是否足够完整和准确，可以基于它进行规划？*

- **范围对齐。** 每个重要发现都能追溯到已同意的任务范围。偏离轨道的探索以补充内容形式呈现，而非混入正文。
- **范围完整。** 审查者能看到调查了什么、发现了什么、还有哪些未回答。重大缺口暴露而非隐藏。
- **证据锚定。** 每个非显而易见的发现都包含来源引用 `[src: path:line]`。冲突发现暴露而非静默合并。
- **置信度标记。** 对非平凡的发现标记置信度（high / medium / low）。推测不是调研。
- **约束优先。** 所有边界性信息（范围、约束、约定）出现在关键性和支撑性信息之前。不会出现"读者读到一半才发现有个重要的前提被埋在后面"的情况。
- **知识边界清晰。** 产物的结尾应清晰说明报告的知识边界——确定了什么、未确定什么，让读者知道报告的结论止于何处。

## 最低标准

以下内容单独不能算作完成调研：

- 仅有文件列表，没有行为理解
- 仅有 grep 输出，没有综合
- 将设计方案当作事实呈现
- 将提议或解决方案作为调研发现呈现
- 来自单一来源的发现，未经交叉验证
- 未回答的对规划有实质影响的问题

优先使用紧凑的表格或图表——它们减少阅读成本，使产物更容易审查。

## 约束

- 不要在调研产物中包含已决定的实现方案作为调研事实。
- 不要在调研产物中包含新功能想法、架构提议或解决方案建议。
- 不要写实现代码。
- 不要将调研产物变成规划。
- 在有实质性调研问题未解决时，不要最终确定产物。

## 完成标准

仅在以下条件满足时才能离开此阶段：

- `docs/<task-slug>.research.md` 已存在
- 产物基于代码库证据
- 影响规划的实质性歧义已被解决或暴露
- 所有当前调研审查评论已被处理
- **产物通过提交前自检：每条声明的类型已做判断并放置合理，每个声明都有来源锚点，通过"描述存在"检验，未解决问题已明确列出**
- 用户已明确批准调研产物可用于规划
