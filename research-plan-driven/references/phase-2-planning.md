# 规划

## 目的

基于已批准的调研，创建详细、可审查、具体的实现方案。

## 执行步骤

1. 基于最新批准的调研产物（如果选择了快捷流程，则基于任务澄清阶段的共同理解）。
2. 将预期实现方案定义得足够具体，确保编码时无需临时发明架构。
3. 明确列出预期会变更的确切文件、组件、接口、契约或边界。
4. 记录重要的技术决策及其理由。
5. 使风险、权衡、范围限制和未决决策易于审查。
6. 将实现方案写入规划产物。产物组织结构从审阅者的认知路径推导——模拟首次接触此计划的审阅者，按"读者需要先知道什么、再知道什么"的先后顺序组织，而非按主题分类排列章节。没有预设章节。认知路径按以下层次展开，每一层回答读者在此阶段最关心的问题：

   - **全景心智模型** — "系统改完后长什么样？" 模块关系、新增/改动的组件边界、结构层面的全貌。
   - **行为模型** — "一次完整流程怎么走？" 时序、数据流转、状态变化。连接心智模型和合约级细节的中间层。
   - **合约级变更细节** — "每个文件/接口最终变成什么样？" 签名、边界行为、错误语义。粒度标准见"合约级粒度定义"。

   决策理由附着在对应的层次中（架构决策跟心智模型，接口决策跟合约级变更）。风险和非目标在合约级变更细节之后、验证策略之前展开——此时读者已拥有理解每项风险的完整上下文。
7. 以审阅者视角通读规划产物，检查逻辑一致性和信息完整性。
8. 根据 `guides/pre-review.md` 对规划产物进行预审。
9. 请用户审查规划产物。尚未开始实现；等待明确批准。


## 审查原则

产物必须能让审查者自信地回答：*这个规划可以安全地实现吗？*

**产物完成后逐条自检：**

- **读者能否理解任务背景和目标？**（让随机读者能确认"这是否是与我相关的任务"即可，非复制调研内容）
- **读者能否在进入文件级细节前，已建立变更后系统的全景心智模型？**（可以是图或文，不限形式。要点是"先看到整体再看到具体"）
- **每个变更文件的最终状态是否明确？**（变成什么样而非做了什么操作。接口、数据形状、签名变化必须显式写出。要达到合约级粒度——见下方定义）
- **重要决策是否有理由和被否方案？**（隐式决策是审查阻塞点。无实质取舍的决策注明"默认选项"即可）
- **风险和非目标是否已明确暴露？**（审查者不应在实现中期才发现"这个场景没考虑"）

## 合约级粒度定义

"合约级粒度"是指：变更描述达到让审阅者能判断设计正确性、让实现者无需再做设计决策的程度。

**反例（不够）**：
> 新增用户列表查询接口。

**正例（够）**：
> `UserService.list(opts: { limit?: number; cursor?: string }) → { items: User[]; total: number; nextCursor?: string | null }`
> - limit 默认 20，最大 100，超出返回 400
> - cursor 为空返回第一页
> - 空结果返回 `{ items: []; total: 0 }`，不抛异常
> - 鉴权：需认证，当前用户只能查自己组织下的用户

**判断标准**：如果实现者拿到计划后，还需要自己决定"空结果抛不抛异常""limit 超了怎么办"，说明粒度不够。

## 最低标准

在以下情况下，规划需要在创建待办清单前进行修订：
- 主要实现决策仍为隐式
- 受影响文件或边界未具体命名
- 代码变更的预期结构模糊
- 变更描述未到达合约级粒度
- 验证方法缺失或模糊
- 关键设计缺口需要在实现阶段临时解决

## 约束

- 不要写实现代码。
- 不要在此阶段创建待办清单。
- 不要将规划批准视为实现批准。
- 不要添加已批准调研未证明的范围。
- 不要留下重要的设计缺口在实现阶段临时填补。

## 完成标准

仅在以下条件满足时才能离开此阶段：

- `docs/<task-slug>.plan.md` 已存在
- 规划具体且可审查
- 事后自检已执行（见下方"事后自检"）
- 所有当前规划审查评论已被处理
- 用户已明确批准规划产物

## 事后自检

规划产物编写完成后，使用以下清单逐项检查。**编写过程中不引用此清单**——它仅在产物写完后作为验证工具使用。

每项确认已覆盖，或判断不适用（必要时注明原因）。

| 维度 | 核心问题 | 何时可跳过 |
|---|---|---|
| **范围对齐** | 背景是否让读者确认"这是否与我相关"？ | 任务极简单，标题足以说明 |
| **整体心智模型** | 进入细节前是否先看到变更后的系统全貌？ | 单文件单函数改动 |
| **合约级变更** | 每个变更文件的最终状态是否写到签名、边界行为、错误语义级？ | 纯重构(行为不变) |
| **关键决策** | 实质取舍是否附理由和被否方案？默认选项是否标注？ | 无实质取舍的配置型变更 |
| **风险/非目标** | 哪些明确不做？哪些已知风险？ | 零风险的简单变更 |
| **行为模型** | 审阅者能否理解变更后的系统一次完整运行过程？ | 孤立原子变更、纯新增（不修改既有行为链） |
| **验证策略** | 如何确认实现正确？ | 无歧义的原子变更 |
