# 规划

## 目的

基于已批准的调研，创建详细、可审查、具体的实现方案。

## 执行步骤

1. 基于最新批准的调研产物（如果选择了快捷流程，则基于任务澄清阶段的共同理解）。
2. 将预期实现方案定义得足够具体，确保编码时无需临时发明架构。
3. 明确列出预期会变更的确切文件、组件、接口、契约或边界。
4. 记录重要的技术决策及其理由。
5. 使风险、权衡、范围限制和未决决策易于审查。
6. 将实现方案写入规划产物。产物组织结构从审阅者的认知路径推导——模拟首次接触此计划的审阅者，按"读者需要先知道什么、再知道什么"的先后顺序组织，而非按主题分类排列章节。没有预设章节。认知路径按以下顺序展开：

   - **背景/范围** — 开门见山：这是什么任务？一句话足以确认"与我相关"。
   - **<全景心智模型标题>** — "系统改完后长什么样？" 模块关系、新增/改动的组件边界、结构层面的全貌。标题用域名术语（如"变更范围总览"）。跨文件架构级决策 inline 附在这里。
   - **<行为流程标题>** — "一次完整流程怎么走？" 时序、数据流转、状态变化。连接心智模型和合约级细节的中间层。标题用域名术语（如"搜索流程（改动后）"）。
   - **变更清单** — 读者知道流程如何运作后、进入详细变更前，提供一份轻量的文件级清单：列出所有涉及文件、变更性质（新增/修改/重构）、一句话摘要。作用是让读者带着一张"地图"进入详细阅读。
   - **<文件级变更标题>** — 按文件逐个展开。单文件级决策理由 inline 附着在对应变更点旁。深度边界：描述到**签名 + 核心逻辑/关键分支**为止，不粘贴完整实现体。
   - **边界** — 合并为两部分：
     1. 范围边界：✅ 做的事 / ❌ 不做的事 / ⚠️ 需确认的事
     2. 已知风险：保留「风险 — 缓解」两列，去掉概率/影响
   - **验证策略** — 末尾集中。每项为**可观测的通过标准**，而非验证动作。格式是"<看到什么>"，不是"<运行什么>"。
   - **决策日志** — 可翻页附录，不在主阅读路径。记录跨文件/架构级决策和时间戳，供后续 session 追溯。

   **规则：**
   - **读者先知道。** 每一步问：读者现在最需要知道什么？答案决定下一段内容，而不是"下一个方法论层是什么"。
   - 认知路径层次不是目录标题。产物标题用域名术语。
   - 深度边界判断：如果一段变更描述用 Git diff 展示更合适——它越界了，缩回到合约级。
7. 以审阅者视角通读规划产物，检查逻辑一致性和信息完整性。
8. 根据 `guides/pre-review.md` 对规划产物进行预审。
9. 请用户审查规划产物。尚未开始实现；等待明确批准。


## 审查原则

**读者先知道。** 产物必须能让审查者自信地回答：*这个规划可以安全地实现吗？*

**产物完成后逐条自检：**

- **读者能否理解任务背景和目标？**（让随机读者能确认"这是否是与我相关的任务"即可，非复制调研内容）
- **读者能否在进入文件级细节前，已建立变更后系统的全景心智模型？**（可以是图或文，不限形式。要点是"先看到整体再看到具体"）
- **每个变更文件的最终状态是否明确？**（变成什么样而非做了什么操作。接口、数据形状、签名变化必须显式写出。要达到合约级粒度——见下方定义）
- **重要决策是否有理由和被否方案？**（隐式决策是审查阻塞点。无实质取舍的决策注明"默认选项"即可）
- **风险和非目标是否已明确暴露？**（审查者不应在实现中期才发现"这个场景没考虑"）
- **边界是否清晰？**（✅做 / ❌不做 / ⚠️确认 的分类是否合理？风险与缓解是否对应？）

## 合约级粒度定义

"合约级粒度"是指：变更描述达到让审阅者能判断设计正确性、让实现者无需再做设计决策的程度。

**反例（不够）**：
> 新增用户列表查询接口。

**正例（够）**：
> `UserService.list(opts: { limit?: number; cursor?: string }) → { items: User[]; total: number; nextCursor?: string | null }`
> - limit 默认 20，最大 100，超出返回 400
> - cursor 为空返回第一页
> - 空结果返回 `{ items: []; total: 0 }`，不抛异常
> - 鉴权：需认证，当前用户只能查自己组织下的用户

**判断标准**：如果实现者拿到计划后，还需要自己决定"空结果抛不抛异常""limit 超了怎么办"，说明粒度不够。

**上界——不粘贴完整实现：** 描述到**签名 + 核心逻辑/关键分支**为止。完整源码应在实现阶段的 Git diff 中呈现，不在规划产物中。
> 反例：粘贴一个 30 行完整函数的全部代码。
> 正例：描述接口签名、边界行为、关键分支逻辑（如"若 content_type=opinion 则走分支 A，否则走分支 B"）。

**边界判断：** 如果一段变更描述用 Git diff 展示更合适——它越界了，缩回到合约级。

## 最低标准

在以下情况下，规划需要在创建待办清单前进行修订：
- 主要实现决策仍为隐式
- 受影响文件或边界未具体命名
- 代码变更的预期结构模糊
- 变更描述未到达合约级粒度
- 验证方法缺失或模糊
- 关键设计缺口需要在实现阶段临时解决

## 约束

- 不要写实现代码。
- 不要在此阶段创建待办清单。
- 不要将规划批准视为实现批准。
- 不要添加已批准调研未证明的范围。
- 不要留下重要的设计缺口在实现阶段临时填补。

## 完成标准

仅在以下条件满足时才能离开此阶段：

- `docs/<task-slug>.plan.md` 已存在
- 规划具体且可审查
- 事后自检已执行（见下方"事后自检"）
- 所有当前规划审查评论已被处理
- 用户已明确批准规划产物

## 事后自检

规划产物编写完成后，使用以下清单逐项检查。**编写过程中不引用此清单**——它仅在产物写完后作为验证工具使用。

每项确认已覆盖，或判断不适用（必要时注明原因）。

| 维度 | 核心问题 | 何时可跳过 |
|---|---|---|
| **范围对齐** | 背景是否让读者确认"这是否与我相关"？ | 任务极简单，标题足以说明 |
| **全景心智模型** | 进入细节前是否先看到变更后的系统全貌？ | 单文件单函数改动 |
| **行为模型** | 审阅者能否理解变更后的系统一次完整运行过程？ | 孤立原子变更、纯新增（不修改既有行为链） |
| **变更清单** | 进入详细变更前是否有所有涉及文件的概览清单？ | 单文件改动 |
| **合约级变更** | 每个变更文件的最终状态是否写到签名、边界行为、错误语义级？ | 纯重构(行为不变) |
| **边界** | ✅ ❌ ⚠️ 是否完整？风险与缓解是否对应？ | 零风险的简单变更 |
| **验证策略** | 每项是否是可观测的通过标准，而非验证动作？ | 无歧义的原子变更 |
| **决策日志** | 跨文件/架构级决策是否已记录理由和时间戳？ | 无实质取舍的配置型变更 |
