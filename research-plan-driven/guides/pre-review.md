# 预审

## 目的

从选定的审查角色视角，对规划产物进行聚焦的预审——批评性的、基于证据的、选择性的。

---

## 必要输入

- `docs/<task-slug>.research.md` — 已批准的调研
- `docs/<task-slug>.plan.md` — 待审查的规划

任一缺失或不足以支持审查 → 退出并告知用户。

---

## 候选审查角色

| 角色 | 强匹配信号 | 主要关注点 |
|------|-----------|-----------|
| 架构师 | 新抽象、服务边界、子系统交互、所有权变更 | 结构、边界、耦合 |
| 测试工程师 | 分支行为、状态转换、重试、回退、顺序依赖 | 逻辑遗漏、隐蔽情况 |
| 软件工程师 | helpers、wrappers、间接层、重构较多的结构 | 可读性成本、局部复杂度 |
| 性能工程师 | 大数据量循环、查询、缓存、并发、高吞吐路径 | 延迟、吞吐量、瓶颈 |
| 安全工程师 | auth、权限、secrets、外部可达表面、用户输入 | 误用风险、暴露面 |
| API 设计师 | request/response 形状变更、版本策略、导出接口 | 兼容性、契约清晰度 |
| 前端工程师 | 异步 UI、loading/error 状态、DOM 时序、无障碍 | 竞态、UI 一致性 |
| 产品审查者 | 面向用户的行为变更、工作流修改、业务规则调整 | 需求对齐、验收标准 |
| 文档工程师 | 运维变更、配置更新、CLI 输出变更、上线流程 | 产物完整性、升级清晰度 |

选择 **1-3 个**：优先覆盖实质性风险的最小角色集合。无强匹配则退出。

---

## 派发与收集

为每个角色独立提供：角色 + 视角 + 产物路径 + **对抗性立场**（假设规划至少有两个需找到的实质性缺陷）。

每个角色输出按严重级别分类的发现（严重 / 重要），附带 Plan 锚点和修订建议。

---

## 发现标准

一个发现有效需同时满足：

- **锚定：** 有 `plan.md` 的直接锚点 + `research.md` 锚点（或说明 research 沉默的实质性理由）
- **实质性：** 会影响实现可靠性、验证清晰度、架构安全性、兼容性或用户批准信心

**不报告：** 风格偏好、无产物证据的推测、实现细节苛求、泛泛最佳实践、主观建议。
**丢弃：** 无锚点、发明需求、超出角色领域、纯重复的发现。
**全部不可用 → 预审失败**，回到正常审查流程。

---

## 严重级别

| 级别 | 定义 |
|------|------|
| **严重** | 可能导致执行失败、无效验证、兼容性破坏、严重安全暴露 |
| **重要** | 可能导致返工、歧义、错误选择，尚不构成灾难性阻塞 |
| **次要** | 不进入预审输出 |

---

## 综合规则

1. 对照产物验证每个保留问题，**不机械接受**角色主张
2. 去重，保留证据最强的表述
3. 丢弃次要（Minor）发现
4. 每个 plan 锚点后附加括号简注章节摘要（如 `§三 3.2（参数说明）`）
5. 决定是否附带修订建议——仅在能给出具体、可操作的修订时才提供
6. 预审期间**不修改规划**

---

## 模板

写入 `docs/<task-slug>.pre-review.md`。

```md
# 预审：<task-slug>

## 审阅人
<角色 1>、<角色 2>

## 总体评估
<plan 是否 ready、主要偏差、用户应重点关注什么>
**审阅建议**: <直接进入用户 review / 建议审阅后再决定 / 不建议进入 review>

## 严重问题

> <问题描述 + 为何严重>
- **Plan 位置**：<节名>
- **修订建议**：<具体建议>

## 重要问题

> <问题描述 + 为何重要>
- **Plan 位置**：<节名>
- **修订建议**：<具体建议>

## 待决策问题

- **背景**：<现状> **影响**：<不决策的影响> **需要决策**：<用户决定什么>
```

---

## 约束

- 无强匹配角色时不运行预审
- 不自动修改规划产物
- 不将预审备注视为批准

---

## 审查流程

1. 检查必要输入 → 不足则退出
2. 选择 1-3 个强匹配角色 → 无则退出
3. 并行派发审查角色
4. 按发现标准评估输出，丢弃不可用者
5. 综合可用发现，写入 `docs/<task-slug>.pre-review.md`
6. 告知用户，回到正常审查流程
