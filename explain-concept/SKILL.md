# explain-concept

概念不是孤岛。解释一个概念 = **说清**它是什么 + **划界**它不是什么的 + **归位**它在哪一层 + **织网**它和谁有关联。

目的不是让用户背诵定义，是让他真正理解——不会被邻近概念搞混、知道什么时候用它而不是用邻居。

## 原则

1. **问意图先于给答案。** 同样问"闭包是什么"，在读代码遇到障碍和系统学 JS 是两种解释。先读上下文，推断为什么问。
2. **四层按意图定深度。** 不是每次都铺满。判完意图，决定每层讲到什么粒度。
3. **界定 > 定义。** 一句大白话说清它是什么就够了。真正的理解是从"它不是 X"开始建立的——每个易混淆邻居一句话划清界限。
4. **不编造动机。** 不是所有概念都有清晰的设计动机。演化型概念诚实说"历史堆出来的"。

## 流程

### 1. 判意图

从上下文推断用户为什么问这个问题：

| 信号 | 意图 | 策略（强调/弱化） |
|------|------|-------------------|
| 前文讨论代码时突然提一个概念 | 遇到障碍 | 定义·界定优先，够用就停 |
| 主动对比两个概念 | 校正混淆 | 界定拉满，归属·关联可省 |
| "我在学 X" | 系统学习 | 四层均匀 |
| "X 和 Y 选哪个" | 做决策 | 界定·关联优先，弱化归属 |
| "你刚提到 X" | 好奇心延伸 | 定义·关联，中等深度 |
| 无上下文，直接问 | 未知 | 问一句"你是在读代码还是系统学" |

拿不准 → 往浅了走。跳过用户知道的比跳过他需要的代价小。

### 2. 校准

从对话历史推断用户已知什么概念——那些已知概念就是锚点。用户主动提到一个桥接概念 → 最高置信度信号。

拿不准 → 往低了估。用户不答 → 当零基础。

### 3. 定深度

按意图决定四层各自的深度：

```
定义：一句话 ~ 一段（视意图调节）
界定：0 ~ N 个邻居，每个一句话划清界限
归属：跳过 ~ 上层 + 下层各一句
关联：跳过 ~ 配套 + 同场景 + 协同
```

### 4. 展开

先说定义——一句话，大白话，能替换成日常用语而不丢意思。

再界定——每个易混淆邻居一句话说清根本差异。选最多 3 个，优先最常见的混淆。并排列出，一眼看完：

```
| 目标概念 | 易混淆概念 |
|---------|-----------|
| 决定性差异 1 | 决定性差异 1 |
| 决定性差异 2 | 决定性差异 2 |
```

再归属——上层概念是什么（它属于哪一类）、下层概念是什么（它的子类型或具体形式）。一行层级线代替 prose：

```
上层概念 → 目标概念 → 下层概念
```

再织网——水平关联：和谁配套使用、和谁在同一场景出现、和谁协作构成某个模式。

### 5. 收束

抛一个邻接方向或保持安静。别问"明白了吗？"

## 四层框架

概念解释的结构化参考：

| 层 | 回答的问题 | 验证方式 | 例子：闭包 |
|----|-----------|---------|-----------|
| 定义 | 它是什么 | 一句话大白话，替换日常词不丢意思 | 函数带着出生地的变量一起走 |
| 界定 | 它不是什么 | 每个易混淆邻居一句话划清根本差异 | 不是匿名函数、不是静态作用域 |
| 归属 | 它在哪一层 | 上层/父类 + 下层/子类型 | 上是高阶函数，下是词法作用域 |
| 关联 | 它和谁配合 | 配套使用、同场景出现、协作模式 | 配套：回调、IIFE；同场景：异步 |

## 完成标志

三层都过关才算讲透了：

1. 用户能用自己的大白话说清它是什么。
2. 用户能把它和最常混的那个邻居区分开。
3. 用户知道它上面是什么、下面是什么、和谁配合。

这三层都满足 → 抛邻接方向或保持安静。不满足 → 缺哪层补哪层。
