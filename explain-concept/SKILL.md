# explain-concept

用户抛出一个概念，就是在**概念空间**里设了一个**坐标原点**。解释 = 围绕原点沿多个维度**探索航行**，探索到覆盖用户意图的区域后**返航**。

探索可能进入**迷雾**——用户不熟的术语区域，需要额外解释成本。好的航行在用户的**版图**（已知坐标）内规划航路，不让探索变成迷路。

术语（原点、版图、迷雾、航行、返航等）是内部导航隐喻，只在技能内部使用。对外输出时用自然语言而非这些术语——不说「定原点」，直接确认概念；不说「返航」，直接总结。

## 原则

1. **意图决定半径**——同样问"闭包"，在读代码遇到障碍和系统学 JS 需要的探索半径不同。先判意图，再定航程。
2. **在版图内航行**——优先用已知概念做参照坐标。进入迷雾是成本，控制它。
3. **够用就返航**——探索到用户能在版图内定位原点即可，不必遍历所有维度。
4. **空间是迭代的**——首次探索定原点，后续每次从已探索空间的新锚点起航，把边界往外推。版图累增，不重置。

## 流程

以下流程可迭代。首次从用户提出的概念定原点。回锚后用户可能指向已探索空间中的任意节点（主原点、临时驻锚过的必要前置、或追问中指向的关联概念）选为新锚点，从该锚点重复 Stage 1→4 向外探索，Stage 5（指锚）决定何时进入下一轮迭代。空间在每次迭代中扩张，直到用户觉得当前区域的重要节点都踩过了。

### 1. 定原点 + 测半径

定原点：用户问的概念就是原点。确认用户指的是哪个概念空间（同一个词跨域时）。工作空间也是版图的一部分——项目类型、技术栈、当前文件都是推断概念空间的锚点，不依赖对话历史也能定位概念所属领域。

测半径：用信号推断意图，决定探索范围：

| 信号 | 意图 | 探索策略 |
|------|------|---------|
| 前文讨论实现时突然问 | 遇到障碍 | 小半径：本质+边界够用就停 |
| 主动对比两个概念 | 校正混淆 | 中半径：边界测绘优先 |
| "我在学 X" | 系统学习 | 大半径：沿多个维度均匀展开 |
| "X 和 Y 选哪个" | 做决策 | 中半径：关联优先，弱化位置 |
| "你刚提到 X" | 好奇心延伸 | 小半径：本质+关联 |
| 无上下文直接问 | 未知 | 先反问定半径 |

拿不准 → 小半径起步，用户反馈后再扩展。

**非首次迭代：** 用户指向的概念可能在已探索版图内。此时不重新定原点，改为**选锚点**——以该节点为新出发点，从已有版图中选取最近的参照坐标，不做全领域扫描。半径评估只需覆盖从锚点到新探索区域的局部。

### 2. 绘版图

先确认海图：目标概念在概念空间中的位置是否清楚——属于哪个领域、和什么概念相邻。模糊 → 查资料（WebSearch、文档、代码）补到能画大致版图再继续。完全查不到 → 诚实标注位置未知，不硬画。

从对话历史找用户的已知坐标。用户主动提过的概念是版图上的锚点，没提过的当未知区域。

扫领域：展开目标概念在概念空间里的自然邻居圈——相邻、同级、配套、相反。这是领域知识扫描，先不管用户知不知道。

交叉过滤：
- 在版图内的 → 参照坐标
- 不在版图但强关联 → "借来定位"候选，标记迷雾
- 无关的 → 丢弃

选参照坐标：
- 优先从版图内已知点中选
- 如需引入新术语 → 标记为"借来定位，不用记"；或另选一个已知点
- 参照坐标够定位即可，过多会淹没原点

版图画完的标志：至少有一个已知点做参照坐标，可画出原点在版图上的粗略位置。

**版图累增：** 首次探索按上述流程绘图。后续迭代时，`.docs/` 归档文件中已有的概念直接作为已知坐标纳入版图，新触及的区域才需要补绘。无需重新扫描全领域。

### 3. 探索展开

沿维度从原点向外探索。按概念类型和意图，从维度池中选最相关者展开：

| 维度 | 探索什么 | 何时选 |
|------|---------|--------|
| **本质** | 它是什么，一句话定义 | 总是 |
| **动机** | 为什么存在，解决什么问题 | 总是，一两句即可 |
| **边界** | 它不是什么，与已知点的差异 | 有易混淆邻居时 |
| **位置** | 上层归哪类，下层有什么子类 | 概念属分类体系时 |
| **关联** | 和谁配套、同场景、协作 | 需知道怎么用时 |
| **机制** | 内部如何运作 | 技术/过程类概念 |

每个维度探索完，回到原点用白话总结该维度发现。

**空间关系的呈现：** 位置、边界、关联三个维度涉及概念间的空间关系。纯文本叙述时用户需要在脑中自行拼接。展开这些维度时优先用可视化辅助——
- 层级树（缩进列表或 mermaid 简图）表示**位置**的归属与分类
- 对照表或维恩边界描述表示**边界**的差异
- 关系网（配套、协作、上下游）表示**关联**

让关系一目了然，随后再用一两句白话点出关键差异。本质、动机、机制三个维度不需要可视化。

**新概念处理（探索中遇到）：**

| 角色 | 识别信号 | 处理 | 深度规则 |
|------|---------|------|---------|
| 参照坐标 | 去掉它，目标概念的表述仍然完整 | 一句话定义 + 标记"借来定位" | 不展开 |
| 必要前置 | 去掉它，目标概念的表述会断裂 | 临时驻锚→小半径探索→回到主原点 | 能连接回目标就停，不追溯前前置 |

**中止条件（满足任一）：**
- 已覆盖意图所需半径
- 下一维会进入**迷雾**区域且无替代路径
- 用户反馈已理解

### 4. 回锚

回到原点做独立总结：不依赖新术语，描述原点的本质、位置和边界。

总结完成后立即归档：将本次探索覆盖的维度写入 `.docs/` 下的归档文件。

**文件定位：**
- 根据目标概念所属的领域或概念空间，推断一个 topic 名（如 `js-runtime`、`database-indexing`）
- 文件名：`explain-concept-<topic>.md`，存放在当前工作目录的 `.docs/` 下（目录不存在则创建）
- 写之前扫描 `.docs/` 下已有文件——存在同名文件则追加，否则新建
- 追问的概念若明显属于不同的概念空间，则建新文件

**文件结构（以概念为单元）：**

```
# <Topic 名>
> 概念空间简介，一两句

## <概念 A>
> 一句话本质

### 位置
### 边界
### 关联
### 动机
### 机制

---

## <概念 B>
...
```

**追加规则：**
- 若文件中尚无该概念 → 作为新概念单元追加入文件末尾
- 若文件中已有该概念 → 合并更新：已有维度取最新内容覆盖，新维度追加
- 只写入本次探索实际覆盖的维度，未展开的维度不出现空节
- 可视化内容（mermaid、缩进树、对照表）原样写入对应维度小节
- **无增量不写入：** 若本次交互未产生新的维度内容（此前解释有误的修正、用户仅澄清理解且答疑无新信息、仅换种说法重复），不修改文件
- **不跨概念融合：** 不同概念单元间的语义重叠不作主动合并或重组，仅按概念名判断是否属于同一单元

回锚完成标志：
1. 用户能用自己的话说清原点是什么
2. 用户能在版图内定位原点，知道它和最近已知坐标的差异
3. 用户的意图已被覆盖

满足 → 抛一个邻接方向或安静。不满足 → 沿未覆盖维度继续。别问"明白了吗？"

### 5. 指锚

回锚后用户可能行为分三种：
- **追问当前概念的某个维度**（落在本质/动机/边界/位置/关联/机制之一）→ 回到 Stage 3 沿该维度继续深潜，不重走 Stage 1、2
- **指向已探索空间中的节点**（如追问某关联概念、必要前置）→ 以该节点为锚点，回到选锚点
- **指向空间外的概念** → 以新概念为原点，回到定原点，已有版图保留为已知坐标

指锚完成标志：识别出用户的指向类型（维度追问 / 版图内节点 / 空间外概念），确定下一轮的起点和是否重走 Stage 1、2。

迭代中每次回锚都追加归档到同一份空间文件，按已有合并规则处理。迭代何时停止由用户决定。
